¿Por qué usar un middleware de audio?

Los motores de juego ofrecen audio básico (AudioSource en Unity, AudioComponent en Unreal). Los juegos profesionales requieren variación dinámica, lógica de mezcla compleja y audio reactivo al gameplay sin modificar código. Wwise y FMOD Studio separan la lógica de audio del código del juego: el diseñador de sonido controla todo el comportamiento desde una herramienta dedicada, mientras el programador solo llama eventos por nombre.

  • Events: acciones que el engine dispara ("Play_Footstep_Wood")
  • States: contexto global del juego (Combat, Exploration, Pause)
  • Switches: variante por objeto (FootstepMaterial, WeaponType)
  • RTPCs: parámetros numéricos continuos (Vehicle_Speed, Player_Health)
  • SoundBanks: paquetes binarios de assets cargados por nivel

Feynman · M1

¿Cuál es la diferencia fundamental entre un Switch y un State en Wwise?
Ver respuesta

Los States son globales al juego: cuando cambias el State a "Combat", afecta a todos los sonidos que tengan lógica de State configurada. Los Switches son por Game Object: cada NPC puede tener su propio valor de "FootstepMaterial" = "Stone" mientras otro tiene "Wood". El programador cambia un State con SetState() (sin gameObject) y un Switch con SetSwitch() (con gameObject específico).

Feynman · M2

¿Por qué las Entry Cues son fundamentales para que las transiciones musicales no produzcan clics o saltos?
Ver respuesta

Sin Entry Cues, Wwise transicionaría al segmento destino en cualquier punto de la reproducción — cortando en medio de un compás, creando discontinuidades rítmicas y armónicas. Las Entry Cues marcan puntos musicalmente coherentes (downbeat, inicio de compás, resolución armónica). Wwise espera a que el segmento actual alcance su Exit Cue, y solo entonces hace la transición al Entry Cue del segmento destino, garantizando que el corte ocurra en un punto musicalmente "legítimo".

Oclusión vs Obstrucción

ConceptoDefiniciónEfecto en audio
OclusiónEl sonido pasa a través de un material sólidoAtenuación de volumen + filtro paso-bajo
ObstrucciónObjeto entre fuente y listener pero con camino indirectoAtenuación parcial, preserva grave

OcclusionManager.cs — Implementación Unity

public class OcclusionManager : MonoBehaviour {
    [SerializeField] private Transform listener;
    [SerializeField] private AkGameObj[] audioSources;
    private float updateInterval = 0.1f;

    void UpdateOcclusion(AkGameObj src) {
        Vector3 dir = src.transform.position - listener.position;
        float dist = dir.magnitude;
        int hits = Physics.RaycastNonAlloc(
            listener.position, dir.normalized,
            new RaycastHit[4], dist,
            LayerMask.GetMask("Walls"));
        float occlusion = Mathf.Clamp01(hits * 0.5f);
        float obstruction = (hits == 1) ? 0.3f : 0f;
        AkSoundEngine.SetObjectObstructionAndOcclusion(
            src.gameObject, listener.gameObject,
            obstruction, occlusion);
    }
}

Feynman · M3

¿Cuál es la diferencia práctica entre oclusión y obstrucción, y cómo afectan al audio de forma diferente?
Ver respuesta

La oclusión ocurre cuando el sonido tiene que atravesar un material sólido (paredes, suelo): el resultado es atenuación de volumen y filtrado paso-bajo (los graves penetran, los agudos son absorbidos). La obstrucción ocurre cuando hay un objeto en la línea de visión directa pero existe un camino indirecto (reflexiones): la atenuación es parcial y los graves se preservan mejor. En Wwise: Occlusion afecta al dry path y wet path; Obstruction afecta solo al dry path, el reverb (sonido reflejado) permanece intacto.

Feynman · M4

¿Por qué la música se sirve siempre desde streaming y los SFX cortos desde RAM?
Ver respuesta

La música dura minutos — precargarla en RAM consumiría decenas de MB para una sola pista. El streaming lee el audio desde disco en chunks pequeños durante la reproducción, pagando solo un buffer mínimo en RAM. Los SFX cortos (<2s) deben dispararse con latencia cero: si estuvieran en disco, el I/O seek añadiría latencia impredecible (50-200ms). En RAM ya están en memoria y el motor puede iniciar la reproducción de forma inmediata en el siguiente frame de audio (1.33ms @ 48kHz/64 samples).

Ducking via side-chain

Cuando hay diálogo activo, el Music Bus recibe un side-chain del Dialogue Bus que lo atenúa automáticamente. El diseñador configura el attack (rápido = inmediato), release (lento = fade natural) y el ratio de reducción (–6 a –12 dB) sin que el programador intervenga.

Parámetro duckingValor típicoEfecto
Attack Time50–100 msVelocidad de bajada del volumen
Release Time500 ms–1 sVelocidad de recuperación al terminar diálogo
Reduction–6 a –12 dBCuánto baja la música durante el diálogo

Feynman · M5

¿Qué ventaja tiene el ducking via side-chain frente a bajar el volumen de música desde código?
Ver respuesta

El side-chain es proporcional y automático: si hay mucho diálogo, más ducking; si es suave, menos. Tiene envelope (attack/release) para que no sea un click abrupto. El diseñador controla todos los parámetros en Wwise sin que el programador sepa que existe. Con código manual, el programador tendría que detectar cuándo el diálogo inicia/termina, gestionar el volumen con Lerp, y actualizar la lógica cada vez que se añade un bus nuevo. El side-chain es declarativo y no requiere ningún código adicional.

AudioManager.cs — Singleton con pool de instancias

public class AudioManager : MonoBehaviour {
    public static AudioManager Instance { get; private set; }
    private Dictionary<string, EventInstance> loops =
        new Dictionary<string, EventInstance>();

    void Awake() {
        if (Instance != null) { Destroy(gameObject); return; }
        Instance = this;
        DontDestroyOnLoad(gameObject);
    }

    public void PlaySFX(string path, Vector3 pos) =>
        RuntimeManager.PlayOneShot(path, pos);

    public void PlayLoop(string path, GameObject go) {
        if (loops.ContainsKey(path)) return;
        var inst = RuntimeManager.CreateInstance(path);
        RuntimeManager.AttachInstanceToGameObject(inst, go.transform);
        inst.start();
        loops[path] = inst;
    }

    public void StopLoop(string path) {
        if (!loops.TryGetValue(path, out var inst)) return;
        inst.stop(STOP_MODE.ALLOWFADEOUT);
        inst.release();
        loops.Remove(path);
    }

    void OnDestroy() {
        foreach (var inst in loops.Values) {
            inst.stop(STOP_MODE.IMMEDIATE);
            inst.release();
        }
        loops.Clear();
    }
}

Feynman · M6

¿Qué es un Snapshot en FMOD y cuándo se usa en lugar de modificar parámetros individuales?
Ver respuesta

Un Snapshot en FMOD es un preset de mezcla que, cuando se activa, sobreescribe múltiples parámetros simultáneamente: volumen de buses, efectos activos, compresión, etc. Se usa cuando necesitas cambiar el estado global del audio de una sola vez (pause menu: baja música, filtra SFX; underwater: filtra todo con paso-bajo). En lugar de actualizar 20 parámetros individualmente desde código, el programador solo activa/desactiva el Snapshot con una llamada, y el diseñador gestiona qué cambia exactamente dentro del Snapshot.

Feynman · M7

¿Por qué se usan ScriptableObjects para referenciar eventos de audio en lugar de strings directos en el código?
Ver respuesta

Con strings directos ("Play_Footstep_Wood"), si el diseñador de sonido renombra el evento en Wwise, el programador tiene que buscar todas las ocurrencias del string en el código. Con ScriptableObjects, el nombre del evento se configura en el Inspector del asset — cambiar el nombre en un lugar actualiza todos los objetos que referencian ese ScriptableObject. Además, los SOs permiten añadir lógica extra (atenuación mínima/máxima, validación de nombre) sin tocar MonoBehaviours. También son serializables y aparecen en el Inspector con referencias tipadas.

MetaSounds — Audio procedural nativo UE5

MetaSounds es el sistema de audio procedural nativo de Unreal 5. Permite crear sonidos con lógica DSP visual (similar a Faust o Pure Data) sin plugins externos. Para juegos que no necesitan Wwise/FMOD, MetaSounds ofrece síntesis, sample playback y modulación en un editor nodal integrado.

Feynman · M8

¿Cuándo elegirías MetaSounds nativo de UE5 frente a integrar Wwise en Unreal?
Ver respuesta

MetaSounds es adecuado para proyectos indie o académicos donde la lógica de audio es relativamente simple, no se requiere el Wwise Profiler, y el presupuesto no justifica la licencia. Sus ventajas: cero dependencias externas, integración perfecta con UE5, síntesis procedural. Wwise en Unreal es necesario cuando: el juego tiene lógica de audio compleja (Interactive Music, múltiples States anidados), se necesita el Wwise Profiler para diagnóstico, hay un equipo de sound designers que ya conoce Wwise Authoring Tool, o se requiere multiplatforma con optimizaciones específicas por plataforma (ADPCM en Switch, AT9 en PS5).

Feynman · M9

¿Por qué la atenuación de audio en juegos no sigue la ley del cuadrado inverso exacta de la física?
Ver respuesta

La ley del cuadrado inverso (–6 dB por duplicación de distancia) funciona en campo libre sin obstrucciones. En juegos, aplicarla directamente haría que un NPC a 4 m de distancia ya fuera inaudible en muchos casos, lo cual rompería el gameplay. Además, los juegos no simulan campo libre: hay reverb de habitación, paredes que reflejan sonido, etc. Los diseñadores usan curvas en S con un plateau cercano (volumen máximo hasta X metros), una caída más agresiva en distancia media, y una caída suave al final para que el sonido siga siendo localizable. La curva específica depende de la mecánica: footsteps muy agresivos (solo relevantes cerca), explosiones muy suaves (percibidas a 150 m).

M10 · Audio Procedural
Síntesis Procedural — OnAudioFilterRead
Modal synthesis, impactos físicos y síntesis por modos de resonancia

¿Por qué síntesis procedural?

Los samples pregrabados no se adaptan a la física del impacto: un disparo siempre suena igual aunque la velocidad sea diferente. La síntesis procedural genera el audio en tiempo real basándose en los parámetros físicos del evento (velocidad de impacto, material, masa).

ProceduralImpact.cs — Síntesis modal por modos de resonancia

public class ProceduralImpact : MonoBehaviour {
    public enum Material { Metal, Wood, Plastic }
    private float[] metalModes   = { 320f,  640f, 1200f, 2400f };
    private float[] woodModes    = { 180f,  380f,  760f, 1400f };
    private float[] plasticModes = { 520f,  980f, 1800f, 3200f };
    private float metalDamp = 2f, woodDamp = 12f, plasticDamp = 6f;

    void OnCollisionEnter(Collision col) {
        float energy = Mathf.Clamp(col.relativeVelocity.magnitude / 10f, 0f, 1f);
        for (int i = 0; i < currentAmps.Length; i++)
            currentAmps[i] = energy * (1.0f / (i + 1));
        phase = 0f;
    }

    void OnAudioFilterRead(float[] data, int channels) {
        for (int s = 0; s < data.Length; s += channels) {
            float sample = 0f;
            for (int m = 0; m < currentModes.Length; m++)
                sample += currentAmps[m]
                    * Mathf.Exp(-currentDamp * phase / sampleRate)
                    * Mathf.Sin(2f * Mathf.PI * currentModes[m] * phase / sampleRate);
            data[s] = data[s + 1] = Mathf.Clamp(sample, -1f, 1f);
            phase++;
        }
    }
}

Feynman · M10

¿Qué ventaja tiene la síntesis modal procedural frente a usar un sample de impacto pregrabado?
Ver respuesta

Un sample pregrabado es estático: siempre suena igual independientemente de si el impacto fue a 1 m/s o 50 m/s. La síntesis modal calcula las frecuencias de resonancia del material (modos) y sus amplitudes en función de la energía del impacto (col.relativeVelocity.magnitude). Un golpe suave produce un tono tenue de corta duración; un golpe fuerte produce el mismo timbre pero con más amplitud y sustain. Además, el metal suena diferente que la madera porque tienen modos y damping distintos. La síntesis también ocupa 0 bytes de disco — no hay samples que cargar ni gestionar en SoundBanks.

Feynman · M11

¿Por qué el HRTF personalizado da mejor localización que el HRTF genérico, y qué opciones existen en 2024 para obtenerlo sin laboratorio?
Ver respuesta

El HRTF encapsula cómo la forma específica de tus orejas, cabeza y torso colorizan el sonido según la dirección de llegada. Un HRTF genérico es el promedio de muchas personas — funciona aceptablemente para azimut frontal pero falla en elevación (arriba/abajo), ya que los pliegues del pabellón auricular son únicos para cada persona. En 2024, opciones para HRTF personal sin laboratorio: (1) Apps de foto de oído que usan ML para estimar el HRTF a partir de imágenes (ej: Personalized HRTF de Sony, Embody); (2) Selección subjetiva de un catálogo de HRTFs en formato SOFA donde el usuario elige la que mejor localiza; (3) Escáneres 3D con smartphone del pabellón auricular para modelos físicos computacionales.

Feynman · M12

¿Qué es una "voz virtual" en Wwise y cómo reduce el coste de CPU sin perder el estado del sonido?
Ver respuesta

Una voz virtual es aquella cuyo procesamiento DSP ha sido suspendido porque está fuera del rango audible o por debajo del umbral de virtualización. No consume CPU de audio, pero Wwise sigue trackando su posición en el tiempo, su posición 3D en el mundo y sus parámetros actuales. Cuando el listener se acerca o la voz vuelve al rango audible, Wwise la "reactiva" desde el punto correcto de la reproducción — el sonido se escucha como si hubiera estado sonando todo el tiempo. Esto permite tener 500 fuentes de sonido en una escena grande sin pagar CPU por las 400 que están lejos del jugador.

validate_audio_naming.py

import json, sys, re, pathlib

VALID_PREFIXES = ['Play_', 'Stop_', 'Pause_', 'Resume_']
VALID_CATEGORIES = ['SFX', 'Music', 'Amb', 'UI', 'VO']

def validate(wwise_project_path):
    banks_info = pathlib.Path(wwise_project_path) / 'GeneratedSoundBanks' / 'SoundbanksInfo.json'
    if not banks_info.exists():
        print("No SoundbanksInfo.json — skipping validation")
        return True
    data = json.loads(banks_info.read_text())
    errors = []
    for bank in data.get('SoundBanks', []):
        for event in bank.get('IncludedEvents', []):
            name = event['Name']
            if not any(name.startswith(p) for p in VALID_PREFIXES):
                errors.append(f"BAD PREFIX: {name}")
            parts = name.split('_')
            if len(parts) < 3 or parts[1] not in VALID_CATEGORIES:
                errors.append(f"BAD CATEGORY: {name}")
    if errors:
        print("NAMING ERRORS:\n" + "\n".join(errors))
        sys.exit(1)
    print(f"All events validated OK")

validate(sys.argv[1])

Feynman · M13

¿Por qué es imprescindible usar Git LFS para los assets de audio y qué ocurre si no se usa?
Ver respuesta

Git almacena cada versión de un archivo binario de forma completa en el historial. Un WAV de 50 MB modificado 100 veces = 5 GB solo en ese archivo, más toda la historia del repo. Sin LFS, clonar el repositorio implica descargar el historial completo — potencialmente cientos de GB. Git LFS almacena el binario en un servidor externo y guarda solo un puntero (texto pequeño) en el historial de Git. Al hacer checkout, solo descarga las versiones que se necesitan. El repo se mantiene manejable y los clones son rápidos incluso con miles de assets de audio.

Feynman 1 · Switches vs Events múltiples

¿Por qué usar Switches en lugar de múltiples Events separados para las pisadas de diferentes superficies?
Ver respuesta

Con Switches, un solo Event "Play_Footstep" selecciona automáticamente el sonido correcto según el valor del Switch Group "Surface_Type" (Wood, Stone, Metal…). El programador solo llama a un evento; el diseñador gestiona la lógica de selección en Wwise sin tocar código. Con eventos separados, el programador debe conocer y mantener una lista de eventos por superficie — cualquier añadido de superficie requiere código nuevo. El Switch centraliza la decisión en el middleware y hace el sistema extensible sin modificar el motor del juego.

Feynman 2 · Music Segments y Entry Cues

Explica cómo funciona el "Music Segment" y por qué las "Entry Cues" son fundamentales para transiciones musicales sin clicks.
Ver respuesta

Un Music Segment define una sección temporal de música con marcadores: Entry Cue (punto de entrada legítimo), Exit Cue (punto de salida), Custom Cues (triggers de stingers). Cuando Wwise debe transicionar de un segmento a otro, no lo hace inmediatamente — espera a la próxima Entry Cue del segmento destino para garantizar que la transición ocurra en un punto musicalmente coherente (downbeat, compás, resolución armónica). Sin Entry Cues, la transición cortaría en cualquier punto, produciendo saltos rítmicos y armónicos perceptibles que rompen la inmersión.

Feynman 3 · Voice Limiting global vs por objeto

¿Cuál es la diferencia entre voice limiting "global" y "per game object" en Wwise y cuándo usar cada uno?
Ver respuesta

El límite global (Max Voices en Project Settings) define cuántas voces totales puede haber en el sistema en cualquier momento. El límite per game object define cuántas instancias de un mismo sonido puede emitir un objeto específico. Ejemplo: una ametralladora sin límite per-object acumularía decenas de disparos superpuestos. Con per-object Max=8 + Discard Oldest, siempre hay como máximo 8 disparos simultáneos de ese arma. Para SFX de impacto: global moderado + per-object estricto. Para ambientes: global generoso + per-object = 1.

Feynman 4 · Curva logarítmica en atenuación

¿Por qué se usa una curva personalizada (no lineal) en la Attenuation curve de volumen en Wwise?
Ver respuesta

La ley del cuadrado inverso pura (–6 dB por duplicación de distancia) es demasiado agresiva para juegos: a 4 metros, el sonido ya sería inaudible en muchos casos, rompiendo el gameplay. Los diseñadores usan curvas en S: un plateau cercano (volumen máximo hasta X metros), una caída rápida en zona media para que el audio se desvanezca progresivamente, y una caída muy suave al final para que el sonido siga siendo localizable hasta el máximo de atenuación. La curva específica depende del tipo: footsteps muy agresivos (solo relevantes a corta distancia), explosiones muy suaves (perceptibles a 150 m).

Feynman 5 · Ducking via side-chain

¿Cómo funciona el ducking mediante side-chain en Wwise y cuál es su ventaja respecto a bajar el volumen de la música manualmente desde código?
Ver respuesta

El ducking via side-chain conecta el análisis de amplitud de un bus (ej: Dialogue Bus) a un atenuador en otro bus (ej: Music Bus). Cuando hay diálogo activo, la música baja automáticamente X dB con un attack/release configurable, sin intervención del código del juego. Ventajas vs código manual: es proporcional (si hay mucho diálogo, más ducking; si es suave, menos), tiene envelope (attack/release) para que no sea un click abrupto, y el diseñador controla todos los parámetros en Wwise sin que el programador sepa que existe. Con código manual, habría que detectar inicio/fin de diálogo, gestionar Lerp del volumen, y actualizar la lógica cada vez que se añade un bus nuevo.
