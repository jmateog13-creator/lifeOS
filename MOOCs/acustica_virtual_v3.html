M1 · Acústica de Sala
Frecuencia de Schroeder
Modos propios, zona de baja frecuencia, transición a campo difuso

La frecuencia de Schroeder: umbral de campo difuso

Por debajo de la frecuencia de Schroeder (fs), el campo sonoro de una sala está dominado por modos propios discretos: frecuencias resonantes que dependen de la geometría. Por encima, los modos se solapan suficientemente para formar un campo difuso estadístico, donde el tratamiento wave-based (FDTD) puede reemplazarse por ray tracing y modelos energéticos.

f_s = 2000 · √(RT60 / V) [Hz]

Donde RT60 es el tiempo de reverberación en segundos y V el volumen en m³. Sala de conciertos (V=15000m³, RT60=2s): fs ≈ 23Hz. Sala de control (V=50m³, RT60=0.3s): fs ≈ 49Hz.

Modos propios de una sala rectangular

f_{lmn} = (c/2) · √((l/Lx)² + (m/Ly)² + (n/Lz)²) [Hz]

l, m, n = índices del modo (enteros no negativos, no todos cero). c = velocidad del sonido (≈344 m/s). Los modos axiales (solo un índice ≠0) son los más fuertes; tangenciales y oblicuos son más débiles.

  • Sabine: RT60 = 0.161·V/A. Válido para ᾱ < 0.2 (salas muy reverberantes).
  • Eyring: RT60 = −0.161·V/(S·ln(1−ᾱ)). Más preciso para ᾱ altos.

import numpy as np
import matplotlib.pyplot as plt

def room_modes(Lx, Ly, Lz, c=344, max_freq=500):
    modes = []
    max_l = int(2 * max_freq * Lx / c) + 1
    max_m = int(2 * max_freq * Ly / c) + 1
    max_n = int(2 * max_freq * Lz / c) + 1
    for l in range(max_l + 1):
        for m in range(max_m + 1):
            for n in range(max_n + 1):
                if l == m == n == 0:
                    continue
                f = (c / 2) * np.sqrt(
                    (l/Lx)**2 + (m/Ly)**2 + (n/Lz)**2
                )
                if f <= max_freq:
                    n_zeros = sum(1 for x in [l,m,n] if x == 0)
                    mode_type = ['Oblicuo','Tangencial','Tangencial','Axial'][n_zeros]
                    modes.append({'f': f, 'l': l, 'm': m, 'n': n, 'type': mode_type})
    return sorted(modes, key=lambda x: x['f'])

def schroeder_frequency(rt60, V):
    return 2000 * np.sqrt(rt60 / V)

V = Lx * Ly * Lz

    print(f"  ({m['l']},{m['m']},{m['n']}) = {m['f']:.1f}Hz [{m['type']}]")

Feynman — Modos axiales

¿Por qué los modos axiales son más problemáticos que los oblicuos?

Ver respuesta

Los modos axiales (solo una dimensión implicada) tienen la mayor energía y el menor factor de pérdidas relativo. Solo "rebotan" entre dos paredes paralelas y solo encuentran material absorbente en esas dos superficies. Los modos tangenciales afectan 4 superficies, los oblicuos las 6. A igual coeficiente de absorción, los modos axiales tienen más tiempo para decaer y crean coloración tonal más severa. En un cubo perfecto, los primeros modos en las tres dimensiones son iguales → triple refuerzo en una sola frecuencia → resonancia devastadora para mezcla.

Quiz rápido

¿Qué ocurre con la frecuencia de Schroeder si duplicas el RT60 manteniendo el volumen constante?

Ver respuesta

fs aumenta un factor √2 (≈1.41x). La fórmula es fs = 2000·√(RT60/V). Si RT60 se duplica, el argumento de la raíz se duplica, por lo que fs se multiplica por √2. No se duplica (eso requeriría RT60×4) ni se cuadruplica.

M2 · Simulación
Image Source Method
Reflexiones especulares, complejidad O(N^m), pyroomacoustics

ISM: reflexiones como fuentes imagen

El Image Source Method modela las reflexiones especulares de una sala como si vinieran de fuentes virtuales: la imagen de la fuente real reflejada en cada pared. Las reflexiones de orden N corresponden a las imágenes de imágenes, recursivamente.

Complejidad: O(N^m) — N paredes, m = orden de reflexión

Para una caja rectangular (6 paredes) hasta orden 3: 6 + 30 + 126 = 162 fuentes imagen. Para orden 10: >60 millones. El ISM es exacto (reflexiones especulares) pero prohibitivo para órdenes altos. Práctico hasta orden 6-8 para geometrías simples.

  • Direct sound: primera llegada, sin reflexiones.
  • Early reflections (ER): <50ms, dan sensación de espacio.
  • Late reverberation (>50ms): campo difuso estadístico.

Feynman — ISM y órdenes altos

¿Por qué ISM se vuelve prohibitivo para órdenes altos pero sigue siendo útil?

Ver respuesta

ISM es O(N^m): para 6 paredes y orden 10, hay 6^10 ≈ 60 millones de fuentes imagen, muchas inaccesibles. Sin embargo, ISM es exacto para reflexiones especulares de baja frecuencia donde la longitud de onda es comparable a la geometría: exactamente en la zona donde la teoría geométrica (ray tracing) no es válida. Por eso se usa híbrido: ISM para reflexiones tempranas (orden bajo) + ray tracing estadístico para la cola tardía.

Quiz rápido

¿Qué representa la "direct-to-reverberant ratio" (DRR) en acústica de sala?

Ver respuesta

La DRR es la energía del sonido directo respecto a la energía reverberante total. Un DRR alto indica que el oyente está cerca de la fuente (sonido directo domina); uno bajo indica que la reverberación domina. Es fundamental para percibir la distancia de una fuente sonora: cerebros y sistemas de audio lo usan como clave de distancia.

M3 · Simulación
FDN — Feedback Delay Network
Matriz de retroalimentación ortogonal, densidad de ecos, diseño de reverb

FDN: reverb artificial de alta calidad

El Feedback Delay Network es la arquitectura de reverberación artificial más usada en procesadores profesionales (Lexicon, Bricasti). Consiste en N líneas de retardo retroalimentadas a través de una matriz de mezcla ortogonal para preservar la energía y evitar la coloración tonal.

x[n] = Σ_k A_{ik} · g_k · x_k[n − d_k] + b_i · u[n]

A = matriz ortogonal de mezcla. d_k = longitudes de retardo (en muestras). g_k = ganancias de feedback. b_i = vector de entrada.

  • Hadamard: matrices de tamaño potencia de 2 (N=4,8,16,32). Muy eficiente computacionalmente.
  • Householder: H = I - 2vv^T, v unitario. Generalización flexible.
  • Longitudes de retardo: primos relativos, distribuidos en rango [d_min, d_max].
  • Ganancias de feedback: g_k = 10^(−3d_k/(fs·RT60)). Garantizan RT60 objetivo.

Lab 3.1 — FDN de 8 canales con matriz Hadamard

Ver código Python 
import numpy as np
from scipy.linalg import hadamard

def design_fdn(N=8, rt60=2.0, sr=44100):
    H = hadamard(N).astype(float) / np.sqrt(N)
    from sympy import primerange
    d_min = int(sr * 0.005)
    d_max = int(sr * 0.050)
    primes = list(primerange(d_min, d_max + 1))
    indices = np.round(np.linspace(0, len(primes)-1, N)).astype(int)
    delays = [primes[i] for i in indices]
    gains = np.array([10**(-3.0 * d / (sr * rt60)) for d in delays])
    return {'delays': delays, 'gains': gains, 'A': H, 'N': N}

class FDN:
    def __init__(self, design):
        self.N = design['N']
        self.delays = design['delays']
        self.gains = design['gains']
        self.A = design['A']
        self.buffers = [np.zeros(d) for d in self.delays]
        self.ptrs = [0] * self.N
        self.b = np.ones(self.N) / np.sqrt(self.N)
        self.c = np.ones(self.N) / np.sqrt(self.N)

    def process_sample(self, x):
        s = np.array([self.buffers[k][self.ptrs[k]] for k in range(self.N)])
        mixed = self.A @ (self.gains * s)
        for k in range(self.N):
            self.buffers[k][self.ptrs[k]] = mixed[k] + self.b[k] * x
            self.ptrs[k] = (self.ptrs[k] + 1) % self.delays[k]
        return float(np.dot(self.c, s))

    def process_block(self, input_signal):
        output = np.zeros(len(input_signal))
        for i, x in enumerate(input_signal):
            output[i] = self.process_sample(x)
        return output

design = design_fdn(N=8, rt60=1.5, sr=44100)
fdn = FDN(design)
impulse = np.zeros(44100 * 3)
impulse[0] = 1.0
rir_fdn = fdn.process_block(impulse)
print(f"Delays: {design['delays']}")

Feynman — Matriz ortogonal en FDN

¿Por qué la matriz de mezcla del FDN debe ser ortogonal?

Ver respuesta

Si la matriz no es ortogonal (|det| ≠ 1), no preserva la norma (energía) de los vectores que transforma. Una matriz con |det| < 1 reduce la energía en cada iteración → la reverberación decae más rápido de lo diseñado. Una con |det| > 1 → inestabilidad. La ortogonalidad garantiza que toda la energía se disipa solo a través de las ganancias g_k, que son los únicos parámetros que controlan el RT60. Además, la mezcla completa en cada iteración maximiza la densidad de ecos y evita coloración tonal.

Quiz rápido

¿Cuál es la propiedad matemática clave de la matriz de mezcla en un FDN bien diseñado?

Ver respuesta

Debe ser ortogonal: A·A^T = I. Esto garantiza que |det(A)| = 1, preservando exactamente la energía en cada ciclo de mezcla. La ortogonalidad es la única propiedad que asegura estabilidad, control de RT60 y mezcla uniforme simultáneamente.

M4 · Simulación
NUPOLS — Convolución Particionada
Convolución en bloques, latencia mínima, UPOLS/NUPOLS, implementación eficiente

El problema de la convolución de RIRs largas

Una RIR de 2 segundos a 44.1kHz tiene 88200 muestras. La convolución directa (O(N²)) sería prohibitiva en tiempo real. La convolución vía FFT (overlap-add/overlap-save) reduce la complejidad a O(N·log N), pero introduce latencia igual al tamaño del bloque.

UPOLS: Uniform Partitioned OLS — todos los bloques del mismo tamaño L
NUPOLS: Non-Uniform — primer bloque L₀ mínimo, resto crecen geométricamente
  • UPOLS con L=128: latencia 3ms, CPU alta (muchas FFTs pequeñas).
  • NUPOLS: latencia ≈ L₀ (igual a bloque de audio), CPU reducida para cola larga.
  • En práctica: L₀=64 samples (1.5ms a 44.1kHz), bloques posteriores doblan tamaño (128, 256, 512...).

Lab 4.1 — UPOLS: convolución particionada uniforme

Ver código Python 
import numpy as np
from scipy.fft import rfft, irfft

def upols_convolve(signal, h, block_size=512):
    L = block_size
    fft_size = 2 * L
    n_partitions = int(np.ceil(len(h) / L))
    H_partitions = []
    for i in range(n_partitions):
        h_part = h[i*L:(i+1)*L]
        h_part = np.pad(h_part, (0, fft_size - len(h_part)))
        H_partitions.append(rfft(h_part))
    fdl = [np.zeros(fft_size//2 + 1, dtype=complex)] * n_partitions
    fdl_ptr = 0
    output = np.zeros(len(signal) + len(h) - 1)
    n_blocks = int(np.ceil(len(signal) / L))
    for b in range(n_blocks):
        x_block = signal[b*L:(b+1)*L]
        x_block = np.pad(x_block, (0, L - len(x_block)))
        X = rfft(np.concatenate([x_block, np.zeros(L)]))
        fdl[fdl_ptr] = X
        Y = np.zeros(fft_size//2 + 1, dtype=complex)
        for i in range(n_partitions):
            idx = (fdl_ptr - i) % n_partitions
            Y += fdl[idx] * H_partitions[i]
        y_time = irfft(Y)
        out_start = b * L
        output[out_start:out_start + L] += y_time[:L]
        output[out_start + L:out_start + 2*L] += y_time[L:]
        fdl_ptr = (fdl_ptr + 1) % n_partitions
    return output[:len(signal)]

sr = 44100
t_rir = np.linspace(0, 2, 2*sr)
h = np.exp(-3 * t_rir) * np.random.randn(len(t_rir)) * 0.1
h[0] = 1.0
signal = np.random.randn(sr * 5)
output = upols_convolve(signal, h, block_size=512)
print(f"Input: {len(signal)} samples, Output: {len(output)} samples")

Feynman — Por qué FFT reduce O(N²) a O(N·log N)

¿Por qué la convolución directa es O(N²) pero la FFT la reduce a O(N·log N)?

Ver respuesta

En convolución directa, para calcular cada muestra de salida se necesita multiplicar la señal con todos los N coeficientes de la RIR: N operaciones por muestra × N muestras = O(N²). La FFT convierte la convolución en multiplicación elemento a elemento en el dominio de frecuencia: 2 FFTs (O(N·log N) cada una) + N multiplicaciones complejas + 1 IFFT = O(N·log N) total. Para una RIR de 88200 samples: N² ≈ 7.8×10⁹ vs N·log N ≈ 1.5×10⁶ — una reducción de 5000x.

Quiz rápido

¿Qué ventaja tiene NUPOLS sobre UPOLS para convolución en tiempo real?

Ver respuesta

NUPOLS mantiene latencia mínima (igual al primer bloque L₀) con menor CPU total. El primer bloque es corto para procesar la parte perceptualmente crítica (sonido directo + early reflections) con baja latencia; los bloques posteriores son más grandes y se procesan en hilos separados con latencia permitida, amortizando el costo de FFT de la cola tardía.

M5 · Espacialización
Oclusión & Difracción
Keller edge diffraction, modelos de oclusión, ray tracing con obstáculos

Cuando un rayo choca con un obstáculo

El ray tracing geométrico asume propagación rectilínea: si hay un obstáculo entre fuente y receptor, el sonido se bloquea completamente. Físicamente, el sonido se difracta alrededor de los bordes. La teoría de difracción de Keller (GTD/UTD) extiende la óptica geométrica añadiendo rayos difractados.

p_diffracted = D(θ, θ') · p_incident · e^{-jkr} / √r

D(θ, θ') = coeficiente de difracción (función del ángulo de incidencia θ y observación θ', y del ángulo de cuña n). Para una cuña de 90°: n = 2. Para arista de 270° interior: n = 2/3.

  • Occlusion factor: 0 = completamente bloqueado, 1 = directo libre. Interpolado por rayos de sombra.
  • Portal acoustics: modelar puertas/ventanas como zonas de transmisión con filtro de corte.
  • Low-pass filtering: una pared bloquea HF pero transmite LF. Filtro LP cuyo fc depende del material y grosor.

Ver código Python 
import numpy as np
from scipy.signal import butter, filtfilt

def apply_occlusion_filter(audio, occlusion_factor, sr=44100,
                            wall_material='concrete'):
    fc_map = {
        'concrete': 200,
        'drywall': 800,
        'glass': 3000,
    }
    fc = fc_map.get(wall_material, 500)
    attenuation_db = (1 - occlusion_factor) * -40
    attenuation_lin = 10 ** (attenuation_db / 20)
    if occlusion_factor < 0.99:
        nyq = sr / 2
        b, a = butter(2, fc / nyq, btype='low')
        filtered = filtfilt(b, a, audio)
    else:
        filtered = audio.copy()
    return filtered * attenuation_lin

sr = 44100
t = np.linspace(0, 1, sr)
speech = np.sin(2*np.pi*440*t) + 0.5*np.sin(2*np.pi*880*t)
occluded = apply_occlusion_filter(speech, 0.1, sr, 'concrete')
print(f"Atenuación media: {20*np.log10(np.std(occluded)/np.std(speech)+1e-8):.1f}dB")

Feynman — Por qué el grave penetra mejor las paredes

¿Por qué el sonido grave penetra mejor las paredes que el agudo?

Ver respuesta

La transmisión a través de una pared depende de la impedancia de la pared. A bajas frecuencias, el movimiento de la pared está controlado por su masa (ley de masa: TL aumenta 6dB por octava con la frecuencia). Esto significa que las frecuencias bajas mueven la pared más fácilmente y transmiten más energía. Adicionalmente, la difracción es proporcional a λ/d: las ondas largas (graves) difractan más alrededor de los bordes del obstáculo que las ondas cortas (agudos).

Quiz rápido

¿Qué tipo de filtro modela mejor el efecto de una pared de hormigón en oclusión de audio para juegos?

Ver respuesta

Filtro pasa-bajos con fc muy bajo (100-300Hz) + atenuación fuerte. El hormigón tiene muy alta masa, atenúa fuertemente todas las frecuencias pero transmite algo de los graves. La fc baja captura que casi solo pasan las frecuencias más graves, y la atenuación fuerte refleja la alta impedancia del material.

M6 · Espacialización
HRTF & Armónicos Esféricos
Head-Related Transfer Function, representación SH, espacialización 3D

HRTF: cómo el cuerpo filtra el espacio

La Head-Related Transfer Function (HRTF) codifica las modificaciones espectrales y temporales que el torso, cabeza y pabellón auricular imprimen al sonido que llega desde una dirección específica (azimut θ, elevación φ).

HRTF(θ,φ,f) = P_ear(θ,φ,f) / P_free_field(f)

Los Armónicos Esféricos Y_l^m(θ,φ) forman una base ortonormal completa para funciones en la esfera. Expandir la HRTF en SH permite compresión eficiente y renderizado con Ambisonics.

H(θ,φ,f) = Σ_{l=0}^{L} Σ_{m=-l}^{l} ĥ_l^m(f) · Y_l^m(θ,φ)

Orden L: L=4 → (L+1)²=25 coeficientes. L=7 → 64 coeficientes. En práctica: L=4 para Ambisonics de 4to orden.

    hrtf = sofar.read_sofa(sofa_path)
    positions = hrtf.SourcePosition

    idx = np.argmin(dist)

    hrir_right = hrtf.Data_IR[idx, 1, :]
    sr = hrtf.Data_SamplingRate

def spatialize(mono_audio, hrir_left, hrir_right):
    left  = fftconvolve(mono_audio, hrir_left)[:len(mono_audio)]
    right = fftconvolve(mono_audio, hrir_right)[:len(mono_audio)]
    stereo = np.stack([left, right], axis=1).astype(np.float32)
    max_val = np.max(np.abs(stereo))
    if max_val > 0:
        stereo /= max_val
    return stereo

def synthetic_hrtf(azimuth_deg, sr=44100, head_radius=0.085):
    c = 344.0

    itd = head_radius / c * (np.sin(az_rad) + az_rad)

    if azimuth_deg > 0:

        hrir_right = hrir_base
    else:
        hrir_left  = hrir_base
        hrir_right = np.roll(hrir_base, itd_samples)
    return hrir_left, hrir_right

Feynman — Front-back confusion con HRTF genérica

¿Por qué es difícil distinguir sonidos delante vs atrás con HRTF genérica?

Ver respuesta

El pabellón auricular crea filtros espectrales ("pinna notches") distintos para adelante/atrás que son altamente individualizados: dependen de la forma exacta de tu oreja. Con una HRTF genérica, las pinna notches no coinciden con las de tu morfología: tu cerebro no reconoce los patrones espectrales como señales de localización válidas y hay ambigüedad frontal/trasera (front-back confusion problem). Las HRTFs personalizadas eliminan esto pero requieren medición individual.

import torch
import torch.nn as nn

class HRTFTransformer(nn.Module):
    def __init__(self, n_anthro=17, d_model=64, n_heads=4,
                 n_layers=3, n_freqs=128, n_sh=25):
        super().__init__()
        self.token_embed = nn.Linear(1, d_model)
        self.freq_scales = nn.Parameter(torch.ones(n_freqs, n_anthro))
        encoder_layer = nn.TransformerEncoderLayer(
            d_model=d_model, nhead=n_heads,
            dim_feedforward=256, dropout=0.1, batch_first=True
        )
        self.transformer = nn.TransformerEncoder(encoder_layer, n_layers)
        self.head = nn.Linear(d_model * n_anthro, n_freqs * n_sh)
        self.n_freqs = n_freqs
        self.n_sh = n_sh

    def forward(self, x, freq_idx=None):
        batch_size = x.size(0)
        if freq_idx is not None:
            x = x * self.freq_scales[freq_idx]
        tokens = self.token_embed(x.unsqueeze(-1))
        z = self.transformer(tokens)
        z_flat = z.reshape(batch_size, -1)
        out = self.head(z_flat)
        return out.view(batch_size, self.n_freqs, self.n_sh)

Feynman — Por qué Token Scaling mejora frente a MLP estándar

¿Por qué el Token Scaling mejora la predicción de HRTF frente a un MLP estándar?

Ver respuesta

Diferentes medidas antropométricas son relevantes para diferentes frecuencias. El ancho de la cabeza determina principalmente el ITD (frecuencias bajas). Las dimensiones del pabellón auricular crean las pinna notches que informan elevación a frecuencias medias-altas (4-16kHz). Un MLP estándar aplica el mismo peso a todas las medidas para todas las frecuencias, ignorando esta selectividad. El Token Scaling permite que el modelo aprenda f_k: cuánto peso dar a cada medida en función de la frecuencia, capturando la especificidad frecuencial de cada estructura anatómica.

Quiz rápido

¿Qué métrica estándar se usa para evaluar la calidad de una HRTF predicha vs medida?

Ver respuesta

LSD — Log-Spectral Distance en dB. Mide la diferencia media cuadrática entre los espectros logarítmicos de la HRTF predicha y la medida. Es perceptivamente relevante porque el oído percibe el espectro en escala logarítmica. Valores típicos: LSD < 6 dB se considera buena predicción; el estado del arte (HRTFformer) alcanza ~4.2 dB.

Feynman — Por qué se bakea el audio de sala en juegos

¿Por qué se "bakea" (pre-computa) el audio de sala en juegos en lugar de calcularlo en tiempo real?

Ver respuesta

El ray tracing acústico completo (ISM + difracción + scattering) requiere cientos de milisegundos para converger. A 60fps, el presupuesto total de CPU del juego es ~16ms. No hay forma de calcular acústica completa en tiempo real con ese presupuesto. El baking pre-computa las RIRs para una cuadrícula de posiciones (probe volumes) durante el desarrollo. En runtime, el motor interpola los valores bakeados de los probes cercanos al listener: baked para la cola tardía, ray tracing real-time solo para las primeras reflexiones críticas.

Quiz rápido

¿Qué es un RTPC en Wwise y para qué se usa?

Ver respuesta

Un RTPC (Real-Time Parameter Control) es un parámetro del juego que controla propiedades de audio en tiempo real. Por ejemplo, la distancia al listener controla el volumen y el nivel de reverb de una fuente. El RTPC conecta la lógica del juego (un float que cambia cada frame) con los parámetros de la cadena de audio de Wwise, sin necesidad de recompilar el proyecto.

Feynman — Por qué MetaSound ejecuta a nivel de muestra

¿Por qué MetaSound ejecuta el grafo a nivel de muestra y no de frame?

Ver respuesta

El render de gráficos ocurre a 60fps (16ms por frame). Los artefactos de audio a esa resolución temporal serían claramente audibles: el oído detecta discontinuidades de amplitud hasta 1ms. La síntesis de audio requiere control sample-accurate (cada 1/44100s ≈ 0.02ms) para evitar aliasing, clicks y distorsión. Si MetaSound controlara parámetros solo a 60fps, los cambios bruscos de frecuencia o amplitud crearían zipper noise. Ejecutar el grafo a nivel de muestra permite síntesis limpia aunque los parámetros del juego vengan a frecuencias menores.

#include <immintrin.h>
void fir_sse(float* output, const float* input,
             const float* coefs, int N, int n_samples) {
    for (int i = 0; i < n_samples; i += 4) {
        __m128 acc = _mm_setzero_ps();
        for (int k = 0; k < N; k++) {
            __m128 x = _mm_loadu_ps(&input[i - k]);
            __m128 h = _mm_set1_ps(coefs[k]);
            acc = _mm_fmadd_ps(x, h, acc);  // acc += x * h
        }
        _mm_storeu_ps(&output[i], acc);
    }
}

// Cargo.toml: cpal = "0.15"
use cpal::traits::{DeviceTrait, HostTrait, StreamTrait};

struct FDN {
    delays: Vec<Vec<f32>>,
    ptrs: Vec<usize>,
    gains: Vec<f32>,
    hadamard: [[f32; 4]; 4],
}

impl FDN {
    fn new(delay_sizes: Vec<usize>, gains: Vec<f32>) -> Self {
        let n = delay_sizes.len();
        let delays = delay_sizes.iter().map(|&d| vec![0.0f32; d]).collect();
        let ptrs = vec![0usize; n];
        let h = 0.5f32;
        let hadamard = [
            [ h,  h,  h,  h],
            [ h, -h,  h, -h],
            [ h,  h, -h, -h],
            [ h, -h, -h,  h],
        ];
        FDN { delays, ptrs, gains, hadamard }
    }

    fn process(&mut self, input: f32) -> f32 {
        let n = self.delays.len();
        let s: Vec<f32> = (0..n)
            .map(|k| self.delays[k][self.ptrs[k]])
            .collect();
        let sg: Vec<f32> = s.iter().zip(&self.gains)
            .map(|(si, gi)| si * gi).collect();
        let mut mixed = [0.0f32; 4];
        for i in 0..4 {
            for j in 0..4 {
                mixed[i] += self.hadamard[i][j] * sg[j];
            }
        }
        let b = 1.0 / (n as f32).sqrt();
        for k in 0..n {
            self.delays[k][self.ptrs[k]] = mixed[k] + b * input;
            self.ptrs[k] = (self.ptrs[k] + 1) % self.delays[k].len();
        }
        s.iter().sum::<f32>() / n as f32
    }
}

fn main() {
    let host = cpal::default_host();
    let device = host.default_output_device().unwrap();
    println!("Dispositivo: {}", device.name().unwrap());
}

Feynman — Por qué el hilo de audio nunca debe adquirir un mutex

¿Por qué el hilo de audio en C++ nunca debe adquirir un mutex?

Ver respuesta

El hilo de audio opera con una deadline estricta: si el buffer de 64 samples a 44.1kHz no se llena en 1.45ms, hay un glitch audible (xrun). Un mutex puede bloquearse por tiempo indeterminado si otro hilo lo sostiene → el callback no retorna a tiempo → xrun. La solución es usar estructuras lock-free: ring buffers atómicos (SPSC queue) para pasar mensajes de la GUI al hilo de audio, y doble buffering para parámetros. En Rust, el compilador ayuda a verificar la seguridad de estos patrones.

Quiz rápido

¿Qué ventaja ofrece SIMD AVX2 sobre código escalar para procesamiento de audio?

Ver respuesta

SIMD AVX2 procesa 8 floats simultáneamente en una sola instrucción (8x throughput teórico). Para operaciones típicas de DSP (FIR, convolución, síntesis) que aplican la misma operación a muchas muestras consecutivas, esto reduce dramáticamente el tiempo de cómputo. Con AVX-512 el factor es 16x. En práctica, el speedup efectivo es menor por overhead de loads/stores, pero típicamente 3-6x sobre código escalar.

Feynman — Por qué Faust compila mejor que C++ manual a veces

¿Por qué Faust compila a C++ optimizado mejor que código C++ escrito manualmente a veces?

Ver respuesta

Faust tiene un compilador específico de dominio que conoce la semántica del DSP de audio: sabe que ciertas operaciones son commutativas en el dominio de señal, que las constantes en la tasa de audio pueden pre-calcularse, y puede aplicar transformaciones algebraicas que clang/gcc no puede hacer por falta de conocimiento de dominio. Puede desenrollar bucles óptimamente para SIMD, reorganizar el grafo de señal para minimizar dependencias de datos, y eliminar operaciones identidad. Además, el grafo Faust es puro (sin efectos secundarios), lo que permite análisis global imposible en C++.

Realiza la convolución en tiempo real con baja latencia mediante UPOLS particionado. import numpy as np

def schroeder_curve(rir, sr):
    energy = rir ** 2
    cum_energy = np.cumsum(energy[::-1])[::-1]
    cum_norm = cum_energy / (cum_energy[0] + 1e-12)
    return 10 * np.log10(cum_norm + 1e-12)

def rt60(rir, sr, method='edt'):
    curve = schroeder_curve(rir, sr)
    t = np.arange(len(curve)) / sr
    if method == 'edt':
        i1 = np.argmin(np.abs(curve - 0))
        i2 = np.argmin(np.abs(curve - (-10)))
    elif method == 'T20':
        i1 = np.argmin(np.abs(curve - (-5)))
        i2 = np.argmin(np.abs(curve - (-25)))
    slope = (curve[i2] - curve[i1]) / (t[i2] - t[i1])
    return -60 / slope

def clarity_c80(rir, sr):
    n80 = int(0.080 * sr)
    e_early = np.sum(rir[:n80] ** 2)
    e_late  = np.sum(rir[n80:] ** 2) + 1e-12
    return 10 * np.log10(e_early / e_late)

def definition_d50(rir, sr):
    n50 = int(0.050 * sr)
    return np.sum(rir[:n50]**2) / (np.sum(rir**2) + 1e-12)

def centre_time_ts(rir, sr):
    t = np.arange(len(rir)) / sr
    e = rir ** 2
    return np.sum(t * e) / (np.sum(e) + 1e-12)

def iso3382_metrics(rir, sr):
    return {
        'RT60_EDT': rt60(rir, sr, 'edt'),
        'RT60_T20': rt60(rir, sr, 'T20'),
        'C80_dB':   clarity_c80(rir, sr),
        'D50':      definition_d50(rir, sr),
        'Ts_ms':    centre_time_ts(rir, sr) * 1000
    }

sr = 44100
t = np.linspace(0, 2, 2*sr)
rir_synth = np.exp(-3*t) * np.random.randn(len(t))
rir_synth[0] = 1.0
metrics = iso3382_metrics(rir_synth, sr)
for k, v in metrics.items():
    print(f"  {k}: {v:.3f}")

Feynman — Por qué EDT suele diferir del RT60 en salas reales

¿Por qué el EDT suele ser diferente del RT60 en salas reales?

Ver respuesta

El RT60 se extrapola de la pendiente en -5 a -25/-35dB, correspondiente a la cola tardía donde el campo es estadísticamente difuso. El EDT se extrapola del inicio de la curva (0 a -10dB), dominado por las primeras reflexiones y fuertemente dependiente de la posición fuente-receptor. En una sala ideal perfectamente difusa, EDT = RT60. En salas reales, si el oyente está cerca de la fuente, el campo local no es difuso y EDT puede diferir significativamente de RT60.

Quiz rápido

¿Qué indica un valor de C80 = -5 dB en una sala de conciertos?

Ver respuesta

Que el sonido tardío (>80ms) domina: la sala es muy reverberante y puede sonar "empastada". C80 negativo significa que hay más energía tardía que temprana. Para música de cámara y ópera, se prefiere C80 positivo (+2 a +4 dB) para claridad. Para música sinfónica romántica, C80 levemente negativo (-2 a 0 dB) da la sensación de envolvente deseada.

Examen Feynman — Explica sin notas

Q1 · Schroeder y el límite de ray tracing

¿Por qué ray tracing no funciona por debajo de la frecuencia de Schroeder y qué método alternativo es exacto allí?

Ver respuesta

Ray tracing asume que el sonido viaja como partícula con longitud de onda despreciable respecto a la geometría. Por debajo de fs, la longitud de onda es comparable o mayor que las dimensiones de la sala: el sonido se difracta por cualquier borde, los modos propios crean interferencias complejas que no se pueden aproximar con rayos. Solo solvers wave-based (FDTD — Finite Difference Time Domain, o BEM — Boundary Element Method) son exactos allí. En práctica, se usa ISM para reflexiones tempranas y ray tracing estadístico para la cola tardía por encima de fs.

Q2 · FDN y la matriz ortogonal

¿Qué pasa con el FDN si la matriz de mezcla no es ortogonal? Da un ejemplo de matriz válida e inválida.

Ver respuesta

Si |det(A)| < 1: la energía decrece en cada ciclo de mezcla además de por las ganancias g_k → RT60 menor al diseñado. Si |det(A)| > 1: la energía crece en cada ciclo → inestabilidad → ruido explosivo. Solo |det(A)| = 1 (ortogonal) preserva la energía exactamente. Ejemplo válido: la matriz de Hadamard normalizada (det = ±1). Ejemplo inválido: una matriz identidad escalada (I × 1.1) tiene det = 1.1⁴ > 1 → inestabilidad.

Q3 · HRTF y personalización

¿Por qué una HRTF genérica causa front-back confusion y cómo lo resuelve HRTFformer? ¿Cuántos coeficientes SH se necesitan para L=4?

Ver respuesta

Las pinna notches (filtros espectrales de la oreja) que distinguen adelante/atrás son únicas para cada persona. Una HRTF genérica tiene notches en posiciones frecuenciales distintas a las tuyas: el cerebro no las interpreta como señales de localización válidas. HRTFformer predice tus pinna notches específicas desde medidas de tu oreja (17 medidas antropométricas CIPIC), produciendo una HRTF personalizada cuyas notches coinciden con tu morfología. Para L=4: (L+1)² = (4+1)² = 25 coeficientes SH.

Q4 · NUPOLS y latencia mínima

¿Por qué NUPOLS puede tener menor latencia que UPOLS para la misma RIR? Calcula la latencia de NUPOLS con L₀=64 a 44.1kHz.

Ver respuesta

UPOLS procesa todos los bloques con el mismo tamaño L → latencia = L. Para una RIR larga, si L es pequeño para baja latencia, hay muchas FFTs pequeñas → alta CPU. Con L grande para reducir CPU, la latencia aumenta. NUPOLS usa L₀ pequeño solo para el primer bloque (sonido directo + early reflections, perceptualmente críticos) y bloques mayores para la cola tardía procesados en hilos separados. La latencia total es L₀/sr = 64/44100 ≈ 1.45 ms (mínima), independientemente de la longitud total de la RIR.

Q5 · Validación ISO 3382

¿Por qué C80 es la métrica más importante para validar la simulación de una sala de cámara? ¿Qué método usa alta SNR con equipo de consumo para medir RIRs?

Ver respuesta

C80 mide el balance entre energía temprana (primeras reflexiones <80ms, que aportan claridad e inteligibilidad) y energía tardía (reverberación). Para música de cámara, la articulación de cada nota debe ser audible: C80 determina si la sala permite distinguir staccato de legato. RT60 solo mide cuánto tarda en decaer la energía total pero no cómo se distribuye temporalmente. Un simulador que reproduce RT60 correctamente pero falla en C80 no reproduce la acústica perceptiva real. Para medir RIRs con alta SNR con equipo de consumo: sweep sinusoidal exponencial (ESS) deconvolucionado — el sweep cubre todo el espectro audible y la deconvolución con el sweep inverso extrae la RIR con alta SNR incluso en presencia de ruido de fondo.
