M1 · STFT
Short-Time Fourier Transform
Análisis tiempo-frecuencia: fundamento matemático y artefactos de ventaneo

A Núcleo Teórico

La STFT fragmenta la señal x(n) en tramas cortas superpuestas y aplica la DFT a cada una:

X(m,k) = Σₙ x(n) · w(n − mH) · e^{−j2πkn/N}

donde m=frame, k=bin de frecuencia, H=hop_length, N=tamaño FFT, w=ventana de análisis.

El principio de incertidumbre de Heisenberg limita la resolución simultánea: Δt · Δf ≥ 1/(4π). Aumentar N mejora la frecuencia pero empeora el tiempo.

C Reto HERO — Detector de Onset basado en Spectral Flux

def spectral_flux_onset(y, sr, n_fft=2048, hop=512, threshold=1.5):
    """
    Onset detection via half-wave rectified spectral flux.
    Returns onset times in seconds.
    """
    D    = np.abs(librosa.stft(y, n_fft=n_fft, hop_length=hop))
    diff = np.maximum(0, np.diff(D, axis=1))   # (bins, T-1)
    flux = diff.sum(axis=0)                      # (T-1,)

    window_frames = 43  # ~1 segundo
    onset_frames  = []
    for i in range(len(flux)):
        lo = max(0, i - window_frames // 2)
        hi = min(len(flux), i + window_frames // 2)
        local_mean = flux[lo:hi].mean()
        local_std  = flux[lo:hi].std()
        if flux[i] > local_mean + threshold * local_std:
            onset_frames.append(i)

    times = librosa.frames_to_time(onset_frames, sr=sr, hop_length=hop)
    return times

Q1 · Resolución STFT

Si duplicas el hop_length de 256 a 512 manteniendo n_fft=2048, ¿qué sucede con el espectrograma resultante y por qué?

Ver respuesta

Se reduce el número de frames a la mitad (menos resolución temporal). El hop_length determina cuántas muestras se avanzan entre frame y frame: con hop=512 se generan la mitad de frames que con hop=256, lo que significa que el eje de tiempo del espectrograma tendrá la mitad de puntos. La resolución de frecuencia no cambia porque sigue siendo función de n_fft.

M2 · MFCC
Mel-Frequency Cepstral Coefficients
Pipeline perceptual: Mel filterbank → log → DCT → coeficientes compactos

A Núcleo Teórico

El oído humano percibe el pitch en escala logarítmica (Mel). La escala Mel mapea Hz:

Mel(f) = 2595 · log₁₀(1 + f/700)
f(m) = 700 · (10^{m/2595} − 1)

Pipeline MFCC completo:

Audio → pre-énfasis (0.97) → frames → ventana Hamming → FFT →
|X|² → Mel filterbank → log → DCT → coefs [0..N_mfcc-1]

El coeficiente C₀ representa la energía total (suele descartarse). Los delta MFCC y delta-delta capturan la dinámica temporal:

Δ[t] = (Σₖ k·(c[t+k]−c[t−k])) / (2·Σₖ k²) para k=1..K

Con K=2 el contexto es ±2 frames → vector final: 39 coefs (13+13Δ+13ΔΔ)

Q2 · Log en MFCC

¿Por qué se aplica el logaritmo al filterbank Mel antes de la DCT en el pipeline MFCC?

Ver respuesta

Porque la percepción de volumen es logarítmica (ley de Weber-Fechner). El sistema auditivo humano no percibe la energía de forma lineal: duplicar la energía no duplica la sensación de volumen percibida. Al aplicar el logaritmo se simula esta compresión perceptual, lo que hace los MFCC más robustos a variaciones de volumen y más representativos de cómo el oído "escucha" realmente el espectro.

M3 · Chroma HPCP
Chroma HPCP — Representación Tonal Armónica
Los 12 semitonos del círculo de quintas: invariante a la octava

A Núcleo Teórico

El Chroma HPCP (Harmonic Pitch Class Profile) proyecta toda la energía espectral sobre los 12 semitonos, independientemente de la octava:

HPCP(p) = Σₖ |X(k)|² · cos²(π · (fₖ − f_ref(p)) / f_band) si |fₖ−f_ref(p)| < f_band/2

Donde p∈{0..11} corresponde a C, C#, D, ..., B y f_ref(p) = 440 × 2^{(p-9)/12}

B Laboratorio — Chroma + key detection

import librosa
import numpy as np

y, sr = librosa.load('track.wav', sr=22050)

chroma_cqt = librosa.feature.chroma_cqt(y=y, sr=sr, hop_length=1024, bins_per_octave=36)

MAJOR_PROFILE = np.array([6.35,2.23,3.48,2.33,4.38,4.09,2.52,5.19,2.39,3.66,2.29,2.88])
MINOR_PROFILE = np.array([6.33,2.68,3.52,5.38,2.60,3.53,2.54,4.75,3.98,2.69,3.34,3.17])
mean_chroma = chroma_cqt.mean(axis=1)

def key_detect(chroma_mean):
    best_score, best_key, best_mode = -np.inf, 0, 'major'
    for shift in range(12):
        for profile, mode in [(MAJOR_PROFILE, 'major'), (MINOR_PROFILE, 'minor')]:
            rotated = np.roll(profile, shift)
            score = np.corrcoef(chroma_mean, rotated)[0, 1]
            if score > best_score:
                best_score, best_key, best_mode = score, shift, mode
    NOTES = ['C','C#','D','D#','E','F','F#','G','G#','A','A#','B']
    return f"{NOTES[best_key]} {best_mode}", best_score

key, confidence = key_detect(mean_chroma)
print(f"Tonalidad detectada: {key} (r={confidence:.3f})")

Q3 · Invariancia a la Octava

¿Por qué el Chroma es "invariante a la octava" y qué implicación práctica tiene para cover song identification?

Ver respuesta

El Chroma suma la energía de todos los armónicos del mismo nombre de nota (C4, C5, C6...) en un solo bin. Esto significa que un do en cualquier octava se acumula en el bin 0 del vector Chroma. Para cover song ID, esto es crucial porque un cover puede tocar la misma canción en una octava diferente — al comparar vectores Chroma, las estructuras armónicas serán similares independientemente de la octava en que se ejecuten.

M4 · SSM
Self-Similarity Matrix
Estructura musical como mapa: repeticiones, puentes y secciones reflejadas

A Núcleo Teórico

La SSM compara cada frame con todos los demás. Si los vectores de features están normalizados:

S = X^T · X (cosine similarity, X columnas normalizadas)
S(i,j) = cos(θ) = x_i · x_j / (‖x_i‖ · ‖x_j‖)

Diagonal principalAutocorrelación (siempre 1)
Diagonal paralela brillanteSección repetida (verso, chorus)
Cuadrado brillanteSección estacionaria (drones, pads)
Cuadrado oscuroCambio brusco de sección (drop)
Patrón checkerboardAlternancia A-B-A-B

Q4 · Patrones SSM

En la SSM de una canción pop con estructura Intro-Verso-Chorus-Verso-Chorus-Outro, ¿qué patrón visual esperas ver y por qué?

Ver respuesta

Se esperan cuadrados brillantes repetidos en la diagonal correspondientes a chorus y versos. Cuando una sección se repite (por ejemplo el chorus en el minuto 1 y el minuto 3), sus frames son similares entre sí, produciendo un cuadrado brillante fuera de la diagonal principal que conecta esos dos momentos. El patrón general resultante es una serie de bloques cuadrados brillantes que revelan la estructura repetitiva de la canción.

M5 · Beat Tracking
Beat Tracking — Programación Dinámica
Estimación de tempo y beats: onset function + DP de Ellis (2007)

A Núcleo Teórico

El algoritmo de Ellis plantea el beat tracking como un problema de optimización global:

C*(t) = O(t) + max_τ { α · F(t−τ, τp) + C*(τ) }
F(Δt, τp) = −(log₂(Δt/τp))²

O(t)=onset strength, τp=período de beat esperado, α=peso de regularidad. La penalización F es cero cuando Δt=τp y cae como parábola en escala log.

B Laboratorio — Beat tracking + grid de cuantización

import librosa
import numpy as np

y, sr = librosa.load('track.wav', sr=22050)
tempo, beat_frames = librosa.beat.beat_track(
    y=y, sr=sr, start_bpm=120.0, tightness=100, units='frames', hop_length=512
)
beat_times = librosa.frames_to_time(beat_frames, sr=sr, hop_length=512)
print(f"Tempo estimado: {tempo:.1f} BPM")

def quantize_to_grid(event_times, beat_times, subdivisions=4):
    sub_grid = []
    for i in range(len(beat_times)-1):
        sub_grid.extend(np.linspace(beat_times[i], beat_times[i+1], subdivisions, endpoint=False))
    sub_grid = np.array(sub_grid)
    quantized = sub_grid[np.argmin(np.abs(event_times[:,None] - sub_grid[None,:]), axis=1)]
    return quantized

Q5 · Escala Logarítmica en Beat Tracking

¿Por qué la penalización F(Δt, τp) = −(log₂(Δt/τp))² usa escala logarítmica y no lineal?

Ver respuesta

Porque un beat al doble del período (τ = 2τp, cambio de octava rítmica) es más aceptable que uno a 1.5τp; la escala log da igual penalización a ratios iguales. En términos musicales, el oído percibe las subdivisiones rítmicas de forma multiplicativa: tocar en los tiempos dobles o la mitad es naturalmente más aceptable que tocar "a destiempo". La escala log captura esta simetría perceptual multiplicativa.

Q6 · Data Leakage en Pipelines

¿Por qué aplicar StandardScaler dentro de un Pipeline en lugar de antes del cross-validation?

Ver respuesta

Para que el scaler se ajuste solo en el training fold durante cross-validation, evitando data leakage. Si se escala antes de la CV, el scaler "ve" los datos de test durante el ajuste (fit) porque usa estadísticas de todo el dataset. Dentro del Pipeline, el scaler se entrena únicamente con los datos de train de cada fold, y los datos de test se transforman con esas estadísticas — simulando correctamente un escenario de producción donde el test set es desconocido.

Q7 · Transfer Learning y Canales RGB

¿Por qué se convierte el mel-spectrograma a imagen RGB de 3 canales al usar ResNet pre-entrenada en ImageNet?

Ver respuesta

Porque ResNet espera 3 canales de entrada (RGB) por su primera capa Conv2d con in_channels=3; duplicar el canal permite usar los pesos pre-entrenados. Si se intenta pasar un tensor de 1 canal, la primera capa Conv2d no es compatible dimensionalmente. Al convertir a RGB (duplicando el canal en los 3), se puede aprovechar todo el aprendizaje de representación visual de ImageNet aplicado a patrones tiempo-frecuencia del audio.

import librosa, numpy as np, glob

def analyze_stream(path, sr=22050, block_length=2048, hop_length=512):
    stream = librosa.stream(path, block_length=block_length,
                             frame_length=2048, hop_length=hop_length, mono=True)
    rms_all = []
    for y_block in stream:
        rms = librosa.feature.rms(y=y_block, frame_length=2048, hop_length=hop_length)[0]
        rms_all.extend(rms.tolist())
    return np.array(rms_all)

y, sr = librosa.load('mix.wav', sr=22050)
D     = librosa.stft(y, n_fft=2048, hop_length=512)
D_h, D_p = librosa.decompose.hpss(D, kernel_size=31, power=2.0, margin=1.0)
y_harmonic   = librosa.istft(D_h, hop_length=512)
y_percussive = librosa.istft(D_p, hop_length=512)

def batch_analyze(root_dir, extensions=('wav','mp3','flac')):
    results = {}
    for ext in extensions:
        for path in glob.glob(f"{root_dir}/**/*.{ext}", recursive=True):
            try:
                y, sr = librosa.load(path, sr=22050, mono=True, duration=30.0)
                tempo, _ = librosa.beat.beat_track(y=y, sr=sr)
                results[path] = {'tempo': float(tempo), 'duration': len(y)/sr}
            except Exception as e:
                print(f"Error {path}: {e}")
    return results

Q8 · HPSS — Operación Matemática

¿Qué operación matemática realiza HPSS sobre el espectrograma de magnitud para separar las componentes armónica y percusiva?

Ver respuesta

Filtrado de mediana a lo largo del eje de tiempo (→ armónico, produce líneas horizontales) y de frecuencia (→ percusivo, produce líneas verticales). Las componentes armónicas (tonos sostenidos) crean estructuras horizontales persistentes en el espectrograma, mientras que los transientes percusivos crean estructuras verticales instantáneas. El filtro de mediana horizontal suaviza los transientes conservando lo armónico, y viceversa para el filtro vertical.

Q9 · Streaming vs Standard en Essentia

¿Qué ventaja tiene el modo essentia.streaming sobre essentia.standard para archivos de audio muy largos?

Ver respuesta

Construye un grafo de procesamiento que evita buffers intermedios y permite procesar archivos de cualquier longitud con memoria constante. En el modo standard, cada algoritmo carga el array completo en memoria. En el modo streaming, los datos fluyen de un nodo al siguiente en el grafo sin almacenar el audio completo, lo que es esencial para archivos de horas de duración que no caben en RAM.

Q10 · Limitaciones WASM y Archivos Grandes

Un usuario sube un archivo de 300MB a tu app web con Essentia.js. ¿Qué problema técnico crítico encontrarás y cómo se resuelve?

Ver respuesta

El heap de WASM está limitado por la memoria disponible del tab (~1-4GB); un Float32Array de 300MB de audio más buffers intermedios puede agotar la memoria y crashear el tab. La solución es usar el Web Audio API AudioWorklet para procesar el audio por bloques de forma incremental, o limitar la duración del análisis a los primeros 30-60 segundos del archivo. Los buffers de análisis STFT intermedios pueden ser 3-10× el tamaño del audio original.

Q11 · PCA antes de t-SNE

¿Por qué se aplica PCA a 50 dimensiones ANTES de t-SNE en lugar de aplicar t-SNE directamente a los 720 features?

Ver respuesta

t-SNE escala mal en dimensionalidad alta (O(N²D)); PCA reduce el ruido y acelera t-SNE sin perder estructura relevante. Con 720 features, la matriz de distancias que t-SNE necesita calcular tiene N² × 720 operaciones — computacionalmente prohibitivo para colecciones grandes. PCA a 50 dimensiones preserva la mayor parte de la varianza explicada (generalmente más del 85%) y hace t-SNE factible manteniendo la estructura de clusters.

Q12 · Cover Song ID con Chroma Circular

En Cover Song ID con Chroma, ¿cómo detectas que dos pistas son el mismo tema aunque una esté transpuesta 3 semitonos arriba?

Ver respuesta

Calculando la correlación cruzada circular del vector Chroma: buscar el shift (0-11) que maximiza la correlación; un alto máximo indica cover. Al correlacionar circularmente se prueban todas las transposiciones posibles (los 12 semitonos): si la correlación máxima en algún desplazamiento es alta (empíricamente > 0.85), los dos temas comparten la misma estructura armónica con diferente tonalidad — señal robusta de cover.

Q13 · NamedTemporaryFile y Cross-Platform

¿Por qué se usa un NamedTemporaryFile con delete=False y se elimina manualmente en finally en el endpoint FastAPI?

Ver respuesta

En Windows, un NamedTemporaryFile con delete=True no puede ser abierto por otro proceso mientras está abierto; delete=False + finally garantiza limpieza incluso si hay excepciones. Cuando librosa intenta abrir el archivo temporal por su nombre en Windows, el sistema operativo rechaza el acceso porque el archivo ya está abierto por el contexto del NamedTemporaryFile. Al usar delete=False, se cierra el archivo primero y luego librosa puede accederlo, eliminándolo manualmente en el bloque finally para evitar fugas de archivos temporales.

Q1 · STFT y Resolución

Quieres detectar la nota La4 (440 Hz) con ±5 Hz de precisión usando STFT a 22050 Hz. ¿Cuál es el n_fft mínimo práctico y cómo se calcula?

Ver respuesta

Resolución frecuencial = sr/n_fft. Para <5 Hz/bin: n_fft > 22050/5 = 4410. El siguiente valor práctico (potencia de 2) es n_fft=8192, que da 22050/8192 = 2.69 Hz/bin. En la práctica n_fft=4096 da 5.38 Hz/bin, la opción más común de compromiso. La penalización: mayor n_fft implica mayor ventana temporal y peor resolución en el eje de tiempo (el principio de incertidumbre de Heisenberg aplicado al análisis espectral).

Q2 · DCT en MFCC

¿Qué garantiza la DCT en el pipeline MFCC que la hace superior al simple log-Mel para modelos de ML como SVM o XGBoost?

Ver respuesta

Decorrelaciona los filtros Mel solapados: los MFCC son casi ortogonales entre sí, lo que beneficia modelos que asumen independencia de features. Los filtros Mel se solapan entre sí por diseño, creando alta correlación entre bandas adyacentes. La DCT (tipo II) compacta la energía en los primeros coeficientes y elimina esta correlación, produciendo features aproximadamente ortogonales. SVM con kernel RBF y XGBoost funcionan mejor con features menos correlacionadas y con la información concentrada en pocas dimensiones.

Q3 · Beat Tracking y Métricas Complejas

El algoritmo de Ellis (DP) falla en música con métricas complejas como 5/4 o 7/8. ¿Por qué el modelo matemático es fundamentalmente incompatible con estas métricas?

Ver respuesta

Asume que el período inter-beat es constante (τp fijo); la penalización log-parabólica favorece ratios de 1:2 (subdivisión binaria) pero penaliza fuertemente los ratios irregulares de 5/4 o 7/8. La función F(Δt,τp)=-(log₂(Δt/τp))² tiene mínimos exactamente en τp, τp/2, 2τp (octavas rítmicas) — todos múltiplos o divisores binarios. En 5/4, los beats tienen 5 subdivisiones que no encajan en este modelo, generando errores de cuantización sistemáticos que no se pueden corregir ajustando parámetros.

Q4 · CNN y Augmentación de Datos

¿Cuál de estas augmentaciones de espectrograma es físicamente incorrecta para música y por qué podría dañar el entrenamiento?

Ver respuesta

FlipVertical: invertir el eje de frecuencia (graves a agudos y viceversa). Esto crea un espectrograma físicamente imposible: los armónicos aparecen por debajo del fundamental en lugar de por encima, lo que contradice las leyes acústicas. Un instrumento con fundamental a 100 Hz tiene armónicos a 200, 300, 400 Hz... invertir el espectrograma pone esas energías a 9850, 9750, 9650 Hz (con sr=22050). El modelo entrenaría con ejemplos que no pueden ocurrir en la realidad, perjudicando la generalización.

Q5 · Essentia.js y Limitaciones de Modelos

¿Qué limitación principal tiene Essentia.js vs Essentia Python para inferencia de modelos de clasificación musical pre-entrenados?

Ver respuesta

Los modelos TF.js deben ser más pequeños (Int8/Float16) y convertidos al formato SavedModel/GraphDef compatible con TF.js; no puede usar directamente modelos .h5 de Keras. Los modelos entrenados en Python con TensorFlow/Keras usan Float32 por defecto y se guardan en formatos como .h5 o SavedModel de TF2. Para TF.js se necesita una conversión con tensorflowjs_converter que además puede aplicar cuantización para reducir el tamaño, perdiendo algo de precisión. Adicionalmente, algunas operaciones de TF no están implementadas en TF.js, limitando qué arquitecturas son compatibles.
