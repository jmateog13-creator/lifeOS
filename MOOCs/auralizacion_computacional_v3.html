M1 · FÍSICA DE SALAS

Modos Acústicos, Schroeder y RT60

La física de salas determina la estructura modal a baja frecuencia y el comportamiento difuso a alta frecuencia, separados por la frecuencia de Schroeder.

A · NÚCLEO TEÓRICO

Modos y Frecuencia de Schroeder

Los modos propios de una sala rectangular se calculan: f_lmn = (c/2)·√[(l/Lx)² + (m/Ly)² + (n/Lz)²]. Por encima de la frecuencia de Schroeder el comportamiento es estadísticamente difuso:

f_s = 2000·√(RT60/V) | RT60 = 0.161·V/(S·ᾱ) (Sabine)
RT60 = 0.161·V/(−S·ln(1−ᾱ)) (Eyring, ᾱ > 0.3)

Código representativo

import numpy as np

def schroeder_freq(RT60, V):
    return 2000 * np.sqrt(RT60 / V)

def sabine_rt60(V, S_alpha_pairs):
    total_absorption = sum(S * a for S, a in S_alpha_pairs)
    return 0.161 * V / total_absorption

# Sala de conciertos pequeña: 15×10×5m
Lx, Ly, Lz = 15, 10, 5
V = Lx * Ly * Lz  # 750 m³
surfaces = [(2*Lx*Lz, 0.02), (2*Ly*Lz, 0.02), (Lx*Ly, 0.10), (Lx*Ly, 0.05)]
RT60 = sabine_rt60(V, surfaces)
fs = schroeder_freq(RT60, V)
print(f"RT60 Sabine: {RT60:.2f}s  |  Schroeder: {fs:.1f} Hz")

Q1 · Frecuencia de Schroeder

¿Por qué Boston Symphony Hall (V=18.740m³, RT60=1.85s) tiene f_s de solo ~2 Hz y se considera acústicamente perfecto?

Ver respuesta

Su enorme volumen produce f_s = 2000·√(1.85/18740) ≈ 2 Hz. Por encima de 2 Hz ya hay campo difuso perfecto con densidad de 6+ modos/Hz, sin coloración modal audible. En una sala pequeña (V=50m³) f_s sube a ~200 Hz y los modos individuales colorean el sonido.

M2 · ISM Y PYROOMACOUSTICS

Método de Fuentes Imagen y RIR Computacional

El ISM calcula reflexiones especulares deterministas creando fuentes virtuales espejadas. Complejidad O(N^m) limita el uso a órdenes 3-4.

A · NÚCLEO TEÓRICO

Fuentes Imagen y Atenuación

Cada reflexión de orden m crea una fuente imagen posicionada simétricamente respecto a la superficie. La presión decae con la ley del inverso del cuadrado: P(d) = 1/d².

Complejidad ISM = O(N^m) → para N=6, m=3: 216 fuentes imagen
import pyroomacoustics as pra
import numpy as np

room_dim = [9.0, 7.0, 3.5]
source_pos = [4.5, 1.0, 1.8]
mic_pos = [[4.5, 6.0, 1.2]]

e_absorption, max_order = pra.inverse_sabine(1.2, room_dim)
room = pra.ShoeBox(room_dim, fs=44100,
                   materials=pra.Material(e_absorption),
                   max_order=3, ray_tracing=True)
room.add_source(source_pos)
room.add_microphone(np.array(mic_pos).T)
room.simulate()
rir = room.rir[0][0]
print(f"RIR: {len(rir)/44100*1000:.0f}ms")

Q2 · ISM vs Ray Tracing

¿Por qué el ISM es preferible al ray tracing para reflexiones tempranas pero no para la cola reverberante?

Ver respuesta

El ISM calcula reflexiones deterministas exactas (cada reflexión llega en tiempo y amplitud precisos), fundamentales para la percepción del espacio (efecto Haas, C80). La cola reverberante tardía es estadísticamente difusa: el ray tracing modela bien la densidad estadística sin el costo combinatorio O(N^m) del ISM de alto orden.

M3 · FDN Y HADAMARD

Feedback Delay Networks con Matriz Ortogonal

Las FDN generan reverberación densa e incolora mediante líneas de retardo interconectadas por una matriz Hadamard ortogonal, garantizando preservación de energía.

A · NÚCLEO TEÓRICO

Arquitectura FDN y Condición Colorless

s_i(n) = g_i · z^{−m_i} · (Σ_j A_ij · s_j(n) + b_i · x(n))
H_2 = (1/√2)·[[1,1],[1,−1]] | g_i = 10^(−3·m_i/(RT60·fs))

Para reverberación incolora, la matriz A debe ser ortogonal (A·Aᵀ = I). La Hadamard cumple esto con entradas ±1/√N. Longitudes de retardo primas maximizan la densidad de eco.

import numpy as np

def hadamard_matrix(N):
    if N == 1: return np.array([[1.0]])
    H_half = hadamard_matrix(N // 2)
    top = np.hstack([H_half, H_half])
    bot = np.hstack([H_half, -H_half])
    return np.vstack([top, bot]) / np.sqrt(2)

delays = [1381, 1481, 1583, 1699, 1801, 1867, 2053, 2129]  # primos
rt60 = 1.5; fs = 44100
gains = np.array([10**(-3*d/(rt60*fs)) for d in delays])
A = hadamard_matrix(8)
print(f"Hadamard ortogonal: A·Aᵀ ≈ I = {np.allclose(A @ A.T, np.eye(8))}")

  • ¿Cómo se controla el RT60 en FDN? → Ajustando ganancias g_i = 10^(-3·m_i/(RT60·fs))
  • ¿Cuántos canales son típicos para juegos? → 12–24 canales
  • ¿Diferencia entre FDN y convolución con RIR? → FDN: sintética, eficiente; convolución: exacta pero costosa

Q3 · Densidad de Eco

¿Por qué usar longitudes de retardo primas (1381, 1481...) en lugar de múltiplos regulares en una FDN?

Ver respuesta

Con longitudes múltiplas regulares (ej. 1000, 2000, 3000), los retardos se alinean periódicamente, creando picos de energía en los instantes múltiplos del MCD. Con longitudes primas sin factores comunes, los patrones de coincidencia son mucho más raros y la densidad de eco crece uniformemente. El resultado es reverberación suave sin fluttering ni resonancias metálicas.

M4 · NUPOLS CONVOLUCIÓN

Non-Uniform Partitioned Overlap-Save

NUPOLS permite convolución eficiente de RIR largas combinando particiones cortas (baja latencia) con particiones largas (alta eficiencia espectral), consiguiendo O(N·logN).

A · NÚCLEO TEÓRICO

Partitioned Overlap-Save

La convolución directa de una RIR de N muestras con un bloque de audio de B muestras cuesta O(N·B). La convolución OLS en el dominio frecuencial cuesta O((B+N)·log(B+N)) por bloque. NUPOLS divide la RIR en K particiones de longitud creciente: las primeras particiones son cortas (baja latencia para reflexiones tempranas) y las últimas largas (eficientes para la cola reverberante).

Latencia total = tamaño de la primera partición → puede ser 64–512 muestras
Costo = O(N·logN) vs O(N²) para convolución directa
from scipy.signal import fftconvolve
import numpy as np

def overlap_save_block(signal, h, block_size=512):
    """Convolución OLS básica."""
    N = len(h)
    L = block_size
    M = L + N - 1  # tamaño FFT
    H = np.fft.rfft(h, M)
    output = np.zeros(len(signal))
    prev = np.zeros(N - 1)
    for i in range(0, len(signal), L):
        block = np.concatenate([prev, signal[i:i+L]])
        block = np.pad(block, (0, M - len(block)))
        Y = np.fft.rfft(block) * H
        y = np.fft.irfft(Y)
        end = min(i+L, len(signal))
        output[i:end] = y[N-1:N-1+(end-i)]
        prev = signal[max(0,i):i+L][-N+1:] if i+L < len(signal) else np.zeros(N-1)
    return output
D · DRILLS
  • ¿Qué significa "latencia de primera partición" en NUPOLS? → El delay mínimo antes de que el primer bloque de salida esté disponible
  • ¿Por qué NUPOLS mezcla particiones cortas y largas? → Cortas = baja latencia; largas = eficiencia computacional para la cola
  • ¿Cuándo usar convolución directa vs OLS? → Directa: IRs cortísimas (<100 samples); OLS: IRs largas
  • ¿Qué es el solapamiento (overlap) en OLS? → La región de N-1 muestras del bloque anterior que se pre-pende al bloque actual

Q4 · OLS vs OLA

¿Cuál es la diferencia clave entre Overlap-Save (OLS) y Overlap-Add (OLA) para convolución en tiempo real?

Ver respuesta

En OLS, se descarta el solapamiento al inicio de la salida (las primeras N-1 muestras de cada bloque FFT son "artefactos" que se eliminan). En OLA, se suman los solapamientos de bloques consecutivos de salida. Ambos producen resultados idénticos, pero OLS es más eficiente porque no requiere buffers de suma adicionales: simplemente se selecciona la región válida de la IFFT. OLS es el preferido para implementaciones en tiempo real.

M5 · DIFRACCIÓN Y OCLUSIÓN

Difracción de Keller y Modelos de Oclusión

La difracción de bordes (UTD/Keller) modela el sonido que dobla esquinas. Los modelos de oclusión con LPF dinámico simulan el efecto de objetos interpuestos entre fuente y oyente.

A · NÚCLEO TEÓRICO

Uniform Theory of Diffraction (UTD)

Cuando un rayo acústico incide sobre una arista, una componente difractada se propaga desde la arista en todas direcciones. La amplitud del campo difractado (Keller):

p_diff = D(θ_i, θ_d, φ) · p_inc · e^{jkr} / √r

Donde D es el coeficiente de difracción que depende de los ángulos de incidencia y difracción. En juegos, la oclusión se modela aplicando un LPF cuya frecuencia de corte varía según el porcentaje de bloqueo del ray-cast:

f_cutoff = f_max · (1 − occlusion_factor) → 0 Hz (silencio) a 20 kHz (libre)
import numpy as np
from scipy.signal import butter, filtfilt

def dynamic_occlusion_lpf(audio, occlusion_0_1, fs=44100):
    """
    Aplica LPF cuya fc varía según oclusión (0=libre, 1=totalmente ocluido).
    """
    fc = max(200, 20000 * (1 - occlusion_0_1))
    b, a = butter(2, fc / (fs/2), btype='low')
    gain = 1.0 - 0.9 * occlusion_0_1  # atenuación adicional
    return filtfilt(b, a, audio) * gain

# Uso: objeto a 70% de oclusión
# audio_occluded = dynamic_occlusion_lpf(audio_dry, 0.7)

  • ¿Qué hace el LPF de oclusión perceptivamente? → Simula el efecto de amortiguación de paredes/objetos (el sonido "suena desde lejos o a través de paredes")
  • ¿Por qué usar múltiples ray-casts para oclusión? → Un solo rayo da oclusión binaria (0/1); múltiples rayos producen oclusión gradual más realista

Q5 · Oclusión Dinámica

¿Por qué una puerta entreabierta no produce silencio total aunque bloquee el 80% del área de paso directo?

Ver respuesta

El sonido es una onda, no un rayo. Las ondas se difractan alrededor de los bordes de la puerta: la apertura actúa como una nueva fuente secundaria (principio de Huygens). Además, el sonido puede transmitirse a través de la pared y llegar por reflexiones indirectas. Un modelo con múltiples ray-casts captura la fracción que pasa; la difracción del borde añade energía adicional especialmente en bajas frecuencias (longitud de onda similar al tamaño de la apertura).

M6 · HRTF Y ARMÓNICOS SH

Head-Related Transfer Function y Armónicos Esféricos

La HRTF codifica la información binaural de dirección del sonido. Los Armónicos Esféricos (SH) permiten representar y convolver campos sonoros 3D de forma eficiente.

A · NÚCLEO TEÓRICO

HRTF y Representación SH

La HRTF es la función de transferencia entre un punto del espacio libre y el canal auditivo del oyente para cada dirección (azimut θ, elevación φ). En el dominio de Armónicos Esféricos de orden L, la HRTF se expande:

H(θ,φ,ω) = Σ_{l=0}^{L} Σ_{m=-l}^{l} H_l^m(ω) · Y_l^m(θ,φ)

Con (L+1)² canales SH. Orden L=1 → 4 canales (B-Format); L=3 → 16 canales; L=7 → 64 canales. A mayor orden, mayor resolución espacial pero más costoso. Steam Audio usa orden L=3 para tiempo real.

import numpy as np

def spherical_harmonics_order_1(azimuth_rad, elevation_rad):
    """B-Format de primer orden: W, X, Y, Z."""
    az, el = azimuth_rad, elevation_rad
    W = 1 / np.sqrt(2)
    X = np.cos(el) * np.cos(az)
    Y = np.cos(el) * np.sin(az)
    Z = np.sin(el)
    return np.array([W, X, Y, Z])

# Fuente a 45° azimut, 30° elevación
az = np.radians(45); el = np.radians(30)
sh_coeffs = spherical_harmonics_order_1(az, el)
print(f"B-Format coeficientes: {sh_coeffs.round(3)}")
D · DRILLS
  • ¿Cuántos canales SH tiene orden L=3? → (3+1)² = 16 canales
  • ¿Qué encode el ITD (Interaural Time Difference)? → Diferencia temporal de llegada entre oído izquierdo y derecho (localización lateral)
  • ¿Qué encode el ILD (Interaural Level Difference)? → Diferencia de nivel entre oídos (efecto sombra de la cabeza, especialmente >1.5 kHz)
  • ¿Por qué las HRTF individualizadas mejoran la localización? → El pabellón auricular de cada persona tiene forma única que modifica el espectro de forma diferente

Q6 · HRTF y Elevación

¿Por qué las HRTF genéricas (no individualizadas) tienen problemas para localizar correctamente la elevación del sonido?

Ver respuesta

La discriminación de elevación depende principalmente de las modificaciones espectrales que produce el pabellón auricular (pinna notches): ondulaciones en el espectro a 4-16 kHz específicas de la dirección vertical. Estas notchs espectrales son altamente individuales (dependen de la forma exacta del pabellón). Una HRTF genérica usa la pinna de otra persona, cuyas notchs caen en frecuencias distintas → el sistema auditivo interpreta estas pistas como elevaciones incorrectas. El ITD e ILD (lateralización) son más robustos porque dependen de la geometría global de la cabeza.

M7 · HRTFFORMER TOKEN SCALING

HRTFformer y Personalización con IA

HRTFformer es un Transformer que predice HRTF individualizadas a partir de fotos del pabellón auricular, usando token scaling para generalizar a nuevas direcciones no medidas.

A · NÚCLEO TEÓRICO

Arquitectura HRTFformer

El modelo recibe como entrada: (1) embeddings de dirección (azimut, elevación en SH), (2) embeddings de antropometría del pabellón (de fotos). El token scaling multiplica los embeddings de dirección por un factor aprendido por cabeza, permitiendo al modelo adaptarse a la geometría individual sin reentrenamiento completo.

HRTF_pred(θ,φ) = Transformer(SH(θ,φ) · scale_head + anthro_embed)

La pérdida de entrenamiento combina MSE espectral (magnitud HRTF) con pérdida de Interaural Phase Difference (IPD) para preservar el ITD. El modelo está preentrenado en la base de datos CIPIC (45 sujetos, 1250 direcciones) y se adapta con solo 3-5 fotos del nuevo usuario.

# Pseudo-código de inferencia HRTFformer
import numpy as np

def predict_hrtf(az_deg, el_deg, ear_photo_embedding,
                  transformer_model):
    """
    Predice HRTF para dirección dada y perfil de oído.
    """
    az, el = np.radians(az_deg), np.radians(el_deg)
    sh_input = compute_sh_embedding(az, el, L=7)  # 64 features

    # Token scaling personalizado
    scaled_sh = sh_input * transformer_model.scale_token

    # Concatenar con embedding antropométrico del oído
    combined = np.concatenate([scaled_sh, ear_photo_embedding])

    # Forward pass del Transformer
    hrtf_complex = transformer_model.forward(combined)
    return hrtf_complex  # Espectro complejo (freq_bins,) para L y R

Q7 · Token Scaling

¿Por qué el "token scaling" permite a HRTFformer adaptarse a nuevos usuarios sin reentrenamiento completo?

Ver respuesta

El token scaling aplica un vector de escala aprendible a los embeddings de dirección SH. Durante fine-tuning de un nuevo usuario, solo se actualizan estos vectores de escala (parámetros reducidos) mientras el resto del Transformer permanece congelado. Esto es un caso de parameter-efficient fine-tuning (como LoRA). El Transformer ya aprendió la "gramática" de la HRTF (cómo la geometría de la cabeza modifica el sonido); el token scaling solo ajusta "el acento" (la antropometría específica del nuevo usuario).

M9 · UNREAL METASOUND

MetaSound y Quartz: DSP en Tiempo Real

MetaSound es el sistema de audio moderno de Unreal Engine 5 con procesamiento determinista en el hilo de audio. Quartz proporciona cuantización musical precisa al beat.

A · NÚCLEO TEÓRICO

MetaSound DSP y Quartz Timing

MetaSound opera en el hilo de audio (sample-rate), no en el hilo de juego (60Hz). Los nodos DSP se implementan en C++ como operadores con inputs/outputs tipados. Quartz cuantiza eventos al beat con compensación de latencia del buffer de audio:

Precisión Quartz: ±(1/fs) segundos → compensación exacta al sample
// MetaSound: nodo FM personalizado (C++)
class FFMSynthOperator : public TExecutableOperator<FFMSynthOperator> {
public:
    virtual void Execute() override {
        auto& Output = *AudioOutput;
        const float fc = *CarrierFreq, fm = *ModFreq;
        const float mi = *ModIndex, amp = *Amplitude;
        for (int32 i = 0; i < Output.Num(); ++i) {
            const float ModSig = FMath::Sin(PhaseMod * 2.0f * PI);
            PhaseMod = FMath::Fmod(PhaseMod + fm/SampleRate, 1.0f);
            const float FreqDev = fc + mi * fc * ModSig;
            Output[i] = amp * FMath::Sin(PhaseCarrier * 2.0f * PI);
            PhaseCarrier = FMath::Fmod(PhaseCarrier + FreqDev/SampleRate, 1.0f);
        }
    }
private:
    TDataReadReference<float> CarrierFreq, ModFreq, ModIndex, Amplitude;
    TDataWriteReference<FAudioBuffer> AudioOutput;
    float SampleRate, PhaseCarrier = 0, PhaseMod = 0;
};

Q9 · Quartz y Latencia de Buffer

¿Por qué Quartz debe compensar la latencia del buffer de audio al cuantizar eventos a beats?

Ver respuesta

El motor de juego corre a 60fps (frame cada 16.7ms) y el audio procesa en bloques de 512 samples a 44.1kHz (bloques de 11.6ms). Cuando el juego dice "dispara el beat 1", ese mensaje llega al hilo de audio en algún punto dentro del bloque actual. Sin compensación, el evento puede dispararse hasta 11ms tarde. Para un tempo de 120 BPM (beat cada 500ms), un error de 11ms es 2.2% del beat: perceptiblemente impreciso. Quartz pre-programa el evento exactamente en qué sample del próximo bloque debe comenzar.

Q10 · MultiRay vs SingleRay

¿Por qué usar 7 ray-casts para oclusión es más realista que un solo rayo central?

Ver respuesta

El sonido es una onda que ocupa un volumen de espacio, no un rayo puntual. Una explosión grande emite energía en todas direcciones: parte puede pasar por encima, por los lados, o difractarse por esquinas del obstáculo. Con un solo rayo, si el obstáculo lo bloquea, la oclusión es 100%. Con 7 rayos en abanico, si 3 de 7 están bloqueados, la oclusión es ~43%: mucho más creíble. En una puerta entreabierta, algunos rayos pasan y otros no → oclusión gradual y perceptivamente realista.

M11 · DSP C++ Y RUST

DSP de Bajo Nivel: SIMD, Rust cpal y Csound/Faust

Para latencias sub-milisegundo y máximo rendimiento, el DSP de auralización se implementa en C++ con SIMD o Rust con gestión explícita de memoria, eliminando overhead del GC.

A · NÚCLEO TEÓRICO

SIMD, Rust y Arquitecturas de Bajo Nivel

SSE: 4 floats × op en 1 ciclo → 4× speedup sobre scalar
AVX-512: 16 floats × op → hasta 16× speedup para FIR convolution

Rust con cpal garantiza que el callback de audio nunca se pause para GC, crítico para latencias <5ms. El borrow checker previene data races en buffers compartidos entre hilos de juego y audio.

// C++ SIMD SSE FIR Filter
#include <immintrin.h>

void fir_simd(const float* in, float* out, int n, const float* h, int nh) {
    const int W = 4;  // SSE: 4 floats
    for (int i = 0; i < n; ++i) {
        __m128 acc = _mm_setzero_ps();
        for (int k = 0; k < nh/W; ++k) {
            __m128 x_v = _mm_loadu_ps(&in[i + k*W]);
            __m128 h_v = _mm_loadu_ps(&h[k*W]);
            acc = _mm_add_ps(acc, _mm_mul_ps(x_v, h_v));
        }
        float r[4]; _mm_storeu_ps(r, acc);
        out[i] = r[0]+r[1]+r[2]+r[3];
    }
}

Q11 · GC y Audio

¿Por qué el garbage collector hace imposible garantizar latencias de audio <5ms en Java o Python?

Ver respuesta

El callback de audio debe completarse en exactamente N/fs segundos (512/44100 = 11.6ms). Si el GC pausa el programa 15ms para limpiar memoria, el callback no termina a tiempo, el buffer de audio se vacía y el DAC reproduce silencio o repetición: se escucha como dropout. En C++ (RAII) y Rust (borrow checker), la memoria se libera deterministamente al salir del scope, sin pausas GC. El callback puede siempre garantizar completarse dentro del presupuesto temporal.

Q12 · Híbrido ISM + FDN

¿Por qué el pipeline liveRAZR usa ISM para los primeros 80ms y FDN para la cola, en lugar de ISM completo?

Ver respuesta

Las primeras 80ms de la RIR son perceptivamente críticas: las reflexiones de 5-30ms definen la imagen del espacio (efecto Haas), intimidad y claridad (C80). El ISM las calcula exactamente. Después de 80ms, el cerebro no puede distinguir reflexiones individuales: percibe solo densidad y decaimiento general. La FDN es computacionalmente mucho más eficiente (12 líneas vs miles de fuentes imagen) y perceptivamente equivalente. La unión de ambos es el equilibrio perfecto realismo-eficiencia.

M13 · VALIDACIÓN ISO 3382

Métricas Acústicas ISO 3382 y Validación de Simulaciones

ISO 3382 define RT60, EDT, C80, D50, Ts e IACC. La curva de Schroeder es la herramienta de cálculo universal para todas estas métricas desde una RIR.

A · NÚCLEO TEÓRICO

Métricas ISO 3382

RT60: decaimiento -60dB · EDT: extrapolación primeros 10dB × 6
C80 = 10·log10[E(0→80ms)/E(80ms→∞)] [dB] (claridad musical)
D50 = E(0→50ms)/E(0→∞) (inteligibilidad de voz)
IACC = max|IACF(τ)| ∈ [0,1] (espacialidad binaural)
import numpy as np
from scipy.stats import linregress

def schroeder_curve(rir):
    energy = rir ** 2
    return 10 * np.log10(np.cumsum(energy[::-1])[::-1] /
                         (np.sum(energy) + 1e-12))

def estimate_rt60(rir, fs, early_db=-5, late_db=-35):
    s_db = schroeder_curve(rir)
    t = np.arange(len(s_db)) / fs
    mask = (s_db <= early_db) & (s_db >= late_db)
    if np.sum(mask) < 10: return None
    slope, *_ = linregress(t[mask], s_db[mask])
    return -60.0 / slope

def clarity_c80(rir, fs):
    n = int(0.08 * fs)
    early = np.sum(rir[:n]**2)
    late = np.sum(rir[n:]**2) + 1e-12
    return 10 * np.log10(early / late)

Q13 · Curva de Schroeder

¿Por qué la curva de Schroeder se calcula como integral inversa (de atrás hacia adelante) de la energía de la RIR?

Ver respuesta

La RIR tiene una forma decreciente pero con fluctuaciones aleatorias de muestra a muestra (ruido). Si calculases la energía acumulada de adelante hacia atrás, las fluctuaciones iniciales harían la estimación del RT60 inestable. La integral inversa (de atrás hacia adelante) suma desde el "final de la reverberación hacia el inicio": el proceso es equivalente a medir cuánta energía "queda" a cada instante. El resultado es una curva suave y monótonamente decreciente de la que puedes hacer regresión lineal para estimar el RT60 con alta precisión.

Q14.1 · Corrientes de Retorno

Tienes una sala con RT60 medido de 2.8s y C80 de -6 dB. ¿Qué acciones de tratamiento acústico prioriza el modelo Pareto para mejorar ambos parámetros simultáneamente?

Ver respuesta

El RT60 alto y el C80 negativo indican exceso de energía reverberante tardía vs reflexiones tempranas. La intervención Pareto es añadir absorción en las superficies de reflexión tardía (paredes laterales traseras, techo posterior): esto reduce RT60 y aumenta C80 simultáneamente al reducir el denominador de la relación E(80ms→∞). Las reflexiones tempranas (primeros 80ms, principalmente de techo delantero y paredes laterales cerca del escenario) no deben absorberse: son las que refuerzan la percepción de claridad.

Q14.2 · HRTF y Cone of Confusion

¿Por qué es imposible distinguir una fuente sonora ubicada a 30° delante de una fuente a 30° detrás usando solo ITD e ILD?

Ver respuesta

El ITD e ILD son funciones del azimut relativo al eje interaural: dos posiciones que están simétricamente adelante/atrás a la misma distancia lateral producen el mismo ITD e ILD (están en el mismo "cono de confusión"). La única clave para resolver esta ambigüedad son las pistas espectrales de la pinna (notchs HRTF a 4-16 kHz), que son distintas para adelante vs atrás. Sin HRTF individualizada, muchos usuarios reportan las fuentes "dentro de la cabeza" o con front-back reversal.

Q14.3 · FDN Colorless

Cambiaste la matriz Hadamard por la matriz identidad en tu FDN de 8 canales. ¿Qué escucharás y por qué?

Ver respuesta

La matriz identidad no mezcla las líneas de retardo entre sí: cada línea retroalimenta solo consigo misma. El resultado son 8 reverberaciones independientes de duración igual a cada retardo (8 "pings" resonantes). Como los retardos son fijos, las resonancias están en frecuencias específicas (f = 1/retardo_en_segundos): el resultado es una reverberación metálica con pitch determinado por los retardos, típico efecto "ducha" o "flanger" de tono fijo. La ortogonalidad de Hadamard evita exactamente esto mezclando equitativamente todas las líneas.

Q14.4 · ISM y Orden de Reflexión

Tu juego en tiempo real tiene presupuesto de 5ms para auralización. ¿Por qué no puedes usar ISM de orden 5 en una sala de 6 superficies?

Ver respuesta

ISM de orden 5 en una sala de 6 superficies genera 6^5 = 7776 fuentes imagen. Calcular la atenuación, delay, y contribución de cada fuente imagen requiere O(7776) operaciones por bloque de audio. A 44.1kHz con bloques de 512 samples (≈11.6ms de ventana), este cálculo excede los 5ms de presupuesto por varios órdenes de magnitud. La solución es ISM de orden 1-2 (6-36 fuentes, perfectamente dentro del presupuesto) para reflexiones tempranas y FDN para la cola.

Q14.5 · Validación y Tolerancias

Tu simulación pyroomacoustics predice RT60=1.2s pero la medición Room EQ Wizard da RT60=1.8s. ¿Cuál es la causa más probable y cómo la corriges?

Ver respuesta

La discrepancia más probable es que el modelo ISM asume reflexiones perfectamente especulares (coeficiente de absorción idealizado), pero la sala real tiene mobiliario, personas y superficies irregulares que añaden scattering difuso. El scattering distribuye la energía de reflexiones especulares en todas direcciones, reduciendo la energía que llega directamente al micrófono y prolongando el decaimiento efectivo. Para corregirlo: añadir coeficiente de scattering a los materiales en pra.Material (combinando e_absorption y scatter), activar ray_tracing=True con hybrid simulation, y calibrar comparando con mediciones reales de la sala.
