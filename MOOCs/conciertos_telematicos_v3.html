¿Por qué NMP es diferente al streaming?

El streaming (Spotify, YouTube) tolera 2-5s de latencia. La música colaborativa en tiempo real requiere <25ms end-to-end para que el músico perciba interacción natural. Por encima de 50ms, la interacción musical se rompe (equivale a 17m de distancia física). TCP introduce Head-of-Line Blocking (HOL): un paquete perdido bloquea toda la cola → obligatorio UDP/RTP para NMP.

Latencia de Fibra Óptica

import math

def fiber_latency_ms(distance_km: float, n_refraction: float = 1.4682) -> float:
    """
    SMF-28 @ 1550nm: n = 1.4682
    Latencia ONE-WAY = distance / (c/n) = distance * n / c
    Aprox: 4.9µs/km
    """
    c = 3e8  # m/s
    v_fiber = c / n_refraction
    latency_s = (distance_km * 1000) / v_fiber
    return latency_s * 1000  # en ms

# Madrid-Barcelona (~620km): 3.04ms one-way
print(f"Madrid-Bcn: {fiber_latency_ms(620):.2f}ms one-way")
# Madrid-NY (~7000km): 34.3ms one-way → RTT ≈ 68ms → imposible NMP
print(f"Madrid-NY: {fiber_latency_ms(7000):.2f}ms one-way")
M2 · Protocolo
UDP/RTP vs TCP — Por qué importa
Head-of-Line Blocking, RTP payload, redundancia FEC
⚠ Regla de oro: NUNCA usar TCP para audio en tiempo real. El HOL Blocking de TCP puede introducir hasta 200ms+ de latencia adicional en condición de pérdida de 1 solo paquete.

TCP Head-of-Line Blocking

TCP garantiza entrega en orden. Si el paquete Nº5 se pierde, los paquetes 6,7,8... se retienen en el buffer del receptor hasta que llega la retransmisión del 5. En audio, este retraso crea un "glitch" amplificado por el RTT de la red.

  • RTP sobre UDP: Los paquetes llegan (o no). Si uno se pierde, el siguiente se procesa igualmente. El PLC rellena el hueco.
  • QUIC (HTTP/3): UDP-based, streams independientes → elimina HOL entre streams. Experimental para NMP.

FEC — Forward Error Correction en JackTrip

# JackTrip con redundancia -r2: envía cada paquete 2 veces (diferido 1 paquete)

# Opciones clave:

AR PLC — Predicción Lineal para Ocultación

import numpy as np

def ar_plc(history: np.ndarray, order: int = 10, frames_to_conceal: int = 1) -> np.ndarray:
    """
    Modelo AR de orden p para predecir muestras perdidas.
    history: últimas muestras conocidas (mínimo order)
    """
    from scipy.signal import lfilter

    # Estimar coeficientes AR (Yule-Walker / Levinson-Durbin)
    r = np.correlate(history, history, mode='full')
    r = r[len(r)//2:][:order+1]

    # Resolver sistema Toeplitz: R*a = r[1:]
    from scipy.linalg import solve_toeplitz
    a = solve_toeplitz(r[:order], -r[1:order+1])

    # Generar frames de ocultación
    buf = list(history[-order:])
    concealed = []
    for _ in range(frames_to_conceal * 64):  # 64 muestras/frame
        y_pred = sum(a[i] * buf[-(i+1)] for i in range(order))
        buf.append(y_pred)
        concealed.append(y_pred)

    # Aplicar fade-out para evitar clics
    result = np.array(concealed)
    fade = np.linspace(1.0, 0.3, len(result))
    return result * fade

class NetEQState:
    """Simplificación de la lógica de decisión de NetEQ (WebRTC)"""
    NORMAL = "normal"
    ACCELERATE = "accelerate"
    EXPAND = "expand"
    PLC = "plc"

    def __init__(self, target_buf_ms=50, fs=48000):
        self.target = int(target_buf_ms * fs / 1000)
        self.buffer = []
        self.state = self.NORMAL

    def decide(self, packet_arrived: bool) -> str:
        buf_len = len(self.buffer)
        if not packet_arrived:
            self.state = self.PLC
        elif buf_len > 2 * self.target:
            self.state = self.ACCELERATE
        elif buf_len < self.target // 2:
            self.state = self.EXPAND
        else:
            self.state = self.NORMAL
        return self.state

Arquitectura Bare-Metal para NMP

// PIC32MZ — Audio NMP bare-metal (pseudocódigo)
// I2S DMA → buffer circular → UDP TX sin sistema operativo

#include <sys/attribs.h>
#include "harmony/v2_06/framework/driver/i2s/drv_i2s.h"

#define SAMPLE_RATE   48000
#define BUFFER_SAMPLES 64
#define UDP_PORT       5005

// Doble buffer para DMA (ping-pong)
static int32_t dma_buf_a[BUFFER_SAMPLES];
static int32_t dma_buf_b[BUFFER_SAMPLES];
static volatile bool buf_ready = false;
static volatile int32_t* active_buf = dma_buf_a;

// ISR de DMA — se ejecuta cada 1.33ms (64/48000)
void __ISR(_DMA0_VECTOR, ipl7SRS) DMA0_Handler(void) {
    // Intercambiar buffers (ping-pong)
    if (active_buf == dma_buf_a) {
        active_buf = dma_buf_b;
        udp_send_async(dma_buf_a, BUFFER_SAMPLES * 4);
    } else {
        active_buf = dma_buf_a;
        udp_send_async(dma_buf_b, BUFFER_SAMPLES * 4);
    }
    buf_ready = true;
    IFS4CLR = _IFS4_DMA0IF_MASK;  // limpiar flag
}

// Configurar DMA periódico I2S
void init_audio_dma(void) {
    DCH0CON  = 0x03;               // prioridad 3
    DCH0ECON = (I2S_RX_IRQ << 8); // trigger en I2S RX
    DCH0SSA  = (uint32_t)&I2SXRXBUF;
    DCH0DSA  = (uint32_t)dma_buf_a;
    DCH0SSIZ = 4;
    DCH0DSIZ = BUFFER_SAMPLES * 4;
    DCH0CONSET = _DCH0CON_CHEN_MASK;
}

Ventajas del Bare-Metal para NMP

  • Latencia determinista: Sin preemption de OS → jitter <0.1ms
  • Arranque rápido: <100ms vs 30s de Linux embebido
  • Bajo consumo: PIC32 a 200MHz → ≈0.5W vs Raspberry Pi 3-5W
  • Sin overhead TCP/IP stack completo: lwIP minimal stack, solo UDP/RTP
Desventaja: Sin OS, la gestión de red (pérdida de paquetes, reconexión) debe implementarse manualmente. Usar FreeRTOS con lwIP es un compromiso razonable (añade ~50µs de jitter).

def encode_frame(pcm_float32: list) -> bytes:
    # Convertir float32 [-1,1] a int16
    pcm_int16 = [int(max(-32768, min(32767, s * 32767))) for s in pcm_float32]
    return encoder.encode(bytes(pcm_int16), FRAME_SIZE)

La Latencia como Elemento Compositivo

En lugar de combatir la latencia, compositores como Pauline Oliveros y Mark Dresser la incorporan como elemento musical:

  • Música de latencia larga (>50ms): Canon distribuido — cada performer es un eco del anterior. La latencia fija se convierte en intervalo rítmico.
  • Contrapunto asíncrono: Múltiples performers en ciudades distintas con latencias diferentes → ployphony emergente no planeada.
  • Delay como intervalo: 25ms = ~1/2 pulsación a 120bpm. Se puede componer específicamente para ese espacio.
  • Feedback loops de red: Usar el delay de red como efecto (similar a tape delay).
# Calcular latencia como intervalo rítmico
def latency_as_rhythm(latency_ms: float, tempo_bpm: float) -> dict:
    beat_ms = 60000 / tempo_bpm
    subdivision = latency_ms / beat_ms
    return {
        "latency_ms": latency_ms,
        "tempo_bpm": tempo_bpm,
        "beat_fraction": subdivision,
        "musical_value": f"1/{round(1/subdivision)}" if subdivision < 1 else f"{subdivision:.2f} beats"
    }

# Ejemplo: 25ms con 120bpm → 1/2 del beat (quaver rest)
print(latency_as_rhythm(25, 120))

Q1 · Latencia

¿Por qué no se puede usar WiFi para NMP profesional y qué dos parámetros del buffer de JACK controlan la latencia de audio?

Ver respuesta

WiFi introduce jitter variable (1–20ms) por colisiones, retransmisiones y CSMA/CA. Es impredecible y rompe la temporalidad musical. Los dos parámetros de JACK son: -p (periodo/buffer en samples) que determina T_buf=samples/fs, y -n (número de periodos) que determina el buffer total de seguridad. Con -p 64 -n 2 @ 48kHz: T_buf=1.33ms, T_total=2.67ms.

Q2 · HOL Blocking

Explica qué es el Head-of-Line Blocking de TCP y dibuja mentalmente qué sucede con los paquetes 6,7,8 si el paquete 5 se pierde.

Ver respuesta

TCP garantiza orden de entrega. Si PKT5 se pierde, el receptor envía ACK por PKT4 y espera PKT5 retransmitido. PKT6, PKT7, PKT8 ya llegaron al buffer del kernel pero no pueden ser entregados a la aplicación hasta que PKT5 llegue. En una red con RTT=50ms, esto introduce 50ms+ de retardo adicional justo cuando llegó la retransmisión → inaceptable para audio de 1.33ms de frame.

Q3 · PLC

¿Qué hace el AR PLC cuando se pierde un paquete? ¿Por qué se aplica fade-out en lugar de continuar la predicción indefinidamente?

Ver respuesta

El AR PLC usa los últimos p samples conocidos y los coeficientes autoregresivos (estimados por Levinson-Durbin sobre el audio reciente) para predecir las muestras perdidas: y[n]=Σφᵢy[n−i]. El fade-out se aplica porque el modelo AR es estadísticamente correcto a corto plazo pero la predicción diverge progresivamente. Más allá de ~20ms de ocultación, el modelo produce artefactos o tonos falsos → se atenúa a silencio para evitar ruido percibido peor que el silencio.

Q4 · PTP

¿Por qué AES67 requiere PTP y no simplemente NTP? ¿Cuál es la diferencia en precisión?

Ver respuesta

NTP (Network Time Protocol) sincroniza relojes de sistema con precisión de ~1ms usando software timestamping. AES67 necesita sincronía de muestras de audio @ 48kHz → resolución de 1/48000 = 20.8µs. PTP (IEEE 1588) usa hardware timestamping en el NIC para eliminar el jitter del sistema operativo, logrando offset <1µs. Sin esta precisión, dos canales de audio sincronizados tendrían drift de samples al cabo de segundos.

Q5 · Composición

¿Cómo puede una latencia fija de 25ms convertirse en un intervalo musical a 120bpm? Calcula la fracción de beat.

Ver respuesta

A 120bpm, un beat dura 500ms. Una latencia de 25ms = 25/500 = 1/20 del beat. En términos musicales, es aproximadamente 1/16 de beat (ya que 500/16=31.25ms) o más exactamente una semicorchea de 32avos (500/32=15.6ms ≈ 25ms entre ambos). Un compositor puede diseñar un canon donde el segundo performer llega 1/20 de beat "tarde" sistemáticamente, creando un efecto de flam percusivo o un rubato controlado.
