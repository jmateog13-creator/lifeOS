M1 · Protocolo
DMX512-A y RS-485: Fundamentos Físicos
Señalización diferencial, trama serial, tasa de refresco y RDM bidireccional
T_frame = Break(88µs) + MAB(8µs) + 513 slots × 44µs/slot ≈ 22.6ms → ~44Hz refresco máximo

Arquitectura del protocolo DMX512

DMX512 es un protocolo serial half-duplex asíncrono basado en RS-485 (EIA-485). Opera a 250 kbps con un universo de 512 canales de 8 bits. La trama comienza con un break (≥88µs), seguido de un Mark After Break (≥8µs) y luego el startcode (0x00 para DMX) más los 512 bytes de datos. RS-485 usa señalización diferencial de par trenzado — inmunidad al ruido de modo común. Requiere terminaciones de 120Ω al final del bus para evitar reflexiones.

Simulador de trama DMX512 en Python

class DMX512Frame:
    BREAK_US = 88; MAB_US = 8; SLOT_US = 44; MAX_SLOTS = 512

    def __init__(self, n_slots=512):
        self.data = [0] * min(n_slots, self.MAX_SLOTS)

    def set_rgb(self, start_ch, r, g, b):
        self.data[start_ch-1], self.data[start_ch], self.data[start_ch+1] = r, g, b

    def frame_duration_ms(self):
        total_us = self.BREAK_US + self.MAB_US + (len(self.data)+1)*self.SLOT_US
        return total_us / 1000

    def refresh_rate_hz(self): return 1000 / self.frame_duration_ms()

# Ejemplo: 3 fixtures RGB
frame = DMX512Frame(512)
frame.set_rgb(1, 255, 0, 0)    # Fixture 1: Rojo
frame.set_rgb(4, 0, 255, 0)    # Fixture 2: Verde
frame.set_rgb(7, 0, 0, 255)    # Fixture 3: Azul

# 16-bit resolution (MSB+LSB para pan/tilt):
coarse, fine = 180, 128
pan_16bit = coarse * 256 + fine  # = 46208 de 65535 pasos
print(f"Duración: {frame.frame_duration_ms():.2f}ms → {frame.refresh_rate_hz():.1f}Hz")

Q1 · DMX512 y RS-485

¿Por qué DMX512 tiene una tasa de refresco máxima de ~44Hz y qué ocurre si se conecta el bus sin terminaciones de 120Ω?

Ver respuesta

La tasa de refresco de ~44Hz es un límite físico del protocolo serial: una trama completa con 512 slots tarda Break(88µs) + MAB(8µs) + 513 bytes × 44µs/byte ≈ 22.6ms. No puede haber más de 1000ms/22.6ms ≈ 44 tramas por segundo. No es una limitación de los fixtures sino de la velocidad del bus RS-485 a 250kbps. Sin terminaciones de 120Ω al final del bus: las señales se reflejan en los extremos del bus (impedancia no adaptada), generando "eco" de la señal que suma al original. Esto produce errores de bit, flicker aleatorio en los fixtures y, en el peor caso, comportamiento errático no relacionado con los valores enviados. Las terminaciones absorben la energía al final del bus eliminando las reflexiones.

Transmisor Art-Net en Python

import socket, struct

class ArtNetSender:
    ARTNET_PORT = 6454
    HEADER = b'Art-Net\x00'
    OPCODE_DMX = 0x5000

    def __init__(self, target_ip='255.255.255.255'):
        self.target_ip = target_ip; self.sequence = 0
        self.sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
        self.sock.setsockopt(socket.SOL_SOCKET, socket.SO_BROADCAST, 1)

    def send(self, universe, data):
        self.sequence = (self.sequence % 255) + 1
        dmx = bytes(data);
        if len(dmx) % 2: dmx += b'\x00'
        pkt = (self.HEADER +
               struct.pack('H', 14) +
               bytes([self.sequence, 0]) +
               struct.pack('H', len(dmx)) + dmx)
        self.sock.sendto(pkt, (self.target_ip, self.ARTNET_PORT))

    def send_artsync(self):
        pkt = self.HEADER + struct.pack('H',14) + bytes(4)
        self.sock.sendto(pkt, (self.target_ip, self.ARTNET_PORT))

Q2 · Art-Net

¿Qué identifica el OpCode 0x5000 en un paquete Art-Net y para qué sirve el ArtSync (OpCode 0x5200)?

Ver respuesta

El OpCode 0x5000 identifica un paquete ArtDmx — instruye a los nodos receptores a procesar los bytes de datos como un universo DMX512 secuenciado. Es el paquete más común: se transmite hasta 44 veces por segundo por universo. El ArtSync (OpCode 0x5200) es un paquete de sincronización sin datos DMX enviado como broadcast tras actualizar todos los universos de un frame. Su función: en arrays LED con múltiples universos (ej. un mural de 2000 LEDs = 4 universos), los 4 paquetes ArtDmx llegan en momentos ligeramente distintos. Sin ArtSync, los nodos actualizan sus fixtures al recibir cada paquete → se produce "tearing" (bandas horizontales de diferentes frames). Con ArtSync: los nodos acumulan todos los paquetes del frame y aplican todos los cambios simultáneamente al recibir el ArtSync.

M3 · Multicast
sACN (ANSI E1.31): Multicast y Prioridades
UDP Multicast, IGMP Snooping, prioridades HTP y Per-Address Priority
IP Multicast sACN = 239.255.0.0 + (universe / 256).(universe % 256)
Universo 1 → 239.255.0.1 | Universo 42 → 239.255.0.42 | Universo 300 → 239.255.1.44

Sistema de prioridades sACN

Cada paquete sACN incluye un campo de prioridad de 0 a 200 (default: 100). Con múltiples controladores en el mismo universo, los nodos aplican HTP (Highest Takes Precedence): seleccionan el valor más alto por canal de entre todas las fuentes activas. Per-Address Priority (startcode 0xdd) asigna prioridades discretas por canal individual — permite que un controlador tenga prioridad sobre canales específicos (ej. pan/tilt) mientras otro controla el color.

IGMP Snooping: fundamento de red

IGMP Snooping en switches de Capa 2 examina las solicitudes IGMP Join/Leave, limitando el tráfico sACN exclusivamente a los puertos que lo soliciten. Sin IGMP Snooping, todo el tráfico multicast se comporta como broadcast — en un show con 200 universos a 44Hz, llegan 8,800 paquetes/segundo a cada dispositivo conectado, saturando CPUs de laptops de audio, cámaras y tablets del equipo.

Q3 · sACN Multicast

¿Por qué sACN usa Multicast UDP en lugar de Broadcast como Art-Net? ¿Cuándo es estrictamente necesario activar IGMP Snooping?

Ver respuesta

sACN usa Multicast porque cada universo se envía a una dirección IP específica (239.255.x.x). Los nodos que necesitan ese universo envían IGMP Join a esa dirección; el switch solo reenvía el tráfico a esos puertos. Art-Net Broadcast llega a todos los dispositivos de la red sin distinción. En un show pequeño (8 universos, 10 dispositivos) la diferencia es mínima. IGMP Snooping es estrictamente necesario cuando: (a) hay más de ~20 universos activos — sin él, el multicast se convierte en broadcast de todos modos; (b) hay equipos sensibles en la misma red (consolas de audio, sistemas de video) que no deben recibir carga extra; (c) la red tiene <1 Gbps entre switches. En redes pequeñas dedicadas exclusivamente a iluminación, puede omitirse, aunque siempre es mejor práctica activarlo.

Encoder/Decoder 16-bit DMX en Python

class DMX16BitController:
    MAX_16BIT = 65535

    def encode_16bit(self, value_0to1):
        raw = int(max(0, min(value_0to1, 1.0)) * self.MAX_16BIT)
        return raw >> 8, raw & 0xFF       # coarse, fine

    def decode_16bit(self, coarse, fine):
        return ((coarse << 8) | fine) / self.MAX_16BIT

    def angle_to_dmx(self, angle_deg, total_range=540):
        norm = (angle_deg + total_range/2) / total_range
        return self.encode_16bit(norm)

ctrl = DMX16BitController()

# Demostración de resolución
print(f"8-bit:  {540/256:.2f}°/paso")
print(f"16-bit: {540/65536*1000:.3f} m°/paso")

# Posiciones
for angle in [-270, 0, 90, 270]:
    c, f = ctrl.angle_to_dmx(angle)
    print(f"{angle:+4.0f}° → Coarse={c:3d}, Fine={f:3d}")

Q4 · Resolución 16-bit

¿Por qué la resolución 16-bit es perceptualmente crítica para Pan/Tilt pero generalmente no necesaria para el canal de color RGB?

Ver respuesta

Los motores de Pan/Tilt con 8-bit producen "stepping" mecánico visible: en un barrido de 540°, cada incremento de canal es 540/256 ≈ 2.1°. En fades lentos estos saltos son claramente perceptibles como movimiento escalonado. Con 16-bit: 540/65536 ≈ 0.008° por paso — imperceptible. Para color RGB, el ojo humano tiene sensibilidad de contraste de ~1/200 bajo condiciones de show: múltiples fixtures mezclando colores, distancia al público, y variaciones de intensidad enmascaran la cuantización de 256 pasos. Además, en color el mixing entre tres canales (R, G, B) suaviza efectivamente los pasos — el resultado visual interpolado tiene resolución mucho mayor que cualquier canal individual. Por eso el estándar prioriza 16-bit para movimiento y mantiene 8-bit para color en la mayoría de aplicaciones.

Analizador FFT audio-reactivo en Python

import numpy as np
from scipy.fft import rfft, rfftfreq

class AudioReactiveFFT:
    BANDS = {'sub_bass':(20,60),'bass':(60,250),'mids':(500,4000),'highs':(4000,20000)}

    def __init__(self, fs=44100, fft_size=2048, gamma=0.5):
        self.fs = fs; self.fft_size = fft_size; self.gamma = gamma
        self.freqs = rfftfreq(fft_size, 1/fs)
        self.smoothed = {b: 0.0 for b in self.BANDS}

    def process(self, audio_frame, alpha=0.3):
        window = np.hanning(self.fft_size)
        spectrum = np.abs(rfft(audio_frame[:self.fft_size] * window)) / (self.fft_size/2)
        dmx = {}
        for band, (f_low, f_high) in self.BANDS.items():
            mask = (self.freqs >= f_low) & (self.freqs < f_high)
            energy = float(np.mean(spectrum[mask])) if mask.any() else 0.0
            # EMA smoothing
            self.smoothed[band] = alpha*energy + (1-alpha)*self.smoothed[band]
            # Gamma mapping (logarithmic perception)
            normalized = min(self.smoothed[band] / 0.1, 1.0)
            dmx[band] = int((normalized ** self.gamma) * 255)
        return dmx

Q5 · FFT Audio-Reactivo

¿Por qué el mapeo de magnitudes FFT a valores DMX debe usar una curva exponencial/gamma? Explica también el trade-off entre tamaño de ventana FFT grande vs pequeño para iluminación en vivo.

Ver respuesta

El oído humano percibe en escala logarítmica (decibelios). Las magnitudes FFT son lineales: un sonido que suena "el doble de fuerte" puede tener magnitud 10 veces mayor. Con mapeo lineal: los niveles bajos producen valores DMX casi 0 (fixtures apagados en música tranquila), y los picos saturan el rango al instante — la instalación parece "muerta en silencio" y "saturada en fortissimo". La curva gamma (ej. γ=0.5 = raíz cuadrada) amplifica los niveles bajos: magnitud 0.01 mapea a valor DMX ~12, magnitud 0.04 mapea a ~51. Trade-off FFT: ventana grande (N=4096) da mejor resolución frecuencial (10.8 Hz/bin) pero menor resolución temporal (más frames por análisis → mayor latencia para detectar transientes como kicks). Ventana pequeña (N=512) responde más rápido a cambios bruscos pero mezcla más frecuencias por bin. Para sincronía rítmica (bass kick) se prefiere N=512-1024; para análisis de color/armonía, N=2048-4096.

Filtro EMA y Mitigación de DMX Jitter
IIR paso-bajo de primer orden, priming, trampa del entero en C++
y[n] = α · x[n] + (1 − α) · y[n−1]
f_c = −ln(1 − α) × f_update / (2π) → α=0.15, 44Hz: f_c ≈ 1.06 Hz

EMA correcto en Python y C++

class EMAFilter:
    def __init__(self, alpha=0.1):
        self.alpha = float(alpha)  # obligatorio: float, NO int
        self.y_prev = None; self.first_time = True

    def process(self, x):
        x = float(x)
        if self.first_time:
            self.y_prev = x   # PRIMING: y[0] = x[0] → elimina wind-up
            self.first_time = False
            return x
        y = self.alpha * x + (1.0 - self.alpha) * self.y_prev
        self.y_prev = y
        return y

# ───── TRAMPA DEL ENTERO EN C++ ─────
# INCORRECTO: alpha_int = round(0.001 × 256) = 0
# → y = (0×x + 256×y_prev)/256 = y_prev → filtro CONGELADO

# CORRECTO:
# double ema_filter(double x, double alpha, double* y_prev, bool* first) {
#     if (*first) { *y_prev = x; *first = false; return x; }
#     double y = alpha * x + (1.0 - alpha) * (*y_prev);
#     *y_prev = y; return y;
# }

Q6 · Filtro EMA

¿Qué es el "priming" en un filtro EMA, qué problema evita, y por qué la implementación con enteros puede corromper el filtro con valores de alpha pequeños?

Ver respuesta

El priming inicializa y[0] = x[0] en el primer ciclo. Sin priming, y_prev comienza en 0 y el filtro tarda muchos frames en alcanzar el valor real de la señal ("wind-up"). Si x[0]=200 y α=0.15, sin priming necesitas ~20 frames para que y alcance 200; con priming arranca instantáneamente en 200. La trampa del entero ocurre con α muy pequeño: supón α=0.001 en implementación ×256. alpha_int = round(0.001 × 256) = 0. La fórmula se convierte en y = (0×x + 256×y_prev)/256 = y_prev — el filtro deja de actualizar con la señal nueva, quedando congelado en el último valor. Con double: y = 0.001×x + 0.999×y_prev — funciona perfectamente aunque actualice despacio. Por eso el uso de punto flotante (double) es obligatorio en EMA con alpha pequeño.

Q7 · TouchDesigner Pipeline

¿Cuál es la diferencia entre CHOP y DAT en TouchDesigner para el pipeline de iluminación, y qué es la arquitectura Time Slice?

Ver respuesta

CHOP (Channel Operator) procesa señales numéricas continuas en el tiempo: audio, valores de FFT, posiciones de sensores, canales DMX. Es la columna vertebral del pipeline de iluminación — los valores fluyen como canales numéricos desde el audio hasta los paquetes Art-Net. DAT (Data Operator) procesa texto y tablas: útil para cue lists, configuración de universos, logging y scripts Python, pero no para señales en tiempo real. La arquitectura Time Slice de TouchDesigner mide el tiempo entre cook frames del motor. Cuando el motor detecta que está detrás (cook time > frame time), procesa múltiples "slices" de CHOPs en una sola pasada para ponerse al día, minimizando la latencia percibida en los fixtures y evitando la acumulación de retrasos que degradaría la sincronía rítmica.

Q8 · Theremin Kinect

Describe el pipeline completo de datos desde el sensor Kinect hasta el sonido del theremin virtual, identificando el protocolo de comunicación entre TouchDesigner y Max/MSP.

Ver respuesta

Pipeline completo: (1) Kinect Azure detecta esqueleto y emite posiciones 3D de joints en world space (metros). (2) TouchDesigner lee el Kinect CHOP: extrae /v2/p1/r/hand/tx (posición X mano derecha) y /v2/p1/l/hand/ty (posición Y mano izquierda). (3) Math CHOP normaliza de metros (-1,1) a rango normalizado (0,1). (4) OSC Out CHOP de TD envía los valores como mensajes OSC UDP al puerto 5000 de Max/MSP (/rightX float, /leftY float). El protocolo es OSC (Open Sound Control) sobre UDP — mensajes jerárquicos tipados, sin overhead de TCP, latencia <1ms en LAN. (5) Max/MSP recibe con udpreceive 5000, parsea con OSC-route, escala rightX a frecuencia (40-2000Hz) con slide para portamento suave, escala leftY a amplitud. (6) cycle~ genera la onda sinusoidal a la frecuencia calculada, gain~ controla la amplitud. La latencia total es ~40-60ms (Kinect 33ms + procesado TD + OSC + Max + audio buffer).

def build_osc_float(address, value):
    # OSC string padding a múltiplo de 4

Q9 · OSC y Max/MSP

¿Por qué OSC es preferible a MIDI para la comunicación entre TouchDesigner y Max/MSP en instalaciones interactivas? Compara estructura de datos, latencia y flexibilidad.

Ver respuesta

MIDI: protocolo de 1983, mensajes de 1-3 bytes, valores enteros 0-127, canales fijos (noteOn, controlChange, etc.). Muy universalmente compatible con hardware. Limitaciones: valores cuantizados (128 pasos), sin flexibilidad de dirección (no puedes enviar /kinect/hand/right/x con significado semántico — tienes que mapear manualmente a CC42). OSC: diseñado para aplicaciones multimedia modernas. Ventajas: (1) Valores float de alta precisión (no cuantizados); (2) Direcciones jerárquicas con semántica (/sensor/kinect/body0/hand/right/x) — auto-documentadas; (3) Bundles: múltiples mensajes con timestamp compartido para atomicidad; (4) Velocidad: UDP sin overhead de MIDI DIN serial; (5) Multiplataforma: cualquier lenguaje puede implementarlo. La latencia de ambos en LAN es similar (<1ms), pero OSC elimina las limitaciones de resolución y la necesidad de mapeo manual de funciones.

Q10 · GDTF

¿Qué problema resuelve el formato GDTF y cómo beneficia a los diseñadores de iluminación que trabajan con múltiples plataformas de software?

Ver respuesta

GDTF (General Device Type Format) resuelve la fragmentación de perfiles de fixture: antes de GDTF, cada software de control (grandMA3, ETC Eos, Chamsys MQ) tenía su propio formato propietario de perfil. Un fabricante de fixtures tenía que crear y mantener perfiles separados para cada plataforma. GDTF (desarrollado por MA Lighting, Robe y Vectorworks) es un estándar abierto: un único archivo .gdtf contiene (1) la definición completa de atributos DMX con relaciones MSB/LSB explícitas; (2) geometría 3D para previsualización; (3) ruedas de gobos y colores; (4) metadatos del fabricante. Los beneficios: el software puede interpretar el perfil correctamente sin configuración manual; los diseñadores obtienen previsualización 3D fiel; los cambios de 8-bit a 16-bit están documentados en el perfil, no en la cabeza del técnico. GDTF Share es el repositorio online oficial donde los fabricantes publican perfiles certificados.

Q11 · Topología de Red

¿Por qué se recomienda separar el tráfico Art-Net/sACN en una VLAN dedicada y qué ventaja aportan los nodos conversores distribuidos frente a un único nodo central?

Ver respuesta

VLAN dedicada para Art-Net/sACN: aísla el tráfico de iluminación de otros sistemas (video, audio, gestión). Sin VLAN, un burst de tráfico de video puede retardar los paquetes Art-Net causando drops o jitter en los fixtures. La VLAN garantiza QoS (Quality of Service) — el tráfico DMX recibe prioridad y ancho de banda garantizado. En switches gestionados, se configura DSCP para marcar los paquetes Art-Net con prioridad EF (Expedited Forwarding). Nodos conversores distribuidos vs. centralizado: un único nodo central requiere cableado DMX de largo recorrido desde el rack central a cada grupo de fixtures (~100m máximo por RS-485 tramo). Los nodos distribuidos (OCTO, GigaJet) se ubican en cada rack o barra de luces: reciben Art-Net por Ethernet (kilómetros sin límite) y emiten DMX RS-485 localmente (distancia corta). Ventajas: menos cable DMX, menos puntos de fallo en el cableado, capacidad de añadir zonas sin nueva infraestructura desde la consola.

# Mapping de coordenadas 2D a universos DMX
# Array LED: 100×10 = 1000 LEDs RGB = 3000 canales = 5.86 universos

class LEDArray:
    def __init__(self, width, height, start_universe=0):
        self.w = width; self.h = height
        self.start_uni = start_universe
        self.data = [[0]*512 for _ in range(6)]  # 6 universos

    def set_pixel(self, x, y, r, g, b):
        """Mapea LED (x,y) a canal DMX correcto."""
        pixel_idx = y * self.w + x           # serpentina: si y%2==1, invertir x
        ch = pixel_idx * 3                   # 3 canales por pixel RGB
        universe = ch // 512
        offset = ch % 512
        if offset + 2 < 512:
            self.data[universe][offset] = r
            self.data[universe][offset+1] = g
            self.data[universe][offset+2] = b

    def send_all(self, artnet_sender):
        for i, uni_data in enumerate(self.data):
            artnet_sender.send(self.start_uni + i, uni_data)
        artnet_sender.send_artsync()  # sincronizar frame

Q12 · Pixel Mapping

Para un array LED de 200×20 pixels RGB, calcula cuántos universos DMX necesitas y describe cómo evitar el efecto "tearing" al actualizar los universos.

Ver respuesta

200×20 = 4,000 LEDs × 3 canales (RGB) = 12,000 canales DMX. Universos necesarios: 12,000 / 512 = 23.4 → 24 universos. Para evitar tearing: el tearing ocurre porque los 24 paquetes Art-Net llegan al nodo en momentos ligeramente distintos (~1ms de diferencia entre el primero y el último) — si el nodo aplica cada paquete al recibirlo, el array muestra diferentes frames en distintas filas. Solución: usar ArtSync de Art-Net. El controlador (TD) envía los 24 paquetes ArtDmx y luego un paquete ArtSync broadcast. Los nodos acumulan todos los paquetes del frame sin aplicarlos y aplican todos simultáneamente al recibir el ArtSync. En sACN la sincronización se logra con el paquete de sincronización (startcode 0x00 con synchronization address configurado). Adicionalmente, usar un switch de baja latencia y QoS para minimizar la diferencia de llegada entre paquetes.

Q13 · Wireshark y Diagnóstico

Describe cómo usarías Wireshark para diagnosticar un problema donde algunos fixtures de un array LED muestran valores incorrectos intermitentemente pero el software reporta envío correcto.

Ver respuesta

Workflow de diagnóstico: (1) Capturar en el segmento de red entre el nodo Art-Net y los fixtures (o en el puerto del switch conectado al nodo). Filtro: udp.port == 6454. (2) Verificar tasa de paquetes: Statistics → IO Graph. Debe ser ~44 paquetes/s por universo. Si hay picos o valles, hay pérdida de paquetes o jitter de envío. (3) Examinar sequence number: el campo Sequence en ArtDmx debe incrementar monotónicamente. Si hay saltos (ej. ...12, 14, 15...), se perdió el paquete 13 — pérdida de red. (4) Verificar el universo de cada paquete: ¿está enviando al universo correcto? A veces un bug de configuración envía algunos paquetes al universo 0 (broadcast) en lugar del universo específico. (5) Buscar colisiones de fuente: si hay otro controlador enviando al mismo universo, los paquetes de ambas fuentes aparecen intercalados — el nodo aplica el último recibido (LTP) o HTP dependiendo del protocolo, causando valores impredecibles. (6) Medir tiempo entre paquetes: si algún paquete llega 50ms después del anterior (jitter alto), puede causar glitch visible en los fixtures.

Q1 · sACN vs Art-Net

¿Por qué sACN usa Multicast UDP en lugar de Broadcast como Art-Net? Explica el papel de IGMP Snooping en una instalación con 200 universos.

Ver respuesta

sACN mapea cada universo a una dirección Multicast específica (239.255.x.x). Los nodos que necesitan ese universo envían IGMP Join; el switch solo reenvía el tráfico a esos puertos. Art-Net Broadcast llega a todos los dispositivos sin distinción. En una instalación con 200 universos a 44Hz: Art-Net genera 200×44 = 8,800 paquetes/segundo en broadcast — llegan a todos los dispositivos de la red (laptops de audio, cámaras, tablets del equipo), consumiendo CPU innecesariamente. Con sACN + IGMP Snooping: el switch mantiene una tabla multicast→puertos y reenvía cada universo solo a los nodos que lo han solicitado. El tráfico irrelevante nunca llega a dispositivos no interesados. Sin IGMP Snooping activo en el switch, el Multicast de sACN se comporta como Broadcast de todos modos — IGMP Snooping es requisito indispensable para la eficiencia de sACN en redes grandes.

Q2 · EMA y Enteros

¿Por qué el filtro EMA puede "corromper" si se implementa con enteros en C++? Describe el priming y su importancia.

Ver respuesta

Con alpha muy pequeño (ej. α=0.001), la implementación de EMA con enteros escalados ×256 produce: alpha_int = round(0.001×256) = 0. La fórmula se convierte en y = (0×x + 256×y_prev)/256 = y_prev — el filtro deja de actualizar con la señal nueva, quedando congelado en el último valor. Con double: y = 0.001×x + 0.999×y_prev — funciona correctamente. El priming inicializa y[0] = x[0] en el primer ciclo. Sin priming: y_prev comienza en 0.0, y el filtro tarda muchos frames en alcanzar el valor real de la señal (wind-up). Si el primer valor de audio es DMX=200 y α=0.15, sin priming la salida sería ~10 después del primer frame y tardaría ~20 frames en llegar a ~180. Con priming: la salida arranca en 200 instantáneamente, sin retraso de inicialización. En instalaciones donde el sistema arranca con fixtures ya encendidos, el wind-up produce un "fade-in" no deseado en los primeros segundos.

Q3 · HTP y Prioridades sACN

Con 2 controladores sACN en el mismo universo (prioridades 200 y 100), controlador A envía CH1=200 y controlador B envía CH1=50. ¿Qué valor llega al fixture y por qué?

Ver respuesta

El valor que llega al fixture es 200 (el valor de CH1 del controlador A). HTP (Highest Takes Precedence) evalúa independientemente el valor de cada canal entre todas las fuentes activas y selecciona el mayor. La "prioridad de fuente" (0-200) en sACN determina qué fuente "gana" cuando hay conflicto, pero HTP siempre selecciona el valor más alto del canal. Si A enviara CH1=30 y B enviara CH1=200, HTP seleccionaría 200 aunque B tenga menor prioridad de fuente. Por diseño: HTP garantiza que si alguien está usando el dimmer al máximo, nadie puede "apagarlo" accidentalmente enviando 0 desde otra fuente. Para atributos como Pan/Tilt donde esto no tiene sentido (dos controladores enviando valores distintos causaría movimiento errático), se usa LTP (Latest Takes Precedence) o Per-Address Priority para asignar ownership explícito de canales específicos.

Q4 · Resolución 16-bit

¿Por qué la resolución 16-bit es perceptualmente crítica para Pan/Tilt de moving heads pero no necesariamente para el canal de color? Calcula el paso angular en ambas resoluciones para un fixture con 540° de barrido.

Ver respuesta

8-bit: 540° / 256 pasos = 2.109°/paso. Este salto es claramente visible como stepping mecánico en fades lentos — especialmente con el fixture apuntando a una pared lisa. 16-bit: 540° / 65,536 pasos = 0.008°/paso (8 millidegrees) — imperceptible al ojo humano. Para color RGB: la sensibilidad de contraste de color del ojo humano es ~1/200 bajo condiciones de show iluminado con múltiples fixtures. Con 256 pasos de rojo (8-bit), cada paso es 1/256 = 0.39% del rango — perfectamente dentro del umbral de percepción. Además, el mixing de múltiples canales R+G+B y la mezcla entre fixtures enmascara efectivamente la cuantización individual. Por eso el estándar prioriza 16-bit para movimiento (mecánico y preciso) y mantiene 8-bit para color en la mayoría de aplicaciones salvo dimmers cinematográficos de alta precisión.

Q5 · IGMP Snooping

¿Por qué IGMP Snooping es estrictamente necesario en redes sACN grandes? Describe qué ocurre en una red de 100 universos sin IGMP Snooping activado.

Ver respuesta

Sin IGMP Snooping: el switch no sabe qué puertos han solicitado qué grupos multicast. Por defecto, trata todos los paquetes multicast como broadcast de capa 2 — los replica a todos los puertos del switch. En una instalación con 100 universos a 44Hz: 100×44 = 4,400 paquetes/segundo × ~600 bytes/paquete ≈ 2.1 Mbps de tráfico multicast llega a cada dispositivo conectado al switch. Consecuencias: (1) Laptops de producción de audio y video deben procesar y descartar 4,400 paquetes/segundo innecesariamente — CPU aumenta ~3-8%, puede causar glitches en reproducción de audio; (2) Cualquier dispositivo con stack de red sensible puede verse afectado; (3) En redes con múltiples switches, el tráfico se multiplica exponencialmente. Con IGMP Snooping activo: cada nodo DMX envía IGMP Join solo para sus universos asignados. El switch construye una tabla multicast→puerto y reenvía exclusivamente a los puertos necesarios. El resto de la red no recibe el tráfico de iluminación.
