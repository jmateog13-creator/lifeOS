Q1 · Familias de Operadores

Explica la diferencia entre TOP y CHOP en TouchDesigner y da un ejemplo de cuándo usarías cada uno en una instalación interactiva.

Ver respuesta

TOP (Texture Operator) procesa imágenes y video en la GPU — todo lo visual. CHOP (Channel Operator) procesa canales de datos numéricos en el tiempo — audio, sensores, animación, control. Ejemplo: la webcam entra como TOP (imagen), el audio del micrófono entra como CHOP (valores de amplitud). Para cruzar familias se usan operadores de conversión (CHOP to TOP, SOP to TOP). En una instalación: los joints del Kinect llegan como CHOP, y para usarlos en un shader GLSL se convierten a TOP o se pasan como uniforms.

Q2 · Feedback Loops

¿Cómo funciona el feedback visual en TouchDesigner y qué efectos genera? Describe el papel del Feedback TOP y el Transform TOP dentro del loop.

Ver respuesta

El Feedback TOP introduce un delay de 1 frame, permitiendo que la imagen actual se mezcle con la frame anterior. Con un blend factor menor a 1, los trails se desvanecen gradualmente creando persistencia retiniana artificial. Si añadimos un Transform TOP dentro del loop (escala 1.002, rotación 0.1°), cada frame se expande y rota ligeramente — generando zoom infinito o espirales. Es el fundamento del video synthesis analógico de Paik ahora replicado en GPU. El factor alpha del Composite TOP controla la duración de los trails: 0.80=desvanece rápido, 0.99=persiste muchos frames.

Q3 · Callbacks y OSC

¿Cuándo usarías un CHOP Execute DAT versus un OSC In DAT en TouchDesigner? Describe el flujo de datos en cada caso.

Ver respuesta

El CHOP Execute DAT se conecta a un CHOP específico y ejecuta callbacks cada vez que el valor de un canal cambia (onValueChange, onOffToOn, etc.) — ideal para reaccionar a valores internos de TD: LFO cruzando un umbral, audio superando RMS, Kinect activando una zona. El OSC In DAT escucha en un puerto UDP y ejecuta onReceiveOSC por cada mensaje externo — ideal para recibir control desde dispositivos externos (iPad/TouchOSC, Max/MSP, SuperCollider). En una instalación típica: el Kinect llega como CHOP Execute (evento interno), y el controlador del técnico llega como OSC In DAT (evento externo).

Q4 · Pipeline GPU

Explica qué hace el fragment shader en el pipeline GPU y por qué GLSL permite efectos que serían imposibles (o muy costosos) con TOPs nativos de TouchDesigner.

Ver respuesta

El fragment shader se ejecuta en paralelo para cada píxel de la textura de salida en la GPU. Mientras los TOPs nativos de TD son operaciones fijas (blur, noise, composite), un GLSL fragment shader puede implementar cualquier función matemática arbitraria por píxel: ray marching de SDFs, fractales, plasma, campos de distancia, simulaciones físicas. La GPU ejecuta miles de hilos simultáneos (un shader por píxel), permitiendo calcular un shader 1920×1080 = ~2M píxeles en paralelo. Los TOPs nativos tienen coste fijo bien optimizado pero funcionalidad fija; el GLSL TOP es infinitamente flexible a costa de mayor complejidad de programación.

SDFs y operaciones booleanas

// SDFs: d > 0 exterior, d < 0 interior, d = 0 superficie
float sdSphere(vec3 p, float r) { return length(p) - r; }

float sdBox(vec3 p, vec3 b) {
    vec3 q = abs(p) - b;
    return length(max(q, 0.0)) + min(max(q.x, max(q.y, q.z)), 0.0);
}

float sdTorus(vec3 p, vec2 t) {
    vec2 q = vec2(length(p.xz) - t.x, p.y);
    return length(q) - t.y;
}

// Blend suave (smooth union):
float smin(float a, float b, float k) {
    float h = clamp(0.5 + 0.5*(b-a)/k, 0.0, 1.0);
    return mix(b, a, h) - k*h*(1.0-h);
}

float opUnion(float d1, float d2) { return min(d1, d2); }
float opSubtract(float d1, float d2) { return max(-d1, d2); }

Ray Marching completo con iluminación

out vec4 fragColor;
uniform float uTime;

float map(vec3 p) {
    return smin(sdSphere(p, 0.5), sdTorus(p - vec3(0,0.6,0), vec2(0.4,0.15)), 0.2);
}

vec3 calcNormal(vec3 p) {
    vec2 e = vec2(0.001, 0.0);
    return normalize(vec3(
        map(p+e.xyy)-map(p-e.xyy), map(p+e.yxy)-map(p-e.yxy), map(p+e.yyx)-map(p-e.yyx)));
}

void main() {
    vec2 uv = (vUV.st * 2.0 - 1.0);
    uv.x *= uTD2DInfos[0].res.z / uTD2DInfos[0].res.w;
    vec3 ro = vec3(0.0, 0.5, 2.5);
    vec3 rd = normalize(vec3(uv, -1.5));
    float a = uTime * 0.5;
    mat2 rot = mat2(cos(a),-sin(a),sin(a),cos(a));
    ro.xz = rot*ro.xz; rd.xz = rot*rd.xz;
    float t = 0.0; vec3 col = vec3(0.05);
    for (int i = 0; i < 64; i++) {
        vec3 p = ro + rd*t; float d = map(p);
        if (d < 0.001) {
            vec3 n = calcNormal(p); vec3 light = normalize(vec3(1.0,2.0,1.0));
            float diff = max(dot(n,light), 0.0);
            col = vec3(0.1, 0.6, 0.9)*diff + 0.1; break;
        }
        t += d; if (t > 20.0) break;
    }
    fragColor = TDOutputSwizzle(vec4(col, 1.0));
}

Q5 · Ray Marching

¿Cómo funciona el ray marching y por qué la SDF garantiza que el rayo nunca colisiona prematuramente con la geometría? Explica el concepto de "sphere tracing".

Ver respuesta

El ray marching lanza un rayo desde la cámara y avanza iterativamente. En cada punto p del rayo, consulta la SDF: d = map(p). La SDF retorna la distancia al objeto más cercano en cualquier dirección. El rayo avanza exactamente esa distancia d (safe step) — lo que garantiza que nunca atraviesa un objeto prematuramente, ya que por definición ningún objeto está a menos de d unidades en ninguna dirección. Esto se llama sphere tracing porque es como hacer avanzar una esfera de radio d. Cuando d < epsilon (ej. 0.001), se considera intersección. La ventaja sobre ray casting con mallas es que toda la geometría se describe matemáticamente sin triángulos, permitiendo formas arbitrarias, blending suave (smin) y fractales.

Q6 · Kinect y Gestual

¿Qué es el body tracking del Kinect Azure y cómo se mapean los joints a parámetros visuales en TouchDesigner? Describe el pipeline completo de dato de sensor a efecto visual.

Ver respuesta

El body tracking del Kinect Azure usa el SDK de Microsoft (DNN) para identificar hasta 32 joints del esqueleto humano (cabeza, hombros, manos, rodillas, etc.) en coordenadas 3D del mundo real (metros). En TouchDesigner, el Kinect Azure CHOP expone todos estos joints como canales numéricos. El pipeline es: Kinect Azure (USB 3.0) → Kinect Azure CHOP (joints como canales) → Script CHOP o CHOP Execute DAT (Python: leer y normalizar valores) → Constant CHOP (almacenar) → expresión en parámetro del TOP destino. Ejemplo: hand_r_x (-1.0 a +1.0 metros) → Translate X del Noise TOP. La latencia total (movimiento físico → visual) es ~33ms con 30fps del Kinect más el cook de TD.

Detección de movimiento con OpenCV

def detect_motion(current_frame, prev_frame, threshold=25):
    gray_curr = cv2.cvtColor(current_frame, cv2.COLOR_BGR2GRAY)
    gray_prev = cv2.cvtColor(prev_frame, cv2.COLOR_BGR2GRAY)
    diff = cv2.absdiff(gray_curr, gray_prev)
    _, thresh = cv2.threshold(diff, threshold, 255, cv2.THRESH_BINARY)
    # Calcular porcentaje de movimiento
    motion_ratio = cv2.countNonZero(thresh) / thresh.size
    return motion_ratio  # 0.0 = sin movimiento, 1.0 = movimiento total

Q7 · Computer Vision

¿Cuál es la diferencia entre usar Kinect Azure y MediaPipe para body tracking en instalaciones? ¿Cuándo elegirías cada uno?

Ver respuesta

Kinect Azure usa hardware dedicado (sensor ToF infrarrojo + DNN de Microsoft) con SDK propietario: ventajas: datos 3D reales en metros, funciona en oscuridad, 32 joints con precisión alta, menor CPU. Desventajas: hardware costoso (~400€), solo Windows, necesita SDK K4A. MediaPipe usa una webcam estándar y DNN en CPU/GPU: ventajas: hardware barato (cualquier webcam), multiplataforma, fácil integración con Python en TD, también detecta manos y rostro. Desventajas: coordenadas 2D (estimación de profundidad aproximada), más sensible a la iluminación, mayor carga de CPU. Para instalaciones permanentes en museos se prefiere Kinect Azure (más robusto); para proyectos portátiles o con webcam USB existente, MediaPipe es suficiente.

Q8 · Art-Net vs DMX

¿Por qué Art-Net es preferible a DMX serial en instalaciones grandes con múltiples zonas de iluminación? ¿Qué limitaciones tiene el DMX RS-485 clásico?

Ver respuesta

DMX512 RS-485 es un bus serial que transporta exactamente 512 canales por cable físico a 250kbps. Para controlar 3 zonas independientes necesitas 3 cables DMX separados. Art-Net encapsula múltiples universos DMX en paquetes UDP sobre Ethernet estándar: una única red Cat6 puede transportar miles de universos simultáneamente. Las ventajas son: (1) infraestructura de red ya existente, (2) distancias largas con switches, (3) múltiples fuentes y controladores, (4) TD envía múltiples universos desde una sola interfaz de red. TouchDesigner puede enviar Art-Net a nodos distribuidores (ej. Artnet-DMX node) ubicados junto a cada rack de fixtures, convirtiendo de Ethernet a RS-485 localmente. La limitación de Art-Net es que UDP no garantiza entrega, pero en red local la fiabilidad es prácticamente 100%.

Q9 · Projection Mapping

¿Qué es el edge blending en proyección multipantalla y por qué no basta con simplemente superponer las imágenes en la zona de overlap?

Ver respuesta

El edge blending es la técnica de atenuar gradualmente el brillo en los bordes de cada proyector en la zona de overlap entre dos proyectores adyacentes. Si simplemente se superponen las imágenes, la zona donde dos proyectores coinciden recibe doble cantidad de luz, apareciendo más brillante y con una línea visible. El edge blending aplica un gradiente: el proyector izquierdo reduce su luminosidad de 100% a 0% de derecha a izquierda en la zona de overlap; el proyector derecho hace lo contrario. El resultado es luminosidad uniforme en toda la pantalla. Adicionalmente, hay que corregir la curva gamma en la zona de blend, ya que la percepción humana de brillo es logarítmica y una reducción lineal no produce una transición visualmente suave.

class ShowControl:
    def __init__(self):
        self.cue_table = op('show_cues')  # Table DAT con cues
        self.current_cue = 0

    def go(self):
        cue = self.cue_table[self.current_cue + 1, :]
        cue_type = cue[1].val
        cue_value = cue[2].val

        if cue_type == 'visual':
            op('visual_preset').par.value0 = int(cue_value)
        elif cue_type == 'fade':
            op('master_fade').par.duration = float(cue[3].val)
            op('master_fade').par.gopulse.pulse()
        elif cue_type == 'osc':
            send_osc(cue_value, float(cue[3].val))
        elif cue_type == 'wait':
            run('show_control.advance()', delayFrames=int(float(cue_value)*60))
        self.current_cue += 1

Q10 · Show Control

¿Cómo implementarías un sistema de show control que permita a un técnico no-técnico operar una instalación compleja con una sola tecla, mientras el sistema gestiona automáticamente la sincronización con audio?

Ver respuesta

El diseño óptimo usa: (1) Cue List en Table DAT — cada fila es un cue con tipo (visual, fade, osc, wait) y parámetros, editable en hoja de cálculo; (2) una sola tecla (espacio) dispara el siguiente cue via keyboard callback Python; (3) un Timer CHOP conectado a MIDI Clock de Ableton para que los cambios visuales se produzcan en tiempos musicales exactos (no en ms absolutos); (4) modo autónomo: si no se pulsa la tecla en N segundos, el sistema cicla automáticamente. El técnico solo ve un indicador del cue actual y un botón GO. La complejidad está oculta en el cue list. Para una instalación que funciona sin operador, se añade una cue de tipo 'wait' al final que vuelve al cue 1 (loop automático).

Q11 · FFT y Audio-Reactivo

¿Por qué el mapeo de magnitudes FFT a valores de parámetros visuales debe usar una curva gamma/logarítmica en lugar de un mapeo lineal?

Ver respuesta

El oído humano percibe el sonido en escala logarítmica (decibelios). Las magnitudes FFT son lineales en amplitud: un sonido que suena "el doble de fuerte" puede tener magnitud FFT 10 veces mayor. Con mapeo lineal, los niveles bajos de audio producen valores visuales casi imperceptibles (los parámetros apenas se mueven), mientras los picos saturan el rango al instante — la instalación parece "muerta en silencio" y "saturada en música fuerte". Con curva gamma (ej. γ=0.5 = raíz cuadrada): una magnitud de 0.01 mapea a 0.1, y 0.04 mapea a 0.2 — el rango bajo tiene mucha más sensibilidad. Esto emula la respuesta logarítmica del oído y produce visuales que responden de forma natural y expresiva a toda la dinámica musical.

Q13 · Redes Distribuidas

¿Qué es NDI y cómo facilita las instalaciones multi-máquina en comparación con soluciones basadas en SDI o capturadores de video físicos?

Ver respuesta

NDI (Network Device Interface) de NewTek es un protocolo para transmisión de video de alta calidad (hasta 4K) con baja latencia sobre redes IP estándar (GbE o 10GbE). TouchDesigner incluye soporte NDI nativo: un TD puede enviar su output como fuente NDI y cualquier otro TD en la misma red puede recibirla simplemente seleccionando el nombre de la fuente. Comparado con SDI: (1) No se necesitan cables coaxiales ni capturadores DeckLink — ahorro de ~2000€ en hardware; (2) Una red Ethernet puede transportar múltiples señales simultáneas; (3) Distancias limitadas solo por la red IP (no 100m de SDI). Desventajas: NDI consume más ancho de banda que SDI (requiere GbE por señal 1080p) y tiene latencia ligeramente mayor (2-5 frames vs <1 frame SDI). Para instalaciones en el mismo edificio, NDI es la solución moderna y económica.

Q1 · Familias TD

¿Cuál es la diferencia entre TOP, CHOP, SOP y DAT en TouchDesigner y cómo fluyen los datos entre familias?

Ver respuesta

TOP (Texture Operator) procesa imágenes y video en la GPU — todo lo visual. CHOP (Channel Operator) procesa canales de datos numéricos en el tiempo — audio, sensores, animación. SOP (Surface Operator) procesa geometría 3D — meshes, point clouds. DAT (Data Operator) procesa texto y tablas — Python scripts, JSON, CSV. Los datos solo fluyen dentro de su familia; para cruzar familias se usan operadores de conversión: CHOP to TOP convierte canales numéricos a textura para usarlos en shaders; SOP to CHOP exporta posiciones de vértices como canales; TOP to CHOP extrae estadísticas de imagen. Esta arquitectura permite que cada tipo de dato sea procesado por el hardware más adecuado (GPU para TOP, CPU para CHOP).

Q2 · Ray Marching

¿Cómo funciona el ray marching en GLSL y por qué permite crear geometrías imposibles con mallas triangulares convencionales?

Ver respuesta

El ray marching lanza rayos desde la cámara virtual y avanza iterativamente: en cada punto consulta la SDF (Signed Distance Function) que retorna la distancia al objeto más cercano. El rayo avanza esa distancia (safe step, ya que ningún objeto está más cerca en ninguna dirección). Cuando la distancia es <epsilon, se registra intersección. Las ventajas sobre mallas: (1) No hay triángulos — la geometría es una función matemática que puede ser infinitamente compleja sin coste de memoria; (2) Los blends suaves entre formas (smin) son triviales matemáticamente pero imposibles con mallas; (3) Las modificaciones de dominio (fold space, twist, repeat) crean estructuras fractales sin coste extra; (4) Los parámetros de cualquier SDF pueden ser uniforms animados en tiempo real desde TD.

Q3 · DMX y Art-Net

¿Por qué Art-Net es preferible a DMX RS-485 en instalaciones grandes y qué papel juega el nodo conversor Ethernet-DMX?

Ver respuesta

DMX RS-485 es un bus serial que transporta 512 canales por cable físico (~100m máximo) a 250kbps. Para múltiples zonas de iluminación necesitas múltiples cables desde el controlador central. Art-Net encapsula universos DMX en paquetes UDP sobre red Ethernet estándar: un cable Cat6 puede transportar miles de universos. TouchDesigner envía Art-Net desde un solo puerto de red. Los nodos conversores Ethernet-DMX (ej. ENTTEC ODE, Artnet Node) se ubican en cada rack de fixtures: reciben Art-Net por red y emiten señal DMX RS-485 localmente. Esto elimina el cableado de señal de larga distancia, permite añadir zonas sin nueva infraestructura, y hace que el controlador (TD) sea independiente del número de universos que controla.

Q4 · Robustez

Diseña la estrategia de robustez para una instalación que debe correr 6 meses sin intervención. ¿Qué fallos son los más comunes y cómo se mitigan?

Ver respuesta

Los fallos más comunes en instalaciones 24/7 son: (1) Memory leak / acumulación: mitigación con limpiar cachés de texturas cada 24h a hora de bajo tráfico y monitorizar mem usage con Python. (2) Crash de TouchDesigner: mitigación con LaunchAgent/Task Scheduler que reinicia TD si el proceso muere. (3) Bloqueo de hardware (USB, Kinect): mitigación con watchdog que detecta ausencia de datos del sensor y reinicia el driver. (4) Fallo eléctrico: UPS con autonomía de 15min. (5) Overheating: ventilación forzada del rack, monitoreo de temperatura con alerta. El watchdog interno de TD (Timer CHOP callback cada 60s) comprueba FPS, estado de sensores y conectividad OSC. Si algo falla, aplica recuperación en cascada antes de reiniciar el sistema completo.

Q5 · NDI Multi-máquina

¿Cómo sincronizarías 3 máquinas TouchDesigner en red para que ejecuten cues visuales simultáneamente al ritmo del audio sin slippage perceptible?

Ver respuesta

La arquitectura de sincronización usa tres capas: (1) BPM musical via Ableton Link — todas las máquinas comparten el mismo clock de tempo en tiempo real; los cambios visuales se programan en "beats" no en segundos, eliminando el drift entre máquinas. (2) Sincronización de frame via OSC "director" — la máquina maestra envía mensajes OSC de cue (/show/cue N) en broadcast UDP; las 3 máquinas reciben el mismo mensaje en <1ms (LAN). (3) NTP para timestamps de log — no para sincronización de frame, pero permite correlacionar logs entre máquinas para diagnóstico. NDI solo transmite video, no garantiza sincronización de frame por sí solo. Para sincronización de frame pixel-perfect en arrays LED distribuidos se usa la función ArtSync de Art-Net que envía un paquete de sincronización tras todos los universos de un frame.
