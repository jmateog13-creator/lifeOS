Reglas del Audio Thread

Un sistema hard real-time garantiza que el trabajo se completa antes de un deadline absoluto. Si el audio thread excede este tiempo → glitch de audio (dropout, xrun). No hay segundo intento.

PROHIBIDO en processBlock: new / delete / malloc / free, std::mutex / lock_guard, I/O de disco o red, std::cout / logging bloqueante, system calls que bloquean, funciones matemáticas en números desnormalizados sin FTZ.

AudioProcessor mínimo seguro

class SafeAudioProcessor : public juce::AudioProcessor
{
public:
    void prepareToPlay (double sampleRate, int samplesPerBlock) override
    {
        // Aqui si podemos alocar
        spec.sampleRate = sampleRate;
        spec.maximumBlockSize = (juce::uint32) samplesPerBlock;
        spec.numChannels = (juce::uint32) getTotalNumOutputChannels();
        workBuffer.setSize (2, samplesPerBlock);
        chain.prepare (spec);
    }

    void processBlock (juce::AudioBuffer<float>& buffer,
                       juce::MidiBuffer& midiMessages) override
    {
        juce::ScopedNoDenormals noDenormals;
        const float gain = gainParam.load (std::memory_order_relaxed); // atomic, lock-free
        auto block = juce::dsp::AudioBlock<float> (buffer);
        auto ctx   = juce::dsp::ProcessContextReplacing<float> (block);
        chain.process (ctx);
        for (int ch = 0; ch < buffer.getNumChannels(); ++ch)
            buffer.applyGainRamp (ch, 0, buffer.getNumSamples(), prevGain, gain);
        prevGain = gain;
        // NUNCA: new float[1024]; std::lock_guard lock(mutex); std::cout << gain;
    }

private:
    std::atomic<float> gainParam { 1.0f };
    float prevGain { 1.0f };
    juce::dsp::ProcessSpec spec;
    using Chain = juce::dsp::ProcessorChain<juce::dsp::Gain<float>>;
    Chain chain;
    juce::AudioBuffer<float> workBuffer;
};

Q1 · Priority Inversion

¿Por qué usar std::mutex en el audio thread puede causar un dropout incluso si nadie más está esperando el mutex en ese momento?

Ver respuesta

Priority inversion: si un thread de baja prioridad adquirió el mutex antes de que el audio thread lo necesite, el scheduler puede no desalojarlo rápidamente. El audio thread de alta prioridad queda bloqueado esperando que el thread de baja prioridad libere el mutex — tiempo completamente indeterminado que puede superar el deadline de 1.45ms.

SPSC Lock-Free Ring Buffer

template<typename T, std::size_t Capacity>
class SPSCQueue
{
    static_assert ((Capacity & (Capacity - 1)) == 0,
                   "Capacity debe ser potencia de 2");
public:
    bool push (const T& item)  // UI Thread (producer)
    {
        const auto head = head_.load (std::memory_order_relaxed);
        const auto next = (head + 1) & mask_;
        if (next == tail_.load (std::memory_order_acquire)) return false;
        buffer_[head] = item;
        head_.store (next, std::memory_order_release);
        return true;
    }

    std::optional<T> pop()  // Audio Thread (consumer)
    {
        const auto tail = tail_.load (std::memory_order_relaxed);
        if (tail == head_.load (std::memory_order_acquire)) return std::nullopt;
        T item = buffer_[tail];
        tail_.store ((tail + 1) & mask_, std::memory_order_release);
        return item;
    }

private:
    static constexpr std::size_t mask_ = Capacity - 1;
    std::array<T, Capacity> buffer_ {};
    alignas(64) std::atomic<std::size_t> head_ { 0 }; // cache-line aligned
    alignas(64) std::atomic<std::size_t> tail_ { 0 };
};

Q2 · Acquire-Release

En un SPSC queue, ¿por qué el head usa memory_order_release en el push y el consumidor usa memory_order_acquire para leerlo?

Ver respuesta

Release garantiza que todas las escrituras previas al head (el dato en buffer_[head]) sean visibles antes de que el head avance. Acquire garantiza que el consumidor vea esas escrituras cuando lee el head actualizado — forma la "sincronización acquire-release". Sin esta pareja, el compilador o la CPU podría reordenar las operaciones y el consumidor leería datos no inicializados.

M3 · Parámetros
APVTS — AudioProcessorValueTreeState
Sistema de gestión de parámetros: sync GUI↔DSP, persistencia y automatización DAW

¿Qué ofrece APVTS?

APVTS es el sistema estándar de JUCE para manejar parámetros de plugin. Proporciona: sincronización automática GUI ↔ DSP thread-safe via std::atomic<float>, serialización a XML para preset/recall, integración con la automatización del DAW y undo/redo automático en el host.

Acceso seguro en audio thread: getRawParameterValue("gain") devuelve un puntero a std::atomic<float> — guardar el puntero en el constructor y reutilizarlo en processBlock es O(1) y completamente lock-free.

Q3 · getRawParameterValue

¿Por qué se guarda el puntero devuelto por getRawParameterValue() en el constructor y se reutiliza en processBlock, en lugar de llamar a getRawParameterValue() en cada bloque?

Ver respuesta

getRawParameterValue() hace búsqueda por string en un mapa interno, que es O(n) y podría internamente alocar memoria o adquirir un lock. Guardar el puntero al inicio da acceso O(1) y completamente lock-free en el audio thread. Llamarlo en cada bloque sería una violación de las reglas de RT safety.

M4 · Algoritmos
Newton-Raphson para Ecuaciones Implícitas
Solución iterativa para modelos de saturación analógica no-lineales
out = tanh(in − k · out) [ecuación implícita] f(out) = out − tanh(in − k · out) f'(out) = 1 + k · (1 − tanh²(in − k · out)) out_{n+1} = out_n − f(out_n) / f'(out_n)

Modelo de saturación con Newton-Raphson en JUCE

class ImplicitSaturation
{
public:
    explicit ImplicitSaturation (float k = 0.8f) : k_ (k) {}

    float processSample (float in) noexcept
    {
        float out = prevOut_;  // warm start: señales continuas → convergencia 1-2 iter.

        for (int iter = 0; iter < maxIter_; ++iter)
        {
            const float arg     = in - k_ * out;
            const float tanhArg = std::tanh (arg);
            const float f       = out - tanhArg;
            const float fprime  = 1.0f + k_ * (1.0f - tanhArg * tanhArg);
            const float delta   = f / fprime;
            out                -= delta;
            if (std::abs (delta) < tolerance_) break;
        }
        prevOut_ = out;
        return out;
    }

    void processBlock (float* samples, int numSamples) noexcept
    {
        for (int i = 0; i < numSamples; ++i)
            samples[i] = processSample (samples[i]);
    }

    void reset() noexcept { prevOut_ = 0.0f; }

private:
    float k_ = 0.8f, prevOut_ = 0.0f, tolerance_ = 1e-7f;
    int   maxIter_ = 10;
};

Q4 · Warm Start

¿Por qué inicializar Newton-Raphson con el resultado anterior (warm start) mejora la convergencia en audio? ¿Cuántas iteraciones se ahorran típicamente?

Ver respuesta

Las señales de audio son continuas: la muestra actual es cercana a la anterior. El warm start comienza cerca de la solución y Newton-Raphson converge en 1-2 iteraciones en lugar de 5-7 desde cero. Esto reduce el coste computacional en ~60-75% para señales de audio típicas, crucial para cumplir el deadline del audio thread.

M5 · Optimización
Números Desnormalizados y ScopedNoDenormals
FTZ/DAZ: eliminar el cuello de botella más silencioso del DSP
FTZ (Flush To Zero): bit 15 de MXCSR — convierte resultados desnorm. a 0 DAZ (Denormals Are Zero): bit 6 de MXCSR — trata inputs denorm. como 0 MXCSR |= 0x8040 → activa ambos (FTZ + DAZ)

¿Por qué los IIR generan denormals?

Los números desnormalizados (subnormals) son IEEE 754 de magnitud <2.2×10⁻³⁸. La FPU los procesa hasta 100× más lento. En DSP, aparecen en filtros IIR cuando la señal decae a cero: las realimentaciones producen valores extremadamente pequeños. Los FIR no tienen este problema porque son feedforward, sin realimentación acumulativa.

// RAII para gestión de MXCSR
class ScopedFTZandDAZ
{
public:
    ScopedFTZandDAZ() noexcept { oldMXCSR_ = _mm_getcsr(); _mm_setcsr (oldMXCSR_ | 0x8040u); }
    ~ScopedFTZandDAZ() noexcept { _mm_setcsr (oldMXCSR_); }
    ScopedFTZandDAZ (const ScopedFTZandDAZ&) = delete;
private:
    unsigned int oldMXCSR_;
};

// JUCE 8 provee juce::ScopedNoDenormals — equivalente al patrón anterior:
void processBlock (juce::AudioBuffer<float>& buffer, juce::MidiBuffer& midi) override
{
    juce::ScopedNoDenormals noDenormals;  // Activa FTZ+DAZ — se restaura al salir del scope
    // ... DSP aqui
}

Q5 · FTZ y Threads

Un reverberador sigue produciendo glitches a pesar de tener ScopedNoDenormals en processBlock. ¿Cuál es la causa más probable y cómo se soluciona?

Ver respuesta

ScopedNoDenormals opera sobre el registro MXCSR de la CPU del thread actual. Si el reverberador usa threads adicionales (thread pool para convolución), esos hilos tienen su propio MXCSR no afectado. Solución: aplicar ScopedNoDenormals en cada hilo que procesa audio, o usar el DC-offset trick en los puntos de realimentación del reverberador.

Q6 · GIT_SHALLOW TRUE

¿Por qué es preferible usar GIT_SHALLOW TRUE en FetchContent para JUCE, especialmente en CI/CD?

Ver respuesta

Descarga solo el commit especificado sin todo el historial de git, reduciendo el tiempo de clone de ~10 minutos (historial completo de JUCE con miles de commits) a ~30 segundos. En CI/CD donde cada run descarga las dependencias desde cero, esto es crítico para mantener los pipelines de build rápidos y los costes de cómputo bajos.

M7 · Inicialización
prepareToPlay y dsp::ProcessSpec
El único lugar seguro donde puedes alocar memoria en un plugin JUCE
ProcessSpec { sampleRate, maximumBlockSize, numChannels }

prepareToPlay — zona segura de alocación

prepareToPlay() es llamado por el host antes del streaming de audio, siempre desde el main thread. Es el único lugar seguro para: alocar buffers, preparar cadenas DSP, inicializar variables de estado (filtros, delay lines) y pre-calcular tablas LUT.

void prepareToPlay (double sampleRate, int samplesPerBlock) override
{
    sr_ = sampleRate;
    juce::dsp::ProcessSpec spec;
    spec.sampleRate       = sampleRate;
    spec.maximumBlockSize = (juce::uint32) samplesPerBlock;
    spec.numChannels      = (juce::uint32) getTotalNumOutputChannels();

    chain_.prepare (spec);

    // Pre-alocar delay line (para reverb/echo)
    const int maxDelaySamples = (int) (sampleRate * 2.0);
    delayLine_.resize (maxDelaySamples + samplesPerBlock, 0.0f);

    // Pre-calcular tabla de seno (LUT) — evita std::sin en processBlock
    const int LUT_SIZE = 2048;
    sinTable_.resize (LUT_SIZE);
    for (int i = 0; i < LUT_SIZE; ++i)
        sinTable_[i] = std::sin (2.0 * juce::MathConstants<double>::pi * i / LUT_SIZE);

    // SmoothedValue: 20ms ramp para parámetros (evita zipper noise)
    gainSmooth_.reset (sampleRate, 0.02);
    gainSmooth_.setCurrentAndTargetValue (1.0f);

    chain_.reset();
    DBG ("prepareToPlay: sr=" << sampleRate << " blockSize=" << samplesPerBlock);
}

Q7 · getTailLengthSeconds

¿Por qué getTailLengthSeconds() debe retornar 2.0 si tienes un delay de hasta 2 segundos en el plugin?

Ver respuesta

Informa al host cuánto tiempo debe seguir llamando a processBlock después de que el usuario detenga la reproducción. La delay line contiene audio residual que debe vaciarse (tail) para que no se corte abruptamente. Sin este valor, el host puede detener el procesamiento prematuramente y el usuario escuchará el audio truncado.

processBlock con SmoothedValue y metering

void processBlock (juce::AudioBuffer<float>& buffer, juce::MidiBuffer& midi) override
{
    juce::ScopedNoDenormals noDenormals;
    const int numChannels = buffer.getNumChannels();
    const int numSamples  = buffer.getNumSamples();

    const float gainTarget = gainParam_->load (std::memory_order_relaxed);
    gainSmooth_.setTargetValue (gainTarget);

    if (bypassParam_->load() > 0.5f) return;

    if (gainSmooth_.isSmoothing())
    {
        // Muestra-a-muestra durante transicion (necesario para ramp lineal)
        for (int s = 0; s < numSamples; ++s)
        {
            const float g = gainSmooth_.getNextValue();
            for (int ch = 0; ch < numChannels; ++ch)
                buffer.getWritePointer(ch)[s] *= g;
        }
    }
    else
    {
        // Por bloque cuando estable (SIMD friendly — juce usa AVX/SSE internamente)
        buffer.applyGain (gainSmooth_.getCurrentValue());
    }

    // Metering: calcular RMS y enviar a UI via SPSC lock-free
    float rmsSum = 0.0f;
    for (int ch = 0; ch < numChannels; ++ch) {
        auto* data = buffer.getReadPointer (ch);
        for (int s = 0; s < numSamples; ++s) rmsSum += data[s] * data[s];
    }
    const float rms = std::sqrt (rmsSum / (numSamples * numChannels));
    meterQueue_.push ({ rms, rms, std::abs(buffer.getMagnitude(0,numSamples)) });
}

Q8 · SmoothedValue

¿Por qué SmoothedValue procesa muestra-a-muestra cuando está en transición, pero puede cambiar a por-bloque cuando está estable?

Ver respuesta

Durante la transición, cada muestra necesita un valor diferente del parámetro (rampa lineal o exponencial). Cuando es estable, todas las muestras usan el mismo valor, lo que permite multiplicación por bloque optimizada con SIMD — buffer.applyGain() usa FloatVectorOperations con AVX/SSE internamente, procesando 8 floats por instrucción.

Q9 · endChangeGesture

¿Qué problema ocurre si nunca llamas a endChangeGesture() después de un beginChangeGesture()? ¿Por qué SliderAttachment es preferible?

Ver respuesta

El host (DAW) mantiene abierto el "grab" del parámetro indefinidamente: en algunos DAWs esto bloquea la automatización de otros parámetros o impide que el host procese el evento de soltar el control en la grabación de automatización. SliderAttachment es preferible porque maneja automáticamente este protocolo completo, eliminando la posibilidad de dejar el gesture abierto por error.

class AudioProfiler
{
public:
    struct Stats {
        std::atomic<double> totalUs  { 0.0 };
        std::atomic<long>   callCount{ 0 };
        std::atomic<double> maxUs    { 0.0 };
        double averageUs() const {
            auto count = callCount.load();
            return count > 0 ? totalUs.load() / count : 0.0;
        }
    };

    class ScopedTimer {
    public:
        explicit ScopedTimer (Stats& stats)
            : stats_ (stats), start_ (std::chrono::high_resolution_clock::now()) {}
        ~ScopedTimer() {
            auto end = std::chrono::high_resolution_clock::now();
            double us = std::chrono::duration<double, std::micro> (end - start_).count();
            stats_.totalUs.fetch_add (us, std::memory_order_relaxed);
            stats_.callCount.fetch_add (1, std::memory_order_relaxed);
            double prev = stats_.maxUs.load (std::memory_order_relaxed);
            while (us > prev && !stats_.maxUs.compare_exchange_weak (prev, us,
                       std::memory_order_relaxed)) {}
        }
    private:
        Stats& stats_;
        std::chrono::time_point<std::chrono::high_resolution_clock> start_;
    };

    Stats saturationStats, filterStats, meterStats;
};

Q10 · compare_exchange_weak

¿Por qué usar compare_exchange_weak para actualizar el máximo en el profiler atómico, en lugar de un simple store?

Ver respuesta

El patrón load-compare-store tiene un race condition: entre el load y el store, otro thread puede cambiar el valor. compare_exchange_weak hace la comparación y el store atómicamente, garantizando que solo se actualiza si el valor no cambió. El bucle while reintentar si _weak falla espuriamente (permitido en ARMv8 y similares para mejor rendimiento que la versión strong).

            sr_, midFreq_, midQ_, juce::Decibels::decibelsToGain (midGain_));
        *chain_.get<2>().state = *juce::dsp::IIR::Coefficients<float>::makeHighShelf(
            sr_, hiFreq_, hiQ_, juce::Decibels::decibelsToGain (hiGain_));
    }
};

Q11 · Convolution RT-safe

juce::dsp::Convolution permite cambiar el IR en tiempo real sin glitches. ¿Cómo lo logra siendo que cargar un IR implica malloc?

Ver respuesta

La carga del nuevo IR ocurre en un background thread con el IR preparado en un buffer doble. El audio thread continúa con el IR anterior hasta que el nuevo está listo, luego hace un crossfade lock-free. El swap del puntero al IR activo es una operación atómica — nunca hay malloc en el audio thread.

Examen Feynman — Explica sin notas

Q1 · RT Violation

El siguiente processBlock llama a makeLowPass() cada bloque. ¿Qué problema introduce y cómo se soluciona correctamente?

Ver respuesta

makeLowPass() aloca memoria con new (devuelve ReferenceCountedObjectPtr) — malloc en el audio thread es una RT violation que puede causar glitches. Solución: pre-calcular los coeficientes en prepareToPlay o solo cuando el cutoff cambia significativamente (comparar con lastCutoff_), y en processBlock solo copiar los coeficientes pre-calculados: *filter_.state = precomputedCoeffs_.

Q2 · Newton-Raphson convergencia

Para out = tanh(in − 0.9·out) con in=2.0, ¿cuántas iteraciones Newton-Raphson son necesarias para error <1e-7 con y sin warm start?

Ver respuesta

Con warm start (prevOut ≈ out actual): 2-3 iteraciones, porque el error inicial es ~1e-4 y Newton cuadrático converge rápido. Sin warm start (desde 0): 5-7 iteraciones, el error inicial puede ser ~0.5. Newton-Raphson tiene convergencia cuadrática: el error se eleva al cuadrado en cada iteración. Con k=0.9, la función sí converge siempre (|k| < 1 garantiza convergencia global).

Q3 · MXCSR y threads adicionales

Un reverberador con ScopedNoDenormals sigue produciendo glitches por denormals. ¿Cuál es la causa exacta y qué dos soluciones existen?

Ver respuesta

ScopedNoDenormals opera sobre el registro MXCSR de la CPU del thread actual. Si el reverberador usa threads adicionales (thread pool para convolución particionada), esos hilos tienen su propio MXCSR sin FTZ/DAZ activo. Solución 1: aplicar ScopedNoDenormals en cada hilo que procesa audio. Solución 2: usar el DC-offset trick en los puntos de realimentación: feedback += 1e-25f; previene que los valores caigan al rango subnormal sin afectar el audio.

Q4 · APVTS y undo/redo

Un usuario reporta que la automatización se graba correctamente pero al hacer "undo" en el DAW, el parámetro no vuelve a su valor anterior. ¿Cuál es la causa?

Ver respuesta

El plugin está actualizando el parámetro directamente con getRawParameterValue()->store() en el UI thread en lugar de usar setValueNotifyingHost(). El host no recibe la notificación y no puede registrar el cambio en su sistema de undo. Siempre usar param->setValueNotifyingHost() para cambios iniciados por el usuario, precedido de beginChangeGesture() y seguido de endChangeGesture().

Q5 · Cmajor vs C++

¿Por qué los DSLs como Cmajor pueden garantizar la seguridad en tiempo real por diseño, mientras que C++ no puede garantizarla nunca estáticamente?

Ver respuesta

El compilador de Cmajor analiza estáticamente todo el grafo de procesadores y rechaza en tiempo de compilación cualquier construcción que pueda causar allocaciones de heap, llamadas de sistema bloqueantes o excepciones. En C++, estas son decisiones del programador en tiempo de ejecución: new, std::mutex, system calls pueden aparecer en cualquier función llamada desde processBlock y el compilador no las rechaza. La garantía de Cmajor es estructural, no convencional.
