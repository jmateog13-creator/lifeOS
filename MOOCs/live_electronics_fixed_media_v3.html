Explica con tus palabras por qué se usan varios entornos (Max, SC, FluCoMa) en el mismo workflow en lugar de uno solo.

Paradigma Patchbay

Tipos de objetos en Max/MSP

  • Objetos de control: generan/procesan mensajes — e.g., metro, counter, pack
  • Objetos de audio (~): procesan señal de audio a la tasa del DSP — e.g., cycle~, gain~, dac~
  • Objetos de datos: almacenan y manipulan datos — e.g., coll, dict, table
  • Objetos de interfaz: GUI — e.g., slider, dial, number, live.dial

Q1 · Patchbay

¿Cuál es la diferencia entre un cable de control (gris) y un cable de señal (amarillo) en Max/MSP?

Ver respuesta

Los cables grises transportan mensajes de control (bangs, números, símbolos, listas) que se procesan a la tasa de mensajes del programa. Los cables amarillos (~) transportan señal de audio que se procesa a la tasa de muestreo (típicamente 44100 o 48000 muestras por segundo) en bloques de audio (block_size). Los objetos de audio tienen el sufijo ~ (tilde) y operan en el mundo de la señal con latencia mínima determinada por el block size.

Q2 · Síntesis FM

Explica qué son el ratio y el índice de modulación en síntesis FM y cómo afectan al timbre resultante.

Ver respuesta

El ratio fm:fc determina la frecuencia de la moduladora relativa a la portadora. Ratios enteros (1:1, 2:1) producen timbres armónicos; ratios no enteros generan espectros inarmónicos (campanas, gongs). El índice de modulación (mi) escala la desviación de frecuencia: mi bajo = timbre cercano al seno, mi alto = espectro rico con muchos sidebands. Con mi=0 obtienes la portadora pura; aumentando mi se añaden progresivamente más armónicos/parciales al timbre.

¿Qué es gen~?

gen~: DSP de bajo nivel en Max

gen~ es un subentorno dentro de Max/MSP que compila código a DSP nativo. Todas las operaciones son sample-accurate. Usa operadores matemáticos directamente: +, *, sin, exp. Memoria persistente: history (1 sample), delay (N samples), data (buffer). Se compila a código nativo: muy eficiente, ideal para síntesis compleja.

Oscilador y filtros en gen~

/* gen~ patch — se escribe dentro del objeto gen~ en Max */
// Oscilador de fase acumulada (phasor)
History phase(0);
param freq(440);
phase = wrap(phase + freq / samplerate, 0, 1);
out1 = sin(phase * TWOPI);

// Filtro de 1 polo (low-pass simple)
History yn(0);
param alpha(0.1);
yn = alpha * in1 + (1 - alpha) * yn;
out1 = yn;

// Karplus-Strong en gen~
param freq(220);
param damp(0.999);
int N = int(samplerate / freq);
data buf(N);
History idx(0);
val = peek(buf, idx);
History prev(0);
filtered = (val + prev) * 0.5 * damp;
prev = val;
poke(buf, filtered, idx);
idx = (idx + 1) % N;
out1 = filtered;

Q3 · gen~ vs objetos Max

¿Por qué gen~ es más eficiente que los objetos estándar de Max para DSP de bajo nivel?

Ver respuesta

gen~ compila su contenido a código nativo (C++ compilado) en lugar de interpretar objetos Max en tiempo de ejecución. Las operaciones son sample-accurate: no hay overhead de bloque entre operaciones, lo que elimina latencias adicionales. Además, gen~ permite escribir filtros IIR personalizados, osciladores de fase acumulada y algoritmos como Karplus-Strong directamente con matemáticas a nivel de muestra. En comparación, los objetos Max estándar tienen overhead de gestión de mensajes y comunicación entre objetos.

¿Qué es RNBO~?

Export-first DSP desde Max

RNBO~ (pronunciado "rainbow") es un subentorno de Max 8 que permite compilar patches a código C++ portable. Un mismo patch RNBO puede exportarse a: VST3/AU, Web Audio API, Max for Live, hardware embebido (Raspberry Pi, Daisy) y código C++ nativo.

Q4 · RNBO~ exportación

¿Cuál es la principal ventaja de RNBO~ sobre un patch Max/MSP estándar para distribución de software musical?

Ver respuesta

RNBO~ permite exportar el mismo patch a múltiples targets sin cambiar el código fuente: como VST3/AU para cualquier DAW, como Web Audio API para el navegador, como Max for Live device, o como código C++ para hardware embebido (Raspberry Pi, Daisy Electro-Smith). Esto significa que un algoritmo de síntesis desarrollado una sola vez puede distribuirse como plugin profesional, herramienta web y dispositivo hardware, manteniendo el mismo comportamiento sonoro en todas las plataformas.

Arquitectura de SuperCollider

Cliente + Servidor

  • sclang: lenguaje de programación (cliente). Aquí escribes el código.
  • scsynth: servidor de síntesis (DSP). Recibe instrucciones via OSC.
  • Comunicación: sclang ↔ scsynth via OSC en localhost
// Iniciar servidor
s.boot;
s.waitForBoot({ "Servidor listo".postln; });

// SynthDef con UGens
SynthDef(\fm_synth, { |out=0, freq=440, ratio=2, index=3, amp=0.3, gate=1|
    var env = EnvGen.ar(Env.adsr(0.01, 0.2, 0.7, 0.5), gate,
                         doneAction: Done.freeSelf);
    var mod = SinOsc.ar(freq * ratio) * index * freq;
    var car = SinOsc.ar(freq + mod) * amp * env;
    Out.ar(out, car ! 2);
}).add;

// Pbind: patrón que instancia SynthDefs
Pbind(
    \instrument, \fm_synth,
    \freq, Pseq([261.63, 329.63, 392, 523.25], inf),
    \dur, 0.5
).play;

Q5 · SynthDef y UGens

Explica la diferencia entre sclang y scsynth, y qué protocolo usa uno para comunicarse con el otro.

Ver respuesta

sclang es el lenguaje de programación (cliente) donde escribes el código SuperCollider — gestiona la lógica, patrones, timing y control. scsynth es el servidor de síntesis (DSP) que genera el audio real procesando grafos de UGens. La comunicación se hace via OSC (Open Sound Control) sobre UDP en localhost. Cuando defines un SynthDef con .add, sclang lo compila y lo envía al servidor. Cuando llamas Synth(\nombre), sclang manda un mensaje OSC a scsynth diciéndole que instancie esa definición de síntesis.

JITLib: Just-In-Time Library

Live Coding en SuperCollider

JITLib permite sustituir nodos del servidor en tiempo real sin interrumpir el audio. Los NodeProxy son placeholders que contienen y reemplazan synths de forma transparente: Ndef (NodeProxy con nombre global), ProxySpace (convierte ~var en NodeProxy automáticamente), Tdef (Proxy para tareas), Pdef (Proxy para Pbind).

// Crear un Ndef y redefinir en vivo (crossfade automático)
Ndef(\synth1, { SinOsc.ar(440) * 0.3 }).play;
Ndef(\synth1, { Saw.ar(440) * 0.2 });  // crossfade automático

// ProxySpace convierte ~variables en NodeProxy
p = ProxySpace.push(s);
~kick = { SinOsc.ar(70 * XLine.ar(3, 0.5, 0.3), 0, 0.8) }.play;
~melody = Pbind(\instrument, \fm_synth,
    \freq, Pseq([261, 329, 392], inf), \dur, 0.25);

// Pdef: patrón en vivo modificable sin interrupción
Pdef(\mel, Pbind(\instrument, \default,
    \freq, Prand([261, 329, 392, 440], inf), \dur, 0.5
)).play;

Q6 · JITLib

¿Qué hace JITLib de SuperCollider que es imposible de replicar en Max/MSP estándar?

Ver respuesta

JITLib permite sustituir nodos del servidor de síntesis (NodeProxy) en tiempo real sin interrupciones de audio: cambias el SynthDef que produce un sonido mientras suena, y el servidor hace crossfade automático entre el nodo antiguo y el nuevo según fadeTime. En Max/MSP los objetos DSP no pueden redefinirse en tiempo de ejecución sin interrupciones. Además, ProxySpace crea un "espacio de nombres vivo" donde todas las referencias se resuelven dinámicamente. Esto hace de JITLib la herramienta definitiva para live coding en SuperCollider.

¿Qué es FluCoMa?

Fluid Corpus Manipulation Toolkit

FluCoMa es un proyecto del AHRC (UK) que trae machine learning creativo a los entornos de programación musical. Disponible para Max/MSP, SuperCollider y PureData.

  • Segmentación: dividir corpus por onsets, transientes, novedad espectral
  • Descriptores: extraer MFCCs, espectral centroid, loudness, pitch
  • Análisis: UMAP, PCA, t-SNE para reducción dimensional
  • Clasificación: MLP, KDTree, KNN para clasificación en tiempo real
// FluCoMa en SuperCollider: segmentar → descriptores → clasificar
~buf = Buffer.read(s, "corpus.wav");
~slices = Buffer.alloc(s, 128);

// 1. Segmentación por onsets
FluidBufOnsetSlice.process(s, source: ~buf, indices: ~slices,
    metric: 1, threshold: 0.3, action: { "Slices OK".postln; });

// 2. Extraer MFCCs
~mfcc_buf = Buffer.alloc(s, 128 * 13);
FluidBufMFCC.process(s, source: ~buf, features: ~mfcc_buf,
    numCoeffs: 13, action: { "MFCCs OK".postln; });

// 3. Clasificar en tiempo real
~mlp = FluidMLPClassifier.new(s, [13, 8, 4]);
~mlp.train(maxEpochs: 1000, learnRate: 0.01, action: {
    "Clasificador entrenado".postln; });

Q7 · FluCoMa en live electronics

¿Qué es FluCoMa y por qué es relevante para el live electronics creativo?

Ver respuesta

FluCoMa (Fluid Corpus Manipulation) es un conjunto de herramientas para analizar, segmentar y clasificar corpus de sonido usando machine learning directamente dentro de Max/MSP, SuperCollider o PureData. Permite: segmentar automáticamente corpus en granos significativos mediante onset/transient detection; extraer descriptores (MFCCs, centroid espectral); entrenar clasificadores MLP o reducir dimensionalidad con UMAP/PCA; y navegar el corpus en tiempo real por similitud. Para live electronics significa poder clasificar y seleccionar material sonoro de forma inteligente durante la performance sin intervención manual.

Q8 · Granular vs Phase Vocoder

¿Cuál es la diferencia entre síntesis granular y phase vocoder para time-stretching?

Ver respuesta

La síntesis granular divide el sonido en "granos" (ventanas cortas de 1-100ms) y los reensambla con parámetros de densidad, dispersión y posición — opera en el dominio temporal. El phase vocoder analiza el audio fotograma a fotograma via FFT, extrayendo amplitudes y fases de cada bin de frecuencia, lo que permite manipular independientemente pitch y time — opera en el dominio frecuencial. Para time-stretching, el phase vocoder produce resultados más transparentes porque mantiene la coherencia de fase; el granular introduce artefactos de "batido" pero suena más texturalmente interesante para efectos creativos.

Q9 · VBAP vs Ambisonics

¿Cómo funciona VBAP y en qué difiere de Ambisonics en términos de sweet spot y escalabilidad?

Ver respuesta

VBAP (Vector Base Amplitude Panning) posiciona una fuente sonora activando solo los 2-3 altavoces más cercanos a la dirección deseada, calculando sus amplitudes mediante álgebra vectorial. Es preciso en ángulo pero colapsa si el oyente no está en el sweet spot central, y requiere reconfiguración si cambia el layout de altavoces. Ambisonics codifica toda la información espacial en canales independientes del sistema de reproducción (B-format/HOA); luego un decoder adapta esa señal a cualquier disposición. Ambisonics es más flexible y escalable; VBAP es más eficiente para pocas fuentes en sistemas fijos conocidos.

Q10 · Mastering electroacústico

¿Cuál es el target de LUFS para distribución en festival electroacústico según EBU R128, y qué significa TP=-1?

Ver respuesta

El target EBU R128 para broadcast y festivales electroacústicos es -23 LUFS integrado (LUFS FS = Loudness Units Full Scale). Para distribución online el target es -16 LUFS. TP=-1 significa True Peak máximo de -1 dBTP: el nivel más alto que puede alcanzar la onda tras reconstrucción en el D/A (que puede superar los picos de muestra por interpolación), limitado a -1 dBFS para dejar headroom y evitar saturación en sistemas de reproducción. Se mide con medidores compatibles con ITU-R BS.1770-4 que aplican oversampling para detectar verdaderos picos entre muestras.

Q11 · Mapeo sensor→parámetro

¿Por qué es preferible usar mapeo exponencial o cuadrático en lugar de lineal para sensores de presión o distancia en instrumentos electrónicos?

Ver respuesta

La percepción humana de parámetros como frecuencia, amplitud y timbre es generalmente logarítmica o no lineal. Un mapeo lineal (val → param) resulta en que la mayoría del recorrido útil del sensor queda comprimido en rangos poco perceptibles, mientras que los extremos son demasiado sensibles o insensibles. El mapeo exponencial (val² o exponencial) distribuye la sensibilidad de forma más proporcional a la percepción: más resolución en los valores bajos donde los cambios son más perceptibles, y transiciones más suaves en los valores altos. Para frecuencia se usa mapping logarítmico porque las octavas son logarítmicas.

Q13 · Karplus-Strong

Explica paso a paso cómo funciona el algoritmo Karplus-Strong y por qué produce el timbre de una cuerda pulsada.

Ver respuesta

Karplus-Strong simula cuerdas pulsadas con este proceso: (1) llena un buffer circular (delay line) de longitud N = SR/freq con ruido aleatorio — esto simula la energía inicial de la pulsación; (2) en cada sample, el output es el valor actual del buffer; (3) ese sample se promedia con el siguiente sample del buffer (filtro paso bajo de orden 1): filtered = (val + prev) * 0.5 * decay; (4) el resultado filtrado se escribe de vuelta al buffer (realimentación). El resultado: la longitud del buffer determina la frecuencia fundamental (f = SR/N), y el filtrado iterativo elimina progresivamente las frecuencias altas — exactamente como una cuerda real pierde energía antes en los armónicos altos que en los bajos.

E1 · Granular vs Phase Vocoder

¿Cuál es la diferencia fundamental entre síntesis granular y phase vocoder para time-stretching?

Ver respuesta

La síntesis granular divide el sonido en granos (ventanas de 1-100 ms) y los reensambla con parámetros de densidad, dispersión y posición — opera en el dominio temporal. El phase vocoder analiza fotograma a fotograma via FFT, extrayendo amplitudes y fases de cada bin, permitiendo manipular pitch y time independientemente — opera en el dominio frecuencial. Granular introduce artefactos de "cloud" pero es creativo; el phase vocoder es más transparente para stretching de alta calidad sin artefactos de pitcheo.

E2 · VBAP vs Ambisonics

¿Cómo funciona VBAP y en qué difiere de Ambisonics en términos de sweet spot y escalabilidad?

Ver respuesta

VBAP activa solo los 2-3 altavoces más cercanos a la dirección deseada, calculando amplitudes con álgebra vectorial. Es preciso en ángulo pero depende del sweet spot y requiere reconfiguración si cambia el layout. Ambisonics codifica la información espacial en canales independientes del sistema (B-format/HOA) y un decoder adapta a cualquier disposición. Ambisonics es más flexible y portable; VBAP más eficiente para pocas fuentes en sistemas fijos conocidos. Ambisonics escala a HOA de orden N con (N+1)² canales para mayor resolución espacial.

E3 · JITLib live coding

¿Qué hace JITLib de SuperCollider imposible de replicar en Max/MSP estándar?

Ver respuesta

JITLib permite sustituir nodos del servidor (NodeProxy) en tiempo real sin interrupciones: cambias el SynthDef mientras suena y el servidor hace crossfade automático según fadeTime. En Max/MSP los objetos DSP no pueden redefinirse en tiempo de ejecución sin interrupciones. ProxySpace crea un espacio de nombres vivo donde todas las referencias se resuelven dinámicamente. Esto hace de JITLib la herramienta definitiva para live coding: puedes reescribir y recargar algoritmos enteros de síntesis mientras la performance continúa sin un solo click de audio.

E4 · Karplus-Strong

¿Cómo funciona el modelo de Karplus-Strong y qué parámetros controlan el timbre?

Ver respuesta

Karplus-Strong: (1) llena un buffer circular de longitud N=SR/freq con ruido aleatorio (energía inicial); (2) cada sample: leer del buffer; (3) promediar con el siguiente sample y multiplicar por decay (filtro LP + pérdida de energía); (4) escribir el resultado de vuelta al buffer (realimentación). La frecuencia está determinada por N, el decay controla la duración (más cerca de 1.0 = sustain más largo), y el factor de promediado controla la "stiffness" de la cuerda — aumentando la pérdida diferencial de armónicos altos vs bajos.

E5 · FluCoMa en performance

¿Qué es FluCoMa y por qué es relevante para el live electronics creativo?

Ver respuesta

FluCoMa (Fluid Corpus Manipulation) es un toolkit para analizar, segmentar y clasificar corpus de sonido usando machine learning directamente en Max/MSP, SuperCollider o PureData. Permite: segmentar corpus en granos por onsets/transientes; extraer descriptores (MFCCs, centroid espectral); entrenar clasificadores MLP o reducir dimensionalidad con UMAP; navegar el corpus por similitud en tiempo real. Para live electronics significa clasificar y seleccionar material sonoro de forma inteligente durante la performance, creando instrumentos que "aprenden" del sonido en vivo.
