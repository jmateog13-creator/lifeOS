1.3 MFCC paso a paso

  1. STFT → magnitud²
  2. Banco de filtros Mel (típico 40 o 128 bandas)
  3. log(·) → log-mel
  4. DCT → coeficientes. Quedarse con los 13 primeros
  5. Añadir Δ y Δ² → 39 features por frame
import librosa, numpy as np
y, sr = librosa.load("audio.wav", sr=22050)
mfcc  = librosa.feature.mfcc(y=y, sr=sr, n_mfcc=13, hop_length=512)
d1    = librosa.feature.delta(mfcc, order=1)
d2    = librosa.feature.delta(mfcc, order=2)
feat  = np.vstack([mfcc, d1, d2])   # shape (39, T)
vec = np.hstack([feat.mean(axis=1), feat.std(axis=1)])  # 78-dim

B Laboratorio Guiado · Extractor canónico de 7 features 
import librosa, numpy as np, pandas as pd
from pathlib import Path
SR, N_FFT, HOP = 22050, 2048, 512
FEATS = ["mfcc","chroma","contrast","centroid","rolloff","zcr","rms"]
def extract(path):
    y, _ = librosa.load(path, sr=SR, mono=True)
    f = {}
    f["mfcc"]     = librosa.feature.mfcc(y=y, sr=SR, n_mfcc=13, hop_length=HOP)
    f["chroma"]   = librosa.feature.chroma_cqt(y=y, sr=SR, hop_length=HOP)
    f["contrast"] = librosa.feature.spectral_contrast(y=y, sr=SR, hop_length=HOP)
    f["centroid"] = librosa.feature.spectral_centroid(y=y, sr=SR, hop_length=HOP)
    f["rolloff"]  = librosa.feature.spectral_rolloff(y=y, sr=SR, hop_length=HOP)
    f["zcr"]      = librosa.feature.zero_crossing_rate(y, hop_length=HOP)
    f["rms"]      = librosa.feature.rms(y=y, hop_length=HOP)
    vec = []
    for k in FEATS:
        vec.append(f[k].mean(axis=1))
        vec.append(f[k].std(axis=1))
    return np.concatenate(vec)

E Espacio Feynman

Q1 · Waveform directo en SVM

Tu jefe pregunta: "¿Por qué no metemos directamente el waveform en un SVM?". Responde antes de mirar.

Ver respuesta

Porque la dimensionalidad sería gigantesca (~660k samples para 30 s @ 22050 Hz), las muestras no son comparables temporalmente y un SVM con kernel RBF colapsaría sin features que comprimieran información perceptual. La solución: comprimir a 78 dims con MFCC mean/std (curse of dimensionality + falta de invariancia a desplazamiento + información perceptual diluida).

Q2 · Invariancia a octava

¿Qué feature es invariante a transposición de octava?

Ver respuesta

Chroma. Por construcción suma la energía de todas las octavas en 12 bins (pitch class profile), colapsando C2, C3, C4, C5… en un único bin "C". Es la feature canónica para chord detection y key estimation.

E Espacio Feynman

Q1 · SMOTE antes del split

¿Por qué es un error grave aplicar SMOTE antes del split train/test?

Ver respuesta

Introduces leakage: SMOTE genera ejemplos sintéticos a partir de los reales y, si sintetizas usando muestras que luego van al test, el modelo "ve" información del test durante el entrenamiento, inflando el rendimiento reportado. Por eso imblearn.Pipeline aplica SMOTE sólo a fold de train en cada iteración de CV.

Q2 · class_weight='balanced' perjudicial

¿Cuándo es perjudicial usar class_weight='balanced' en lugar de re-balancear muestras?

Ver respuesta

Cuando la métrica final es accuracy y la distribución de test refleja la del mundo real: balanced sobrepondera la minoritaria y degrada la accuracy global frente a un threshold tuning post-hoc. class_weight optimiza F1 implícitamente, no accuracy.

  3. Reentrenar con todo el 80% usando mejores hyperparams
  4. Evaluar UNA vez en test

Q1 · Overfitting XGBoost

Tu modelo XGBoost da 87% accuracy en train y 67% en validación. ¿Diagnóstico?

Ver respuesta

Overfitting fuerte. Acciones: reducir max_depth, aumentar reg_lambda, bajar learning_rate, activar subsample < 1.0, añadir early stopping. La brecha train-val de 20 puntos grita overfitting.

Q2 · Distribución de producción diferente

Tu cliente usa accuracy global como KPI. La distribución de su producción es muy diferente a la del dataset etiquetado. ¿Cómo ajustar?

Ver respuesta

Reponderar la evaluación con los pesos de la distribución de producción (importance weighting), o re-etiquetar una muestra representativa, o reportar accuracy estratificada por clase. Reportar accuracy de lab cuando la distribución de producción difiere es lying-with-statistics.

Q1 · RF importance vs permutation importance

Tu RF dice que "MFCC[3] mean" es la feature más importante. Permutation importance dice que es "spectral_centroid mean". ¿Quién tiene razón?

Ver respuesta

Depende. RF importance puede inflar features con alta cardinalidad/entropy y subestimar features correlacionadas. Permutation evalúa el impacto real en métrica pero subestima features correlacionadas (otra feature compensa). Lo correcto: reportar AMBAS y SHAP encima. Si los tres coinciden, alta confianza.

5.2 Arquitectura canónica VGG-style + regularización

import torch, torch.nn as nn

class AudioCNN(nn.Module):
    def __init__(self, n_classes=10, n_mels=128):
        super().__init__()
        def block(in_c, out_c):
            return nn.Sequential(
                nn.Conv2d(in_c, out_c, kernel_size=3, padding=1),
                nn.BatchNorm2d(out_c), nn.ReLU(inplace=True),
                nn.Conv2d(out_c, out_c, kernel_size=3, padding=1),
                nn.BatchNorm2d(out_c), nn.ReLU(inplace=True),
                nn.MaxPool2d(2),
            )
        self.backbone = nn.Sequential(block(1,32),block(32,64),block(64,128),block(128,256))
        self.gap = nn.AdaptiveAvgPool2d(1)
        self.fc  = nn.Sequential(nn.Dropout(0.4), nn.Linear(256, n_classes))
    def forward(self, x):
        h = self.backbone(x)
        return self.fc(self.gap(h).flatten(1))

Q1 · MaxPool agresivo en frecuencia

¿Por qué un MaxPool agresivo en el eje frecuencia puede destruir la performance en clasificación de instrumentos?

Ver respuesta

Porque la frecuencia NO es invariante a traslación: una signature espectral a 200 Hz (bombo) ≠ 2000 Hz (platillo). Pooling agresivo en frecuencia mezcla información tonal crítica. Por eso muchas arquitecturas usan pool sólo en tiempo o asimétrico (1×2).

Tu cliente entrena from scratch sobre 1.500 clips y obtiene 0.55 F1. Tú aplicas linear probe sobre embeddings PaSST y obtienes 0.78. Argumenta el trade-off.

Q1 · SDR 8.5 dB

Tu modelo separa vocals con SDR 8.5 dB en MUSDB18. ¿Qué significa exactamente ese número?

Ver respuesta

Es la relación entre energía de la señal estimada bien recuperada y energía del error (suma de interferencia + artefactos + ruido). 8.5 dB ≈ ratio energético 7:1. Calidad audible buena, no de masterización. Para comercial-master grade se necesitan 10+ dB.

Q1 · Error de octava en tempo

¿Por qué un beat tracker reporta a menudo el doble (o la mitad) del tempo correcto?

Ver respuesta

Porque el "beat" es perceptualmente ambiguo: una salsa a 90 BPM puede ser igualmente "sentida" a 180 BPM (subdivisiones). El modelo aprende los dos modos. Por eso MIREX reporta Acc1 (estricta) y Acc2 (tolera octava). Es un problema fundamental, no de implementación.

Q1 · Accuracy tóxica en multi-label

¿Por qué accuracy es una métrica tóxica en multi-label classification?

Ver respuesta

Porque "exact match accuracy" pide acertar TODAS las etiquetas simultáneamente. Con 195 tags, predecir 194 correctamente y 1 mal cuenta como 0. Métricas razonables: F1-sample (parcial), mAP (per-class) o Hamming loss.

12.2 Paired t-test + Cohen's d

from scipy.stats import ttest_rel
from sklearn.model_selection import StratifiedKFold, cross_val_score
skf = StratifiedKFold(n_splits=10, shuffle=True, random_state=42)
scores_A = cross_val_score(model_A, X, y, cv=skf, scoring="f1_macro")
scores_B = cross_val_score(model_B, X, y, cv=skf, scoring="f1_macro")
t, p = ttest_rel(scores_A, scores_B)
diff = scores_A.mean() - scores_B.mean()
cohen_d = diff / np.std(scores_A - scores_B, ddof=1)
print(f"Diff: {diff:.4f}, Cohen's d: {cohen_d:.3f}, p: {p:.4f}")
# d < 0.2: pequeño; 0.5: medio; 0.8: grande

Q1 · p-value sin contexto

"Mi modelo da 92% accuracy, p=0.001 vs baseline. ¡Es mucho mejor!". ¿Qué le falta a esta afirmación?

Ver respuesta

Falta accuracy del baseline (¿91%? ¿70%?), tamaño del test (con 1M muestras casi todo es p<0.001), effect size (Cohen's d), CI 95% de la diferencia, y si la mejora es práctica (¿0.5 puntos sobre baseline humano de 95%?). Sin contexto, "p=0.001" es un número vacío. Reporta diff absoluta + CI + effect size + métrica relativa al baseline humano.

Q1 · Data drift en producción

Tu modelo en lab tiene 0.91 F1. En producción a las 3 semanas baja a 0.78. ¿Diagnóstico?

Ver respuesta

Data drift: la distribución de input (calidad de mic, codec, ruido de fondo, demografía de usuarios) cambió respecto a training. El modelo sigue intacto, lo que cambió es el input. Solución: monitorización drift + reentrenamiento periódico con muestras recientes. Decay temporal en producción = data drift al 95%.

Q1 · Defensa del modelo final

¿Por qué elegiste tu modelo final sobre los baselines? Defiéndelo.

Ver respuesta

Triangulación de evidencias: diff de mAP con CI, p-value paired t-test, Cohen's d, latencia, tamaño, interpretabilidad. NO sólo "porque da más". Si la diff es marginal, también discute si vale la complejidad añadida. El ingeniero ML defiende con métricas, no con intuición.

Q2 · Distribución cambia mañana

¿Qué pasaría si la distribución de tus datos cambia mañana?

Ver respuesta

Tu drift monitoring debería disparar alarma cuando Wasserstein de cualquier feature key supera threshold. Plan: reentrenar con muestras frescas, re-calibrar thresholds por clase, comparar contra modelo actual con paired test antes de promocionar. Monitorización no es opcional, es arquitectura.

Q3 · Discriminación demográfica

¿Tu modelo discrimina demográficamente?

Ver respuesta

Análisis por subgrupos (idioma, época, género de la canción). Si hay disparidad sistemática, documentarla en model card sección "Limitations & Biases". Considerar reweighting o data augmentation dirigida. La ética no es opcional en sistemas de producción.

Q4 · Audio fuera de distribución

¿Qué hace tu modelo cuando recibe un audio que no se parece a nada del training?

Ver respuesta

Idealmente: anomaly detector (M10) flag low-confidence. El servicio devuelve "uncertain" con probabilidad y sugerencias. Nunca un único tag arbitrario con confidence inflada. El modelo calibrado + OOD detector es arquitectura mínima responsable.

Tabla con coste cloud (latencia × throughput × $/hora vCPU) vs valor business (tags automáticos × tracks/día × valor por tag manual evitado). ROI explícito. El ingeniero habla de negocio, no sólo de arquitectura.
