Pareto del sistema visual en vivo

El 80% del rendimiento y fiabilidad recae en el 20% de la teoría técnica: gestión GPU vs CPU (elección de codec), álgebra de blend modes (composición matemática de píxeles), análisis FFT (modulación audio-reactiva) y protocolos de sincronización (Ableton Link, SMPTE LTC). Dominar estos cuatro ejes elimina el 80% de los problemas que aparecen en escenario.

M1 · Codecs
DXV / HAP vs H.264 — GPU Decompression
Intra-frame vs inter-frame, DXT1/DXT5, PCIe pipeline, batch encoding
Inter-frame (H.264): GOP → vectores predictivos → CPU reconstruye frames → stuttering
Intra-frame (HAP/DXV): cada frame = imagen completa DXT1/DXT5 → GPU descomprime en paralelo
Seek latency: H.264 O(GOP size) vs HAP/DXV O(1) — acceso directo a cualquier frame

¿Por qué H.264 mata el rendimiento en VJ?

H.264 usa compresión inter-frame con Group of Pictures (GOP). Para decodificar el frame 87, la CPU necesita el frame I más cercano (quizás el 60) y todos los frames P/B intermedios. En VJ, el scrubbing (cambio de velocidad, bucles no lineales, trigger de clip) obliga a decodificaciones fuera de orden → la CPU no puede seguir el ritmo → dropped frames.

  • DXV 3 (Resolume native): Compresión LZF + DXT1/DXT5. El archivo .mov viaja por PCIe a VRAM; los núcleos de textura de la GPU descomprimen en paralelo sin tocar la CPU.
  • HAP (open source): Usa Snappy compression + DXT1/DXT5. Compatible con múltiples VJ software. HAP Q = mayor calidad (mayor bitrate). HAP Alpha = canal alfa transparente.
  • Ratio de bitrate: HAP es 5-10× más pesado en disco que H.264, pero el acceso es instantáneo (O(1) seek).

Estas son las mismas instrucciones de hardware que usa el motor de un videojuego para texturas. La GPU ya tiene circuitería dedicada → descompresión sin coste de ciclos de cómputo general.

Pipeline Completo: Disco → Pantalla

# ======================================================
# RUTA DE DATOS SIN CUELLO DE BOTELLA (HAP/DXV)
# ======================================================
# 1. NVMe M.2 (3000-7000 MB/s) → archivo .mov HAP
# 2. PCIe Gen4 x16 (64 GB/s) → DMA directo a VRAM
# 3. GPU Texture Units → decompress DXT1/DXT5 en paralelo
# 4. VRAM (RTX 4090: 24GB GDDR6X) → frame en VRAM
# 5. Resolume Compositor → blend modes en shader
# 6. DisplayPort/HDMI → Datapath Fx4 → LED/Projector

# ======================================================
# RUTA CON CUELLO DE BOTELLA (H.264)
# ======================================================
# 1. NVMe → archivo .mp4 H.264
# 2. CPU (decodificación software) → 4-8 núcleos saturados
#    • reconstrucción GOP: I-frame + N P-frames + B-frames
#    • seek no-lineal = reconstrucción desde I-frame más cercano
# 3. RAM (sistema) → copia por memcpy
# 4. PCIe → upload a VRAM (copia extra)
# 5. GPU compositor → frame disponible con latencia variable

# CONCLUSIÓN: H.264 añade 2 copias de datos extra + saturación CPU
# HAP/DXV: 0 copias CPU, descompresión directa en GPU
M2 · Blend Modes
Álgebra de Modos de Fusión
Add, Multiply, Alpha, Screen — operaciones matriciales sobre píxeles [0,1]
Valores normalizados: R,G,B,A ∈ [0,1]
Add: Z = min(1.0, A + B) — suma lumínica, negro = neutro
Multiply: Z = A × B — modulación de amplitud, VCA visual
Alpha: Z = A_color×A_alpha + B_color×(1−A_alpha) — crossfade con antialiasing
Screen: Z = 1 − ((1−B) × (1−A)) — luminosidad limpia, inversión doble
Add
Z = min(1, A+B)
Sumar fotones de capas sin máscaras. Negro (0,0,0) = neutro. Riesgo: clipping en blanco.
Multiply
Z = A × B
VCA visual: valores oscuros de A suprimen B. Blanco (1,1,1) = neutro. Oscurece siempre.
Alpha
Z = Ac·Aα + Bc·(1−Aα)
Crossfade por píxel conservando antialiasing. Requiere canal alfa en la capa superior.
Screen
Z = 1−(1−B)(1−A)
Luminosidad limpia. Equivale a Add pero sin clipping abrupto. Negro = neutro.

Implementación en Shader GLSL

// Fragment shader — todos los blend modes
// A = textura capa superior, B = textura capa inferior
uniform sampler2D texA;
uniform sampler2D texB;
uniform int blend_mode;  // 0=Add, 1=Multiply, 2=Alpha, 3=Screen

void main() {
    vec2 uv = gl_FragCoord.xy / iResolution.xy;
    vec4 A = texture2D(texA, uv);
    vec4 B = texture2D(texB, uv);

    vec3 result;

    if (blend_mode == 0) {
        // ADD — suma lumínica, clamp para evitar HDR overflow
        result = min(vec3(1.0), A.rgb + B.rgb);

    } else if (blend_mode == 1) {
        // MULTIPLY — modulación de amplitud (equivalente a VCA en audio)
        // Negro (0) anula cualquier valor de B → efecto máscara
        result = A.rgb * B.rgb;

    } else if (blend_mode == 2) {
        // ALPHA COMPOSITING (Porter-Duff "over")
        // A_alpha controla cuánto de A y B se mezcla
        result = A.rgb * A.a + B.rgb * (1.0 - A.a);

    } else if (blend_mode == 3) {
        // SCREEN — inversión multiplicativa
        // Equivale a: 1 - (invA * invB)
        result = 1.0 - (1.0 - A.rgb) * (1.0 - B.rgb);
    }

    gl_FragColor = vec4(result, 1.0);
}

// Otros blend modes útiles en VJ:
// Difference: abs(A - B)      → inversión en zonas iguales
// Overlay:   A<0.5 ? 2AB : 1-2(1-A)(1-B)  → contraste extremo
// Subtract:  max(0, B - A)    → sustracción de color

M3 · FFT Reactivo
Análisis Audio-Reactivo por FFT
1024 bins, resolución de banda, mapping de frecuencias a parámetros visuales
Ancho de bin: df = (Fs/2) / N_bins
Con Fs=48kHz, N=1024: df = 24000/1024 ≈ 23.4 Hz/bin
Bin de frecuencia f: bin_index = round(f / df)
Energía de banda: E = Σ |X[k]|² para k en rango de la banda

Los 1024 Bins de Resolume Arena

Resolume ejecuta una FFT de 1024 puntos sobre el audio de entrada (típicamente 48kHz mono). Divide el espectro audible (0–24kHz) en 1024 bandas de ≈23.4Hz cada una. Estas bandas se agrupan en rangos frecuenciales que se pueden asignar a parámetros de clip:

Audio Análisis en Resolume — Configuración Práctica

# En Resolume Arena:
# 1. AUDIO → Audio Analysis panel
#    • Seleccionar fuente: ASIO / Core Audio input channel
#    • FFT size: 1024 (por defecto) → df ≈ 23Hz/bin
#
# 2. Asignar banda a parámetro de clip:
#    • Click derecho sobre parámetro (ej. Opacity de una capa)
#    • "Link to Audio Analysis" → seleccionar banda frecuencial
#    • Configurar:
#      - Sensitivity (ganancia del análisis)
#      - Smoothing (attack/release del envelope)
#      - Min/Max output range
#
# 3. Para análisis avanzado (Python → OSC → Resolume):

import librosa
import sounddevice as sd
from pythonosc import udp_client
import numpy as np

osc = udp_client.SimpleUDPClient("127.0.0.1", 7000)  # puerto OSC Resolume
BLOCK = 1024
SR = 48000

def audio_callback(indata, frames, time, status):
    audio = indata[:, 0]
    D = np.abs(np.fft.rfft(audio * np.hanning(len(audio))))
    freqs = np.fft.rfftfreq(len(audio), 1/SR)

    def band_rms(f_lo, f_hi):
        mask = (freqs >= f_lo) & (freqs < f_hi)
        return float(np.sqrt(np.mean(D[mask]**2))) if mask.any() else 0.0

    bass = min(1.0, band_rms(80, 300) * 8)
    mid  = min(1.0, band_rms(300, 4000) * 5)
    hi   = min(1.0, band_rms(4000, 20000) * 12)

    # Enviar a Resolume via OSC
    osc.send_message("/composition/layers/1/video/opacity", bass)
    osc.send_message("/composition/layers/2/video/speed", 0.5 + mid)
    osc.send_message("/composition/video/brightness", 0.7 + hi * 0.3)

with sd.InputStream(callback=audio_callback, blocksize=BLOCK, samplerate=SR):
    input("Ctrl+C para parar...")

Ableton Link — Sincronización de Fase P2P

Link usa consenso peer-to-peer sobre UDP (sin maestro/esclavo). Todos los nodos en la red calculan continuamente el offset entre sus relojes y acuerdan una línea de tiempo compartida. El parámetro Quantum define la unidad de cuantización de la fase (ej. Quantum=4 = sincroniza al compás de 4 beats).

# Python — Ableton Link via pylinkthon
import link  # biblioteca Python de Ableton Link

a_link = link.Link(120.0)  # tempo inicial 120 BPM
a_link.enabled = True
a_link.startStopSyncEnabled = True

def on_tempo_change(tempo: float):
    print(f"Nuevo tempo de la red: {tempo:.2f} BPM")

# Obtener estado de la sesión (beat, phase, tempo)
clock = a_link.clock()
session_state = a_link.captureAppSessionState()

beat  = session_state.beatAtTime(clock.micros(), quantum=4)
phase = session_state.phaseAtTime(clock.micros(), quantum=4)
tempo = session_state.tempo()
print(f"Beat: {beat:.3f} | Phase: {phase:.3f}/4 | Tempo: {tempo:.2f}")

# Forzar Resolume a seguir Link:
# Resolume → Preferences → Sync → Enable Ableton Link
# Quantum debe coincidir con el set de Ableton (típico: 4 o 8)

SMPTE LTC — Timecode Frame-Accurate

LTC (Linear/Longitudinal Timecode) es una señal de audio digital que codifica una estampa de tiempo absoluta en formato HH:MM:SS:FF (horas:minutos:segundos:frames). Se inyecta como señal de audio (típicamente en el canal derecho de la mesa del DJ mientras la música va al izquierdo) y Resolume la decodifica para disparar eventos en frames exactos.

Scripts FFmpeg para Conversión en Lote

:: ================================================
:: Windows Batch Script — Convertir MP4 a HAP
:: Guardar como "convert_to_hap.bat" en carpeta con .mp4
:: ================================================
@echo off
echo Convirtiendo archivos MP4 a HAP (GPU codec)...

:: HAP estándar (sin alfa)
for %%i in (*.mp4) do (
    echo Procesando: %%i
    ffmpeg -i "%%i" ^
           -c:v hap ^
           -format hap ^
           -compressor snappy ^
           -vf "scale=trunc(iw/2)*2:trunc(ih/2)*2" ^
           "%%~ni_HAP.mov"
)

:: HAP Alpha (con canal de transparencia) — desde .mov con alfa
for %%i in (*_alpha.mov) do (
    ffmpeg -i "%%i" ^
           -c:v hap ^
           -format hap_alpha ^
           "%%~ni_Converted.mov"
)

:: HAP Q (máxima calidad, mayor bitrate — para outputs 4K)
for %%i in (*_4k.mp4) do (
    ffmpeg -i "%%i" ^
           -c:v hap ^
           -format hap_q ^
           "%%~ni_HAPQ.mov"
)

echo Conversión completada.
pause

Advanced Output — Slices y Projection Mapping

Workflow de Output Avanzado para LED Walls

Un procesador de video LED (ej. Novastar, Brompton) exige secciones discontinuas de la señal de video organizada como una textura única. El Advanced Output de Resolume permite mapear regiones específicas del lienzo a salidas físicas distintas.

  1. Diseñar la topología: El técnico LED te dará un plano con la disposición de los procesadores (ej. Procesador A = píxeles 0–383, Procesador B = 384–767)
  2. Crear slices: Advanced Output → Add Slice → definir posición y tamaño (ej. 384×192px) → asignar nombre descriptivo ("Panel_A_izquierda")
  3. Match Output Shape: Seleccionar el slice → botón "Match Output Shape" → la relación de aspecto del input se ajusta automáticamente al slice
  4. Posicionar sobre el lienzo: Mantener Shift+Alt mientras se arrastra = escala desde el centro conservando proporciones
  5. Asignar fuente: Input Source del slice → seleccionar capa o composición específica

M7 · Layer-to-Slice
Layer-to-Slice Routing (L2S)
Ignore Opacity, Bypass, ruteo directo evitando compositor maestro
L2S: Capa → Slice directo, saltando blend modes y opacity del compositor maestro
Bypass: desactiva el pipeline matemático del bus → 0 ciclos GPU en esa ruta
Ignore Opacity: la capa puede estar "apagada" visualmente pero la señal llega limpia al slice

¿Por qué L2S salva la GPU?

Sin L2S: Resolume renderiza toda la composición maestra (ej. 7680×4320), aplica todos los blend modes de todas las capas, y luego recorta slices de ese lienzo gigante. Esto es enormemente ineficiente para setups multi-pantalla con contenidos independientes.

Con L2S: cada capa se envía directamente a su slice de destino sin pasar por el compositor maestro. El motor de render de Resolume solo procesa la resolución del slice (ej. 1920×1080) para esa capa, no la composición entera.

# Configuración L2S en Resolume Arena:
# 1. Capa en la composición: reducida (ej. 1080p)
# 2. Advanced Output → seleccionar Slice de destino
# 3. Input Source del Slice → "Layer 3" (nombre de la capa)
# 4. Activar en las propiedades del routing:
#    ✓ "Ignore Opacity" — la capa puede tener opacity=0 visible
#                          pero la señal viaja al slice sin atenuación
#    ✓ "Bypass"         — elimina del pipeline de la composición maestra
#                          (0 ciclos GPU en el compositor maestro)

# RESULTADO: renderizar 4 capas 1080p independientes → 4 slices
# vs renderizar 1 composición 4K con 4 capas:
# L2S: 4 × 1920×1080 = 8.3 MP por frame
# Sin L2S: 1 × 3840×2160 = 8.3 MP por frame → MISMO coste, pero...
# → L2S: 0 blend modes maestros, 0 mezclas intermedias, 0 copies
# → Sin L2S: N blend modes por capa, copia de texturas entre capas

# Para muchos procesadores (7-10+), L2S reduce dropped frames >50%

# OSC para activar/desactivar bypass en tiempo real:
from pythonosc import udp_client
osc = udp_client.SimpleUDPClient("127.0.0.1", 7000)

# Desactivar capa del compositor maestro (pero mantener señal al slice):
osc.send_message("/composition/layers/3/bypassed", 1)  # 1=bypass on
# La capa 3 no se ve en el preview maestro, pero el slice sigue recibiendo.

FFGL — FreeFrame OpenGL Plugin Architecture

FFGL es el estándar abierto para plugins de efectos de video en Resolume, VDMX, Millumin, etc. Un plugin FFGL es una DLL (Windows) o dylib (macOS) que implementa callbacks de OpenGL y se inyecta en el pipeline de renderizado de Resolume.

// FFGL Plugin mínimo — C++
// Requiere: FFGL SDK (https://github.com/resolume/ffgl)
#include "FFGL.h"
#include "FFGLLib.h"

class ChaserPlugin : public CFreeFrameGLPlugin {
public:
    FFResult ProcessOpenGL(ProcessOpenGLStruct* pGL) override {
        // pGL->inputTextures[0] = textura de la capa de entrada
        // Aquí ejecutamos nuestro shader GLSL
        glUseProgram(m_shader_program);
        glUniform1f(m_param_speed,    m_speed);
        glUniform1f(m_param_offset,   m_offset);
        glUniform2f(m_param_resolution, pGL->width, pGL->height);

        // Dibujar quad con la textura procesada
        draw_fullscreen_quad(pGL->inputTextures[0]);
        glUseProgram(0);
        return FF_SUCCESS;
    }

    // Exponer parámetros a la UI de Resolume
    FFResult SetFloatParameter(unsigned int index, float value) override {
        if (index == 0) m_speed = value;
        if (index == 1) m_offset = value;
        return FF_SUCCESS;
    }

private:
    float m_speed = 1.0f;
    float m_offset = 0.0f;
    GLuint m_shader_program;
};

Examen Feynman

Q1 · Codecs

¿Por qué H.264 causa stuttering en VJ y HAP no? Describe el camino de los datos desde el disco hasta la pantalla en ambos casos.

Ver respuesta

H.264 usa compresión inter-frame (GOP): para reconstruir el frame 87, la CPU necesita el I-frame más cercano (ej. frame 60) + todos los P/B-frames intermedios. En VJ, el scrubbing no-lineal y los triggers fuerzan decodificaciones fuera de orden → CPU saturada → dropped frames. Camino H.264: NVMe → CPU (decodificación software) → RAM → PCIe → VRAM (2 copias + cómputo CPU).

HAP/DXV usa compresión intra-frame: cada frame es una imagen DXT1/DXT5 completa e independiente. Camino HAP: NVMe → PCIe (DMA directo) → VRAM → Texture Units GPU (decompresión paralela en hardware) → compositor. Seek = O(1), 0 carga CPU.

Q2 · Blend Modes

¿Por qué el modo Multiply equivale a un VCA en síntesis de audio? Si la capa A tiene valor RGB (0.3, 0.3, 0.3) y la capa B tiene (0.8, 0.5, 0.2), ¿cuál es el resultado en Multiply?

Ver respuesta

Un VCA (Voltage Controlled Amplifier) en síntesis modula la amplitud de una señal por otra: Output = Signal × Control_Voltage. Multiply hace exactamente lo mismo con píxeles: Z = A × B. La capa A actúa como "señal de control" que modula (atenúa o amplifica) la capa B. Los píxeles negros de A (valor 0) suprimen completamente B → efecto máscara. Los blancos de A (valor 1) dejan pasar B sin cambio.

Cálculo: Z = (0.3×0.8, 0.3×0.5, 0.3×0.2) = (0.24, 0.15, 0.06). El resultado es significativamente más oscuro que B original, porque A es una gris medio-oscuro que atenúa todo.

Q3 · FFT bins

Resolume hace una FFT de 1024 puntos a 48kHz. ¿Cuántos bins cubren la banda de bass [80-300Hz]? ¿Y por qué hay menos resolución espectral en graves que en agudos?

Ver respuesta

df = (48000/2) / 1024 = 24000 / 1024 ≈ 23.4 Hz/bin. Bin de 80Hz: round(80/23.4) = bin 3. Bin de 300Hz: round(300/23.4) = bin 13. Total: 13 - 3 = 10 bins para toda la banda de bass [80-300Hz].

La FFT lineal distribuye los bins de forma uniforme en frecuencia. Los graves tienen muy poco espacio en escala lineal (80-300Hz = 220Hz de rango) vs los agudos (4000-20000Hz = 16000Hz de rango). Por eso el oído humano percibe frecuencias de forma logarítmica (escala Mel/Bark), y para análisis perceptualmente correcto se usan CQT (Constant-Q Transform) o filtros de tercio de octava.

Q4 · SMPTE vs Ableton Link

¿En qué situación usarías SMPTE LTC en lugar de Ableton Link, y qué hace técnicamente que SMPTE sea "frame-accurate"?

Ver respuesta

SMPTE se usa en shows pre-programados donde el contenido visual está coreografiado frame a frame con la música (tours de DJ, shows de arena con láseres y CO2). Ableton Link es para improvisación en tiempo real donde el tempo puede cambiar y varios artistas necesitan beat alignment dinámico.

SMPTE LTC es "frame-accurate" porque es una señal de audio analógica (biphase mark) que codifica HH:MM:SS:FF directamente en la onda sonora. No pasa por red (sin jitter de red), no pasa por software scheduler (sin jitter de OS). La interfaz de audio lo decodifica directamente a timecode y Resolume lee ese valor con la misma precisión del sample rate de audio (44100 o 48000Hz → resolución de ~20µs), mucho más preciso que el frame rate del video (25fps = 40ms entre frames).

Q5 · Layer-to-Slice

¿Por qué activar "Ignore Opacity" y "Bypass" en Layer-to-Slice routing mejora el rendimiento? ¿Qué ocurre matemáticamente en el compositor sin estas opciones?

Ver respuesta

Sin L2S, el compositor maestro procesa cada capa: lee su textura, aplica los efectos, aplica el blend mode (Add, Multiply, etc.) contra las capas inferiores, y acumula el resultado en el frame buffer de la composición completa. Esto ocurre para TODAS las capas, incluso si sus outputs van a pantallas separadas que no necesitan mezclarse.

Bypass detiene completamente el pipeline matemático del bus de esa capa → 0 ciclos GPU en el compositor maestro. La textura de la capa sigue existiendo pero no se procesa en la mezcla principal.

Ignore Opacity permite que la capa tenga opacity=0 en el compositor (invisible en el preview) pero la señal se envía al slice con su valor real (1.0). Sin esta opción, al poner opacity=0 para "ocultarla" del compositor, Resolume también dejaría de enviar señal al slice. Juntos permiten gestión independiente del canal visual por pantalla.
