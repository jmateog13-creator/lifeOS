    M1 · Imagen Digital

    Pixels, canales BGR, tipos NumPy, operaciones básicas

    A · Núcleo Teórico Pareto

    La imagen como array NumPy

    OpenCV representa imágenes como arrays NumPy de forma (H, W, C) con canales en orden BGR (no RGB). Entender la memoria contigua es crítico para performance: img.flags['C_CONTIGUOUS'] debe ser True para llamadas a CUDA o C extensions.

    dtype: uint8 → [0,255] | float32 → [0.0,1.0] | float64 → doble precisión

    Tipos de imagen

    • uint8: estándar para display y almacenamiento. Overflow silencioso en aritmética.
    • float32: necesario para operaciones matemáticas (normalizar con /255.0).
    • int16: para diferencias de frames (puede ser negativo).

    Operaciones fundamentales

    • cv2.resize(img, (w,h)) — interpolación INTER_AREA para reducir, INTER_CUBIC para ampliar.
    • img[y1:y2, x1:x2] — crop sin copia (view NumPy, cuidado con modificaciones).
    • cv2.flip(img, 1) — flip horizontal para efecto espejo (esencial en interfaces gestuales).
    • cv2.cvtColor(img, cv2.COLOR_BGR2RGB) — conversión para matplotlib/MediaPipe.

    M2 · Preprocesamiento

    Blur, threshold, morfología — limpieza de señal antes de detección

    A · Núcleo Teórico Pareto

    Pipeline de preprocesamiento

    El preprocesamiento determina la calidad de todo lo que viene después. Los tres pasos canónicos son: suavizado (eliminar ruido), umbralización (binarizar para segmentación) y morfología (limpiar artefactos binarios).

    Suavizado (Blur)

    • GaussianBlur: kernel (ksize, ksize) con ksize impar. Preserva bordes gradualmente. Sigma=0 → calculado automáticamente.
    • MedianBlur: elimina ruido salt-and-pepper (artefactos de cámara). Kernel=5 suele ser suficiente.
    • bilateralFilter: preserva bordes mientras suaviza texturas. Más lento, ideal para skin detection.

    Umbralización

    Otsu: minimiza varianza intra-clase → threshold óptimo automático
    • cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY+cv2.THRESH_OTSU)
    • adaptativeThreshold: para iluminación no uniforme. Window local calcula umbral propio.

    Morfología

    • Erosión: encoge objetos blancos, elimina ruido fino.
    • Dilatación: expande objetos blancos, cierra huecos.
    • Opening (erosión→dilatación): elimina ruido pequeño manteniendo formas grandes.
    • Closing (dilatación→erosión): cierra huecos en objetos sólidos.

    Lab 2.1 — Pipeline completo de segmentación de mano 

    import cv2
import numpy as np

def preprocess_for_hand(frame):
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
    blurred = cv2.GaussianBlur(gray, (7, 7), 0)
    _, thresh = cv2.threshold(
        blurred, 0, 255,
        cv2.THRESH_BINARY_INV + cv2.THRESH_OTSU
    )
    kernel = cv2.getStructuringElement(cv2.MORPH_ELLIPSE, (5, 5))
    opened = cv2.morphologyEx(thresh, cv2.MORPH_OPEN, kernel)
    closed = cv2.morphologyEx(opened, cv2.MORPH_CLOSE, kernel)
    return closed

cap = cv2.VideoCapture(0)
while True:
    ret, frame = cap.read()
    frame = cv2.flip(frame, 1)
    mask = preprocess_for_hand(frame)
    cv2.imshow("Mask", mask)
    cv2.imshow("Original", frame)
    if cv2.waitKey(1) & 0xFF == ord('q'):
        break
cap.release()

    Pregunta Feynman

    ¿Por qué HSV es mejor que BGR para detección de piel, y qué operación morfológica elimina ruido pequeño sin romper objetos grandes?

    Ver respuesta

    HSV separa la información de color (Hue) de la luminancia (Value). La piel humana tiene un rango de Hue estrecho (~0-20°) independientemente de la iluminación, mientras que Value varía mucho con la luz. En BGR, el mismo tono de piel tiene valores muy diferentes con iluminación distinta. La operación que elimina ruido pequeño sin romper objetos grandes es el Opening (erosión seguida de dilatación): la erosión elimina los objetos pequeños (ruido), y la dilatación posterior restaura el tamaño de los objetos grandes que sobrevivieron.

    M3 · Contornos y Formas

    findContours, approxPolyDP, convexHull, bounding boxes

    A · Núcleo Teórico Pareto

    Análisis de contornos

    Los contornos son curvas continuas que unen puntos con la misma intensidad. Sobre una máscara binaria, representan los bordes de los objetos segmentados. Son la base para extraer descriptores de forma que luego se mapearán a parámetros musicales.

    cv2.findContours(mask, RETR_EXTERNAL, CHAIN_APPROX_SIMPLE)
    • RETR_EXTERNAL: solo contornos externos (ignora huecos interiores).
    • CHAIN_APPROX_SIMPLE: comprime segmentos horizontales/verticales/diagonales, guardando solo extremos.

    Descriptores de forma

    • cv2.contourArea(cnt) → área en píxeles². Útil para volumen/densidad.
    • cv2.boundingRect(cnt) → (x,y,w,h). Posición y tamaño.
    • cv2.convexHull(cnt) → envolvente convexa. Defectos de convexidad = dedos extendidos.
    • cv2.approxPolyDP(cnt, epsilon, True) → simplificación Douglas-Peucker.
    • cv2.moments(cnt) → centroide: cx=M10/M00, cy=M01/M00.
    B · Laboratorio Guiado
    Lab 3.1 — Extracción de descriptores de mano desde contorno 

    def analyze_hand_contour(mask):
    contours, _ = cv2.findContours(
        mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE
    )
    if not contours:
        return None

    cnt = max(contours, key=cv2.contourArea)
    area = cv2.contourArea(cnt)

    if area < 5000:
        return None

    M = cv2.moments(cnt)
    cx = int(M['m10'] / M['m00'])
    cy = int(M['m01'] / M['m00'])

    x, y, w, h = cv2.boundingRect(cnt)
    aspect_ratio = w / h

    hull = cv2.convexHull(cnt)
    hull_area = cv2.contourArea(hull)
    solidity = area / hull_area  # ~1.0 = puño, <0.9 = dedos abiertos

    return {
        'centroid': (cx, cy),
        'area': area,
        'aspect_ratio': aspect_ratio,
        'solidity': solidity,
        'bbox': (x, y, w, h)
    }

    Pregunta Feynman

    ¿Qué es la solidez de un contorno, por qué funciona para detectar puño vs mano abierta, y qué parámetro de approxPolyDP controla la agresividad de la simplificación?

    Ver respuesta

    La solidez es la razón entre el área del contorno y el área de su envolvente convexa (contourArea / hullArea). Un puño cerrado tiene contorno casi convexo (solidez ~0.95). Una mano abierta tiene hendiduras entre dedos que reducen el área del contorno pero no la del hull, bajando la solidez a ~0.6-0.8. Es una métrica elegante que no requiere ML para una clasificación básica. El parámetro que controla la agresividad de approxPolyDP es epsilon: distancia máxima permitida entre el contorno original y el aproximado; a mayor epsilon, más simplificación.

    M4 · LK Sparse Optical Flow

    Lucas-Kanade, goodFeaturesToTrack, calcOpticalFlowPyrLK

    A · Núcleo Teórico Pareto

    Optical flow sparse — Lucas-Kanade

    Lucas-Kanade asume que el flujo es constante dentro de una ventana local de píxeles y calcula el desplazamiento minimizando el error de intensidad entre frames. Solo se calcula en puntos de interés (esquinas), haciendo el método eficiente y robusto.

    I_x·u + I_y·v = -I_t (ecuación de restricción de flujo óptico)

    Para múltiples píxeles en la ventana: sistema sobredeterminado → mínimos cuadrados. La solución existe cuando la matriz A^T·A es invertible (esquinas de Harris, no bordes).

    Pipeline LK en OpenCV

    • goodFeaturesToTrack — detector Shi-Tomasi: maxCorners, qualityLevel, minDistance.
    • calcOpticalFlowPyrLK — pirámide Gaussiana multi-escala: winSize, maxLevel.
    • El vector retornado status indica qué puntos se rastrearon con éxito (1) o perdieron (0).
    • Para reiniciar tracks perdidos: re-detectar cuando N puntos activos < umbral.

    Mapeo musical

    • Magnitud media del flujo → amplitud/volumen.
    • Dirección dominante (atan2) → filtro de color armónico.
    • Dispersión de velocidades → densidad granular.
    B · Laboratorio Guiado
    Lab 4.1 — Tracker LK con reinicio automático 

    import cv2
import numpy as np

lk_params = dict(winSize=(15,15), maxLevel=3,
                 criteria=(cv2.TERM_CRITERIA_EPS | cv2.TERM_CRITERIA_COUNT, 10, 0.03))
feat_params = dict(maxCorners=100, qualityLevel=0.3,
                   minDistance=7, blockSize=7)

cap = cv2.VideoCapture(0)
ret, old_frame = cap.read()
old_gray = cv2.cvtColor(old_frame, cv2.COLOR_BGR2GRAY)
p0 = cv2.goodFeaturesToTrack(old_gray, mask=None, **feat_params)

while True:
    ret, frame = cap.read()
    frame = cv2.flip(frame, 1)
    frame_gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)

    if p0 is not None and len(p0) > 5:
        p1, st, err = cv2.calcOpticalFlowPyrLK(old_gray, frame_gray, p0, None, **lk_params)
        good_new = p1[st==1]
        good_old = p0[st==1]

        flow_vectors = good_new - good_old
        magnitudes = np.linalg.norm(flow_vectors, axis=1)
        mean_mag = float(np.mean(magnitudes))

        for new, old in zip(good_new, good_old):
            a, b = new.ravel().astype(int)
            c, d = old.ravel().astype(int)
            cv2.arrowedLine(frame, (c,d), (a,b), (0,255,255), 2)
            cv2.circle(frame, (a,b), 3, (0,0,255), -1)

        cv2.putText(frame, f"Flow: {mean_mag:.2f}px/f",
                    (10,30), cv2.FONT_HERSHEY_SIMPLEX, 0.7, (0,255,255), 2)

        if len(good_new) < 20:
            p0 = cv2.goodFeaturesToTrack(frame_gray, mask=None, **feat_params)
        else:
            p0 = good_new.reshape(-1,1,2)
    else:
        p0 = cv2.goodFeaturesToTrack(frame_gray, mask=None, **feat_params)

    old_gray = frame_gray.copy()
    cv2.imshow("LK Flow", frame)
    if cv2.waitKey(1) & 0xFF == ord('q'):
        break
cap.release()

    E · Espacio Feynman

    Pregunta Feynman

    ¿Por qué Lucas-Kanade falla en bordes pero funciona bien en esquinas, y qué significa status==1 en la salida de calcOpticalFlowPyrLK?

    Ver respuesta

    La ecuación de LK requiere que A^T·A sea invertible, lo que ocurre cuando hay variación de gradiente en dos direcciones. Un borde solo tiene gradiente en una dirección: la ecuación es subindeterminada (hay infinitas soluciones — el flujo paralelo al borde es invisible, "aperture problem"). Una esquina tiene gradiente en dos direcciones, haciendo A^T·A bien condicionada y el flujo recuperable unívocamente. status==1 significa que el punto fue rastreado exitosamente entre frames (se encontró una buena correspondencia en la pirámide LK).

    M5 · Farneback Dense Flow

    calcOpticalFlowFarneback, visualización HSV, magnitud y dirección

    A · Núcleo Teórico Pareto

    Flujo óptico denso — Farneback

    A diferencia de LK (sparse), Farneback calcula el vector de flujo para cada píxel del frame. Aproxima la imagen local con polinomiales cuadráticos y minimiza la diferencia entre frames. Produce un campo vectorial completo: ideal para capturar movimiento difuso, ondas y texturas en movimiento.

    flow = cv2.calcOpticalFlowFarneback(prev_gray, gray, None, 0.5, 3, 15, 3, 5, 1.2, 0)

    Parámetros clave: pyr_scale=0.5 (pirámide), levels=3 (niveles), winsize=15 (ventana local), poly_n=5 (tamaño vecindad polinomial), poly_sigma=1.2 (suavizado).

    Visualización HSV

    El campo de flujo 2D se visualiza asignando Hue=dirección (0-360°→0-180 en OpenCV) y Value=magnitud normalizada. Saturation=255 fijo. Esta codificación permite ver dirección y velocidad simultáneamente con color intuitivo.

    Descriptores globales

    • mag, ang = cv2.cartToPolar(flow[...,0], flow[...,1])
    • Magnitud media → intensidad de movimiento global.
    • Histograma de ángulos → dirección dominante del movimiento.
    B · Laboratorio Guiado
    Lab 5.1 — Visualización de flujo denso en tiempo real 

    import cv2
import numpy as np

cap = cv2.VideoCapture(0)
ret, old_frame = cap.read()
old_gray = cv2.cvtColor(old_frame, cv2.COLOR_BGR2GRAY)
hsv = np.zeros_like(old_frame)
hsv[..., 1] = 255

while True:
    ret, frame = cap.read()
    frame = cv2.flip(frame, 1)
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)

    flow = cv2.calcOpticalFlowFarneback(
        old_gray, gray, None,
        0.5, 3, 15, 3, 5, 1.2, 0
    )

    mag, ang = cv2.cartToPolar(flow[..., 0], flow[..., 1])
    hsv[..., 0] = ang * 180 / np.pi / 2
    hsv[..., 2] = cv2.normalize(mag, None, 0, 255, cv2.NORM_MINMAX)

    flow_rgb = cv2.cvtColor(hsv, cv2.COLOR_HSV2BGR)

    mean_mag = float(np.mean(mag))
    dominant_angle = float(np.median(ang[mag > np.percentile(mag, 75)]))

    cv2.putText(frame, f"Mag:{mean_mag:.2f} Ang:{np.degrees(dominant_angle):.0f}°",
                (10, 30), cv2.FONT_HERSHEY_SIMPLEX, 0.6, (0, 255, 255), 1)

    cv2.imshow("Dense Flow", flow_rgb)
    cv2.imshow("Original", frame)
    old_gray = gray
    if cv2.waitKey(1) & 0xFF == ord('q'):
        break
cap.release()

    Pregunta Feynman

    ¿Cuándo usar flujo denso vs sparse para control musical, y qué codifica el canal Hue en la visualización HSV del flujo denso?

    Ver respuesta

    Flujo denso (Farneback) captura todo el movimiento del frame; es ideal para performances donde el cuerpo entero es el instrumento o para síntesis dirigida por textura visual. Para "pintar con movimiento" con todo el cuerpo: Farneback. En la visualización HSV, el canal Hue codifica la dirección (ángulo) del vector de flujo, mientras que Value codifica la magnitud.

    M6 · MediaPipe Hands

    21 landmarks 3D, handedness, pinch detection, distancias normalizadas

    A · Núcleo Teórico Pareto

    MediaPipe Hands — arquitectura

    MediaPipe Hands usa un pipeline de dos etapas: detector de palma (SSD) seguido de un estimador de landmarks (red de regresión directa). La detección de palma solo se ejecuta cuando no hay manos rastreadas; luego el estimador es mucho más eficiente al trabajar en la ROI de la palma detectada.

    Los 21 landmarks

    Cada landmark tiene coordenadas (x, y, z) normalizadas (0–1 en x,y respecto al frame; z es relativa a la muñeca). Los índices clave son:

    • 0=Muñeca, 4=Pulgar tip, 8=Índice tip, 12=Medio tip, 16=Anular tip, 20=Meñique tip.
    • Bases de dedos: 5=Índice MCP, 9=Medio MCP, 13=Anular MCP, 17=Meñique MCP.

    Pinch detection

    pinch_dist = √((lm4.x-lm8.x)² + (lm4.y-lm8.y)²)

    Distancia normalizada ≈ 0.05–0.08 = pinch cerrado. Mayor a 0.15 = abierto. Normalizar por distancia muñeca-índice MCP para invarianza a distancia cámara.

import mediapipe as mp

def calc_angle(a, b, c):
    a, b, c = np.array(a), np.array(b), np.array(c)
    ba = a - b
    bc = c - b
    cosine = np.dot(ba, bc) / (np.linalg.norm(ba) * np.linalg.norm(bc) + 1e-6)
    return float(np.degrees(np.arccos(np.clip(cosine, -1.0, 1.0))))

    lm = landmarks.landmark

    def pt(idx):
        return [lm[idx].x, lm[idx].y]

    def vis(idx):
        return lm[idx].visibility

    features = {}

    if all(vis(i) > 0.5 for i in [11, 13, 15]):
        features['elbow_L'] = calc_angle(pt(11), pt(13), pt(15))

    if all(vis(i) > 0.5 for i in [12, 14, 16]):
        features['elbow_R'] = calc_angle(pt(12), pt(14), pt(16))

    if vis(15) > 0.5:
        features['wrist_L_y'] = 1.0 - lm[15].y
    if vis(16) > 0.5:
        features['wrist_R_y'] = 1.0 - lm[16].y

    if vis(15) > 0.5 and vis(16) > 0.5:

    return features

    M8 · One Euro Filter

    Suavizado adaptativo: mínima latencia en movimiento rápido, máximo filtrado en reposo

    A · Núcleo Teórico Pareto

    One Euro Filter — motivación

    Los filtros paso-bajo convencionales tienen un tradeoff fundamental: más suavizado → más latencia. Para control musical esto es inaceptable. One Euro Filter resuelve esto adaptando la frecuencia de corte según la velocidad de la señal: cuando el movimiento es lento (alta jitter, poco movimiento), usa fc bajo (más suavizado). Cuando el movimiento es rápido (gesto expresivo), usa fc alto (mínima latencia).

    ẑ_n = α·z_n + (1−α)·ẑ_{n-1}
    α = 1 / (1 + τ·T_e) donde τ = 1/(2π·f_c)
    f_c = f_cmin + β·|dx̂/dt|

    Parámetros: fcmin (frecuencia mínima ~1Hz), β (velocidad de adaptación ~0.007), dcutoff (corte para derivada ~1Hz).

    Implementación práctica

    Necesita dos filtros: uno para la señal (x_filter) y uno para su derivada (dx_filter). La derivada filtrada se usa para calcular fc adaptativo. Se aplica por coordenada (x, y separados) o por parámetro.

    B · Laboratorio Guiado
    Lab 8.1 — Implementación One Euro Filter en Python 

    import math

class LowPassFilter:
    def __init__(self, alpha=1.0):
        self.alpha = alpha
        self.y = None
        self.s = None

    def set_alpha(self, alpha):
        self.alpha = alpha

    def __call__(self, value):
        if self.y is None:
            self.y = value
            self.s = value
        else:
            self.y = self.alpha * value + (1.0 - self.alpha) * self.s
            self.s = self.y
        return self.y

class OneEuroFilter:
    def __init__(self, freq, min_cutoff=1.0, beta=0.007, d_cutoff=1.0):
        self.freq = freq
        self.min_cutoff = min_cutoff
        self.beta = beta
        self.d_cutoff = d_cutoff
        self.x_filter = LowPassFilter()
        self.dx_filter = LowPassFilter()
        self.last_value = None

    def alpha(self, cutoff):
        tau = 1.0 / (2.0 * math.pi * cutoff)
        te = 1.0 / self.freq
        return 1.0 / (1.0 + tau / te)

    def __call__(self, x):
        if self.last_value is None:
            dx = 0.0
        else:
            dx = (x - self.last_value) * self.freq

        self.last_value = x

        edx = self.dx_filter.set_alpha(self.alpha(self.d_cutoff)) or self.dx_filter(dx)
        cutoff = self.min_cutoff + self.beta * abs(edx)
        self.x_filter.set_alpha(self.alpha(cutoff))
        return self.x_filter(x)

filter_x = OneEuroFilter(freq=30, min_cutoff=1.0, beta=0.007)
filter_y = OneEuroFilter(freq=30, min_cutoff=1.0, beta=0.007)

Cuando el movimiento es muy rápido, la frecuencia de corte aumenta (menos suavizado, mínima latencia), lo contrario de cuando está en reposo.

    M9 · OSC Protocol

    python-osc, SimpleUDPClient, bundles, float32 vs MIDI 7-bit

    A · Núcleo Teórico Pareto

    Open Sound Control (OSC)

    OSC es un protocolo de red para comunicación en tiempo real entre software musical. A diferencia de MIDI (serie, 7-bit), OSC usa UDP con mensajes arbitrariamente tipados y organizados en namespaces jerárquicos como URLs. Es el estándar de facto para control gestual en arte interactivo.

    Ventajas sobre MIDI

    • Resolución: float32 (32 bits) vs MIDI 7-bit (128 valores).
    • Flexibilidad: cualquier tipo de dato (int, float, string, blob).
    • Namespacing: /hand/left/pinch vs CC arbitrarios.
    • Bundles: múltiples mensajes con timestamp atómico.

    Latencia UDP

    En localhost, UDP tiene latencia <1ms. La latencia total del sistema es dominada por el procesamiento de video (~30ms a 30fps), no por la transmisión OSC.

    ¿Por qué se usa UDP y no TCP para OSC en tiempo real, y cuántos valores discretos puede transmitir un CC MIDI de 7 bits comparado con OSC float32?

    Ver respuesta

    TCP garantiza entrega y orden, pero a costa de retransmisiones y control de flujo que introducen latencia variable (jitter). Para audio en tiempo real, un paquete llegando 20ms tarde es peor que uno perdido: el músico ya tomó la siguiente acción. UDP "dispara y olvida": si un paquete se pierde, el siguiente con datos frescos llegará pronto sin esperar retransmisión. En cuanto a resolución, un CC MIDI de 7 bits transmite 128 valores discretos (0–127), mientras que OSC float32 ofrece efectivamente ~16.7 millones de valores en el rango [0,1], lo que equivale a una resolución continua para cualquier parámetro musical.

    import mido
import numpy as np
import time

port = mido.open_output('Python MIDI Out', virtual=True)

PENTATONIC = [0, 2, 4, 7, 9]  # semitones en pentatónica mayor
BASE_OCT = 4

def to_pentatonic(raw_note):
    note_in_oct = raw_note % 12
    octave = raw_note // 12
    dists = [abs(note_in_oct - p) for p in PENTATONIC]
    closest = PENTATONIC[np.argmin(dists)]
    return octave * 12 + closest

def send_note(note, velocity=80, duration=0.1):
    port.send(mido.Message('note_on', note=note, velocity=velocity))
    time.sleep(duration)
    port.send(mido.Message('note_off', note=note))

def send_cc(control, value):
    clamped = max(0, min(127, int(value)))
    port.send(mido.Message('control_change', control=control, value=clamped))

    Recepción OSC en Max/MSP

    En Max, el objeto [udpreceive 7400] recibe paquetes UDP en el puerto especificado. El objeto [osc-route /hand/right/pinch] filtra por dirección y extrae el valor. Los valores float llegan como mensajes Max estándar que pueden conectarse a cualquier objeto de síntesis.

    Cadena típica Max

    [udpreceive 7400] → [osc-route /hand/right/pinch] → [scale 0 1 100 8000] → [filtercoeff~] → [biquad~]

    TouchDesigner — CHOP workflow

    En TD, el nodo OSC In CHOP recibe canales OSC como streams de datos continuos. Los CHOPs (Channel Operators) permiten filtrar, transformar y mapear estos canales a parámetros de síntesis, video o DMX. El paradigma es dataflow: cada nodo procesa el stream en tiempo real.

    CHOP esenciales

    • OSC In CHOP: recepción de mensajes OSC como canales.
    • Filter CHOP: suavizado (equivalente al One Euro Filter en TD).
    • Math CHOP: escalar, invertir, mapear rangos.
    • Limit CHOP: clamping y quantización.
    • Audio Oscillator CHOP: oscilador controlado por canal.

    Pregunta Feynman

    ¿Por qué es necesario [line~] al mapear OSC a parámetros de audio, y en TouchDesigner qué tipo de operador (OP) se usa para procesar datos de control en tiempo real?

    Ver respuesta

    Los mensajes OSC llegan a ~30Hz (cada ~33ms), pero el audio se procesa a sample rate (44100Hz). Si conectas directamente un valor OSC al pitch de un oscilador, el parámetro salta abruptamente en cada mensaje → zipper noise (artefactos de modulación abrupta). [line~] interpola suavemente desde el valor anterior al nuevo en el tiempo especificado (ej: 20ms), eliminando los saltos. En TouchDesigner, el tipo de operador para procesar datos de control en tiempo real es el CHOP (Channel Operator).

    M12 · Síntesis Granular

    Granos, ADSR, pitch shift, densidad — control por parámetros gestuales

    A · Núcleo Teórico Pareto

    Síntesis granular

    La síntesis granular descompone el sonido en micro-eventos (granos) de 1–100ms, cada uno con su propio pitch, amplitud y posición en un buffer de muestra. Al controlar densidad (granos/segundo), tamaño, posición de lectura y dispersión, se obtienen texturas sonoras imposibles en síntesis convencional.

    Parámetros controlables por gesto

    • Densidad (granos/s): magnitud del flujo óptico → más movimiento = más granos.
    • Pitch ratio: posición Y de palma → transponer granos arriba/abajo.
    • Posición de lectura (position): posición X de palma → punto de lectura en buffer.
    • Dispersión de posición: solidez del contorno → puño = granos concentrados, abierta = dispersos.
    • Tamaño de grano: ángulo codo → granos cortos (crispy) a largos (smooth).

    Envolvente de grano

    Hann window: w(n) = 0.5·(1 − cos(2πn/(N−1)))

    La ventana de Hann elimina clicks en los bordes del grano. Sin envolvente, los granos producen artefactos de corte abrupto.

    B · Laboratorio Guiado
    Lab 12.1 — Granulador básico en Python con scipy 

    import numpy as np
import soundfile as sf
import sounddevice as sd
from scipy.signal import windows

def granular_synth(buffer, sr=44100, density=20, grain_size_ms=50,
                   position=0.5, pitch_ratio=1.0, spread=0.1):
    grain_samples = int(sr * grain_size_ms / 1000)
    buf_len = len(buffer)
    output_len = sr  # 1 segundo de output

    output = np.zeros(output_len, dtype=np.float32)
    hann = windows.hann(grain_samples).astype(np.float32)

    n_grains = int(density)
    for i in range(n_grains):
        pos = position + np.random.uniform(-spread, spread)
        pos = np.clip(pos, 0.0, 1.0)
        start = int(pos * (buf_len - grain_samples))
        start = max(0, min(start, buf_len - grain_samples))

        grain = buffer[start:start + grain_samples].copy()

        if pitch_ratio != 1.0:
            indices = np.arange(0, grain_samples, pitch_ratio)
            indices = indices[indices < grain_samples].astype(int)
            grain_pitched = grain[indices]
            if len(grain_pitched) < grain_samples:
                grain_pitched = np.pad(grain_pitched,
                                       (0, grain_samples - len(grain_pitched)))
            grain = grain_pitched[:grain_samples]

        grain *= hann

        t_start = int(i * output_len / n_grains)
        t_end = min(t_start + grain_samples, output_len)
        output[t_start:t_end] += grain[:t_end - t_start]

    max_val = np.max(np.abs(output))
    if max_val > 0:
        output /= max_val * 1.2

    return output

    Pregunta Feynman

    ¿Por qué la ventana Hann elimina los clicks entre granos, y qué parámetro gestual es más natural para controlar la densidad de textura?

    Ver respuesta

    Un grano sin envolvente comienza y termina con valores abruptos que pueden estar lejos de cero → discontinuidad en la señal de audio → click audible. La ventana Hann tiene forma de coseno: comienza y termina suavemente en cero, independientemente del contenido del grano. Al multiplicar el grano por esta ventana, los bordes siempre hacen fade-in y fade-out sin click. Para controlar la densidad granular, el parámetro más natural es la magnitud del flujo óptico (velocidad de movimiento): más movimiento = más granos = textura más densa; el intérprete inmóvil produce silencio casi total.

    M13 · Clasificadores de Gestos

    k-NN, DTW, features de landmarks, reconocimiento en tiempo real

    A · Núcleo Teórico Pareto

    Clasificación de gestos estáticos y dinámicos

    Para gestos estáticos (poses de mano), el feature vector de landmarks normalizados alimenta directamente un clasificador k-NN o SVM. Para gestos dinámicos (secuencias temporales), DTW (Dynamic Time Warping) mide similaridad entre trayectorias de longitud variable.

    Feature engineering para landmarks

    • Distancias relativas: distancias inter-landmark normalizadas por longitud de referencia. Invariantes a escala y posición.
    • Ángulos: ángulos entre segmentos de dedos. Invariantes a rotación en el plano.
    • Coordenadas relativas a palma: landmark_i − centroide_palma. Invariante a traslación.
    feature_vec = [lm[i].x - cx, lm[i].y - cy for i in range(21)] → 42 features

    k-NN para gestos estáticos

    Con k=3–5 vecinos y feature vector de 42 dimensiones, k-NN clasifica gestos estáticos con >95% accuracy en condiciones controladas. No requiere entrenamiento extenso: 10–20 ejemplos por clase son suficientes.

    DTW para gestos dinámicos

    DTW(A,B) = min alineamiento temporal de dos secuencias de landmarks

    Lab 13.1 — Clasificador k-NN de gestos de mano 

    import numpy as np
from sklearn.neighbors import KNeighborsClassifier
import pickle

def extract_features(landmarks):
    lm = landmarks.landmark
    cx = np.mean([lm[i].x for i in [5,9,13,17]])
    cy = np.mean([lm[i].y for i in [5,9,13,17]])
    ref = np.sqrt((lm[5].x-lm[0].x)**2 + (lm[5].y-lm[0].y)**2) + 1e-6

    features = []
    for i in range(21):
        features.append((lm[i].x - cx) / ref)
        features.append((lm[i].y - cy) / ref)

    return np.array(features, dtype=np.float32)

class GestureClassifier:
    def __init__(self, k=3):
        self.clf = KNeighborsClassifier(n_neighbors=k, metric='euclidean')
        self.X_train = []
        self.y_train = []
        self.trained = False

    def add_sample(self, landmarks, label):
        feat = extract_features(landmarks)
        self.X_train.append(feat)
        self.y_train.append(label)

    def train(self):
        if len(self.X_train) >= 3:
            self.clf.fit(self.X_train, self.y_train)
            self.trained = True

    def predict(self, landmarks, threshold=0.6):
        if not self.trained:
            return None, 0.0
        feat = extract_features(landmarks)
        proba = self.clf.predict_proba([feat])[0]
        max_proba = np.max(proba)
        if max_proba < threshold:
            return "unknown", max_proba
        pred = self.clf.classes_[np.argmax(proba)]
        return pred, max_proba

    def save(self, path):
        with open(path, 'wb') as f:
            pickle.dump({'clf': self.clf, 'trained': self.trained}, f)

    Pregunta Feynman

    ¿Por qué normalizar los landmarks al centroide de la palma antes de clasificar, y para qué tipo de gesto es preferible DTW sobre k-NN estático?

    Ver respuesta

    Sin normalizar, el k-NN ve posiciones absolutas distintas como gestos diferentes aunque la forma de la mano sea idéntica (si el usuario mueve la mano por el frame). Al restar el centroide de la palma y dividir por la longitud de referencia, todas las posiciones son relativas a la mano misma: el mismo gesto da el mismo feature vector independientemente de posición y distancia a la cámara. DTW es preferible sobre k-NN estático para gestos dinámicos como "agitar", "círculo" o "deslizar" — secuencias temporales de longitud variable que no se pueden representar como un solo vector de pose.

    Feynman 1 — Arquitectura del sistema

    Pregunta Feynman

    ¿Cuál es la fuente de latencia dominante en un sistema de control gestual a 30fps, y cómo afecta a las decisiones de diseño del pipeline?

    Ver respuesta

    La latencia dominante es el tiempo de procesamiento de un frame de video (~33ms a 30fps). La transmisión UDP de OSC añade <1ms en localhost. El One Euro Filter añade latencia variable pero típicamente <5ms. Esta dominancia del frame time implica que optimizar el procesamiento por frame (MediaPipe, conversiones de color, cálculo de features) tiene mayor impacto que optimizar el protocolo de comunicación. Trabajar a 60fps reduciría la latencia de frame a ~16ms, duplicando la responsividad del sistema.

    Feynman 2 — Decisiones de filtrado

    Pregunta Feynman

    ¿Qué problema resuelve usar bundles OSC en lugar de mensajes individuales, y cómo balancearías β y fcmin de One Euro Filter para un theremin vs un sistema de texturas ambientales?

    Ver respuesta

    Los bundles OSC garantizan atomicidad temporal: todos los parámetros del mismo frame llegan juntos al receptor, evitando que Max/MSP procese combinaciones incoherentes donde palm_x del frame N llega con pinch del frame N-1. Para un theremin (respuesta inmediata necesaria): β alto (~0.05), fcmin moderado (~1Hz). Para texturas ambientales (estabilidad prioritaria sobre respuesta): β bajo (~0.001), fcmin bajo (~0.5Hz) para máximo suavizado en reposo.

    Feynman 3 — Musicalidad del instrumento

    Pregunta Feynman

    ¿Por qué es importante quantizar el pitch a una escala musical en un instrumento gestual, y qué mapeos gestuales resultan más intuitivos musicalmente?

    Ver respuesta

    El control gestual tiene jitter inherente (pequeñas fluctuaciones de posición incluso en reposo). Sin quantización, ese jitter haría el pitch sonar constantemente "desafinado". La quantización a escala (pentatónica, diatónica) convierte el ruido en notas válidas. Los mapeos más intuitivos respetan metáforas físicas: altura de la mano → pitch (arriba = agudo, como en el theremin), velocidad de movimiento → intensidad/densidad, apertura de la mano → apertura espectral del timbre. Estos mapeos son inmediatamente comprensibles para músicos y audiencias sin entrenamiento previo.

    Feynman 4 — Clasificación de gestos

    Pregunta Feynman

    ¿Cuántos ejemplos de entrenamiento por gesto son suficientes para k-NN con landmarks normalizados, y cómo evaluarías la robustez del clasificador?

    Ver respuesta

    10–30 ejemplos por gesto dan buen rendimiento para un solo usuario con landmarks normalizados: el feature space de 42 dimensiones es suficientemente discriminativo para gestos distintos (puño, mano abierta, peace, etc.). Para evaluar robustez: (1) leave-one-out cross-validation sobre las muestras propias, (2) probar con distintos niveles de iluminación, (3) probar con un segundo usuario para detectar si el clasificador está sobre-ajustado a la mano del creador. Los gestos más difíciles de distinguir suelen ser los que comparten configuración de dedos similares (OK vs índice solo).

    Feynman 5 — Síntesis granular gestual

    Pregunta Feynman

    En un granulador controlado por flujo óptico, ¿qué ocurre musicalmente cuando el intérprete se queda inmóvil?

    Ver respuesta

    Cuando el intérprete se queda inmóvil, el flujo óptico cae a cero o casi cero. Si el parámetro density (granos/segundo) está mapeado a la magnitud del flujo, la densidad cae al mínimo configurado → textura muy sparse o silencio casi total. Esto crea una dramaturgia natural: el movimiento produce sonido, el reposo produce silencio o microsonidos. Es una de las metáforas más potentes del control gestual: el músico "da vida" al instrumento con su movimiento, y la inmovilidad es una elección expresiva de silencio, no un fallo del sistema.
